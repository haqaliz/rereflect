# Implementation Plan — capture-seam (backend + frontend)

**Aspect:** `capture-seam` · **Feature:** `urgency-classifier-head` · **Date:** 2026-07-14
**Spec:** `docs/planning/urgency-classifier-head/capture-seam/spec.md`
**Services:** `services/backend-api` (new endpoint + public_api capture), `services/frontend-web` (toggle + client)
**Sequence:** Wave 1 (independent of urgency-core & data-and-config). Land early so signal accrues.

## 0. Summary

Record `AICorrection(correction_type="urgency", corrected_value ∈ {"urgent","not_urgent"})` on every
**user-driven** `is_urgent` change, from BOTH the dashboard and the public API. Grounded against source:
- `create_ai_correction(db, *, organization_id, user_id, correction_type, entity_type, entity_id, signal,
  original_value, corrected_value, feedback_text)` (`services/ai_correction_service.py:15`) — the shared
  helper; commits internally; `user_id` nullable for API-key writes.
- Internal feedback router prefix `/api/v1/feedback` (`feedback.py:23`). `analyze_single_feedback` sets the
  heuristic `is_urgent` (`:112`); `update_feedback` PATCH clears it to re-analyze (`:665`) — **both are
  analyzer-driven, must NOT emit a correction.**
- Public `public_update_feedback` sets `fb.is_urgent = bool(data.is_urgent)` in the "Tags / is_urgent"
  block (`public_api.py`, after the record-only correction block) — currently emits **nothing** for urgency.
- The `corrected_value` vocabulary MUST be exactly lowercase `"urgent"`/`"not_urgent"` to match
  `URGENCY_LABELS` (urgency-core). Treat urgency-core's `URGENCY_LABELS` as the source of truth.

## 1. Project Setup

- No migration (`correction_type` is a free `String(50)`), no new deps, no `.env`.
- Backend + frontend venvs/node for tests.

## 2. Implementation Phases (strict TDD)

### Phase 1 — Shared urgency-label constant (avoid vocab drift)

- **Goal:** one backend source of truth for `"urgent"`/`"not_urgent"` so capture can't drift from the
  classifier vocab.
- **Files:** `services/backend-api/src/services/ai_correction_service.py` (or a small constants module);
  test asserting the two strings.
- **GREEN:** add `URGENCY_CORRECTED_VALUES = ("not_urgent", "urgent")` and a helper
  `urgency_label(is_urgent: bool) -> str` returning `"urgent"` if True else `"not_urgent"`.
- **Cross-service note:** add a comment referencing analysis-engine `URGENCY_LABELS` and add a test in the
  worker/analysis suite (or a shared test) asserting the two vocabularies are identical
  (`("not_urgent","urgent")`). Flag if they ever diverge.
- **Commit:** `feat(urgency-capture): shared urgency corrected-value constants`

### Phase 2 — Internal urgent-toggle endpoint (JWT)

- **Goal:** a dashboard user can set one feedback item's `is_urgent`; on a value change, emit the urgency
  correction with the acting user's id.
- **Files:** `services/backend-api/src/api/routes/feedback.py`; tests
  `services/backend-api/tests/test_feedback_urgent_toggle.py` (new).
- **RED:** tests:
  - `PATCH /api/v1/feedback/{id}/urgent` with `{is_urgent: true}` on a not-urgent item → 200, `is_urgent`
    now True, and exactly one `AICorrection(correction_type="urgency", signal="correction",
    corrected_value="urgent", original_value="not_urgent", user_id=<current user>, entity_id=<fb.id>)`.
  - Setting to the SAME value → no correction row (no-op guard).
  - Cross-org id → 404, no correction.
  - Requires auth (JWT); org-scoped via `get_current_org`.
- **GREEN:** add a new endpoint (dedicated, minimal — do NOT entangle the text-edit `update_feedback` at
  `:636`):
  ```python
  class UrgentUpdateRequest(BaseModel):
      is_urgent: bool

  @router.patch("/{feedback_id}/urgent", response_model=FeedbackResponse)
  def set_feedback_urgent(feedback_id: int, body: UrgentUpdateRequest,
                          db=Depends(get_db), current_user=Depends(get_current_user),
                          current_org=Depends(get_current_org)):
      fb = <query FeedbackItem by id + organization_id == current_org.id>  # 404 if None
      old = bool(fb.is_urgent)
      new = bool(body.is_urgent)
      if new != old:
          fb.is_urgent = new
          db.commit(); db.refresh(fb)
          create_ai_correction(db, organization_id=current_org.id, user_id=current_user.id,
              correction_type="urgency", entity_type="feedback_item", entity_id=fb.id,
              signal="correction", original_value=urgency_label(old),
              corrected_value=urgency_label(new), feedback_text=fb.text)
      return <FeedbackResponse for fb>
  ```
  (Import `create_ai_correction` + `urgency_label`. Match the file's existing `FeedbackResponse`
  construction, e.g. mirror the GET at `:505-550`.)
- **Validation:** `pytest tests/test_feedback_urgent_toggle.py -v`
- **Commit:** `feat(urgency-capture): internal PATCH /feedback/{id}/urgent emits urgency correction`

### Phase 3 — Public API capture

- **Goal:** the public PATCH emits the urgency correction when `is_urgent` changes (currently nothing).
- **Files:** `services/backend-api/src/api/routes/public_api.py`; tests
  `services/backend-api/tests/` (extend the public-api PATCH test).
- **RED:** tests:
  - public `PATCH /feedback/{id}` with changed `is_urgent` → one urgency correction (`user_id=None`,
    `corrected_value` matching new value).
  - unchanged `is_urgent` → no correction.
  - existing `correction` field behavior + tags + workflow_status all unaffected (characterization stays
    green).
- **GREEN:** in the "Tags / is_urgent" block, capture `old = bool(fb.is_urgent)` BEFORE the assignment;
  after `fb.is_urgent = bool(data.is_urgent)`, if `bool(data.is_urgent) != old`, call `create_ai_correction`
  with `user_id=None`, `correction_type="urgency"`, `original_value=urgency_label(old)`,
  `corrected_value=urgency_label(new)`, `entity_id=fb.id`, `feedback_text=fb.text`. Keep it best-effort/
  non-atomic consistent with the existing record-only correction note in that file.
- **Validation:** `pytest tests/ -k public_ -v`
- **Commit:** `feat(urgency-capture): public PATCH /feedback/{id} records urgency correction on is_urgent change`

## 6. Edge Cases & Error Handling

- **R-5 (critical):** the analyzer heuristic set (`feedback.py:112`) and the re-analyze clear (`:665`) must
  NOT create urgency corrections. Add an explicit test that creating/analyzing feedback and the text-edit
  PATCH produce zero urgency corrections.
- No-op guard: same-value set emits nothing (both endpoints).
- Cross-org: 404, no side effect.
- Non-atomic correction insert on the public path is acceptable (append-only signal store) — mirror the
  existing note in `public_api.py`.
- `corrected_value` always lowercase from `urgency_label()` — never trust client casing.
