# Implementation Plan — predict-seam (worker + backend mirror, at ingest)

**Aspect:** `predict-seam` · **Feature:** `urgency-classifier-head` · **Date:** 2026-07-14
**Services:** `services/worker-service` + `services/backend-api` (byte-identical `classifier_predict.py` mirrors)
**Sequence:** Wave 3 (LAST). **Blocked by data-and-config** (mode column/resolver) **and worker-trainer**
(a promoted urgency model to resolve). Highest-stakes — it can change `feedback.is_urgent`.

## 0. Summary

At ingest, apply the resolved per-org urgency model to `feedback.is_urgent` with off/shadow/auto semantics,
**add-only in auto** (escalate `not_urgent→urgent`; NEVER de-escalate). Grounded against source:
- `apply_classifier_override(feedback, db, *, classifier_type, allow_override)` in
  `services/worker-service/src/services/classifier_predict.py` (and the byte-identical backend mirror
  `services/backend-api/src/services/classifier_predict.py`). It already branches `category` vs
  sentiment; add an `urgency` branch. **Mirror test** in BOTH services
  (`test_classifier_predict_mirror.py`) requires the two copies stay identical except import lines,
  docstring header, and the `allow_override` value the CALL-SITE passes — so the urgency branch must be
  added identically to both files.
- Worker call sites: `tasks/analysis.py` LLM path (`:430-436`) and keyword path (`:612-618`) call it for
  sentiment + category with `allow_override=True` (worker owns authoritative auto). Add an urgency call at
  both, `allow_override=True`.
- Backend call sites: `api/routes/feedback.py:154-166` call it for sentiment + category with
  `allow_override=False` (shadow-only; backend never auto-mutates — worker owns it). Add an urgency call at
  both backend sites too, `allow_override=False`, for parity/shadow-logging.
- `predict_label_only(text)` (binary, no signed score) is the right predictor for urgency (like category).

## 2. Implementation Phases (strict TDD)

### Phase 1 — Urgency branch in `apply_classifier_override` (both mirrors)

- **Goal:** off/shadow/auto for urgency, add-only in auto.
- **Files:** `worker .../services/classifier_predict.py`, `backend .../services/classifier_predict.py`
  (identical edit); tests in both services (extend the existing `apply_classifier_override` tests +
  keep `test_classifier_predict_mirror.py` green).
- **RED:** tests (worker copy, `allow_override=True`):
  - `off` (resolver None) → no-op, `is_urgent` unchanged, no shadow log.
  - `shadow` → `is_urgent` unchanged; ONE `rereflect.classifier.shadow` log line with
    `classifier_type="urgency"`, `incumbent_label` = current is_urgent mapped to urgent/not_urgent,
    `challenger_label` from the model. Log fires for would-be de-escalations too (caplog-testable).
  - `auto` + model predicts `"urgent"` while `is_urgent=False` → `is_urgent` becomes `True` (escalation).
  - **`auto` + model predicts `"not_urgent"` while `is_urgent=True` → `is_urgent` STAYS `True`
    (NO de-escalation).** This is the add-only guard — mandatory test.
  - `auto` + no active model (loader None) → incumbent retained, no log.
  - Backend copy test: with `allow_override=False`, `auto` never mutates `is_urgent` (shadow-log only).
  - Mirror test stays green (both files identical but for the documented diffs).
- **GREEN:** in `apply_classifier_override`, extend the prediction block and the auto-apply block:
  ```python
  # prediction block — add an urgency arm alongside category:
  elif classifier_type == "urgency":
      challenger_label = loaded.predict_label_only(text)   # "urgent" | "not_urgent"
      challenger_score = None
      target_field = None
      incumbent_label = "urgent" if getattr(feedback, "is_urgent", False) else "not_urgent"
      incumbent_score = None
  ```
  ```python
  # auto-apply block (inside `if resolved.mode == "auto" and allow_override:`):
  elif classifier_type == "urgency":
      # ADD-ONLY (PRD auto-mode decision): escalate only, never de-escalate.
      if challenger_label == "urgent":
          feedback.is_urgent = True
      # challenger "not_urgent" while incumbent urgent → intentionally NOT applied.
  ```
  Update the module/function docstring in BOTH copies identically to document the urgency arm + add-only
  rule. Keep the shadow-log call shared (it already logs the generic fields).
- **Validation:** `pytest tests/ -k "classifier_predict or override or mirror" -v` in BOTH services.
- **Commit:** `feat(urgency-predict): urgency arm in apply_classifier_override (add-only auto, both mirrors)`

## 6. Edge Cases & Error Handling

- **R-2 (high-stakes):** default `off` never touches `is_urgent`; only per-org opt-in `auto` escalates, and
  even then add-only — a thin model can never turn a real churn flag OFF this pass. The no-de-escalation
  test is the guard.
- `apply_classifier_override` never raises (existing try/except) — a bad artifact leaves the heuristic value.
- Ordering bug risk: applying before the heuristic baseline would make `off` diverge — assert ordering.
- Shadow logs BOTH directions (incl. would-be de-escalations) so a future higher-confidence phase can
  evaluate whether to enable de-escalation.
