### Phase 1 — Backend model columns + migration

**Goal:** `customer_usage_history` can store a trend state and pct.
**Prerequisites:** none.

**Files:**
- `services/backend-api/src/models/customer_usage_history.py` (modify)
- `services/backend-api/alembic/versions/<new_rev>_add_trend_to_usage_history.py` (create)
- `services/backend-api/tests/test_usage_history_trend_columns_migration.py` (create)

**Steps:**
1. **RED** — write the migration test first, modeled on the existing
   `tests/test_usage_trend_fields_migration.py` (the M3.2b sibling, same shape):
   - both columns exist on the table
   - both are nullable
   - a row inserted without either succeeds and reads back `NULL`
   - `upgrade()` → `downgrade()` round-trips, dropping both
   Run it; it must fail on missing columns.
2. **GREEN** — add to the model, after `last_active_at` (`customer_usage_history.py:55`),
   inside the existing "Snapshot payload" block:
   ```python
   usage_trend_state = Column(String(30), nullable=True)
   usage_trend_pct = Column(Float, nullable=True)
   ```
   `Float` needs adding to the `sqlalchemy` import list at `:18-27`.
   **Nullable, with no server_default** — deliberately unlike the `customer_usage` columns
   (which are `nullable=False, server_default="insufficient_history"`). Pre-existing snapshot
   rows genuinely have no known trend state, and `timeline-trend-event` relies on `NULL`
   meaning "unknown, skip" rather than a real state (spec AC4 there).
3. Generate the migration:
   ```bash
   cd services/backend-api && source venv/bin/activate
   alembic revision -m "add trend to usage history"
   ```
   Hand-edit to two `op.add_column` calls + a two-`op.drop_column` downgrade, mirroring
   `a5b63dbbce9b_add_usage_trend_fields.py:37-55`. Confirm `down_revision` is the verified head.
4. Update the model docstring — it currently says *"This aspect computes no trend"*
   (`customer_usage_history.py:8`), which this change makes false.

**Validation:** `pytest tests/test_usage_history_trend_columns_migration.py -v`
**Commit:** `feat(snapshot-trend-columns): usage_trend_state/pct on customer_usage_history`

### Phase 3 — Snapshot records the *current* trend state (the ordering fix)

**Goal:** the snapshot for date N carries the state classified on date N, not the previous one.
**Prerequisites:** Phases 1-2.

**Files:**
- `services/worker-service/src/tasks/usage_metrics.py` (modify, `recompute_usage_scores`)
- `services/worker-service/tests/test_usage_trend_snapshot_state.py` (create)

**The defect, precisely:** the payload dict is appended at `:563-573`; classification happens
at `:583-593`. Adding the trend keys to the literal at `:563` would record the value from the
*previous* run.

**Steps:**
1. **RED** — new test file, following the wiring-test conventions in
   `tests/test_usage_trend_wiring.py` (in-memory SQLite + `StaticPool`, patched
   `get_db_session`, module reload per class, fixed `now`):
   - **AC3:** after one `recompute_usage_scores` run, for every customer the snapshot row's
     `usage_trend_state` equals the `customer_usage` row's `usage_trend_state`.
     *This test must fail if the keys are added to the `:563` literal* — write it so it would
     catch the naive fix, not just the missing column. Seed a customer whose state **changes**
     on this run (e.g. stored `stable`, classifies to `declining`); a customer whose state is
     unchanged cannot distinguish the two implementations.
   - **AC4:** a customer transitioning on day N has day N's snapshot carrying the new state and
     day N-1's carrying the old.
2. **GREEN** — bind the appended dict to a local and amend it after classification:
   ```python
   snapshot_row = { ...existing keys... }
   snapshot_rows.append(snapshot_row)
   ...
   new_trend_state, new_trend_pct = classify_usage_trend(...)   # :586
   snapshot_row["usage_trend_state"] = new_trend_state
   snapshot_row["usage_trend_pct"] = new_trend_pct
   ```
   Place the amendment immediately after the `classify_usage_trend` call and **before** the
   `if trend_state_changed ...` block, so the snapshot records the newly classified value
   whether or not it changed. `snapshot_rows` holds references, so mutating `snapshot_row`
   updates the queued entry.
3. Extend the comment at `:556-562` to state that trend keys are filled in after classification
   and why the literal cannot carry them.
4. **Key-uniformity check:** `_write_usage_history_snapshots` uses `bulk_insert_mappings`
   (`:264-274`), which silently omits columns absent from a mapping rather than erroring. Every
   dict must carry both keys — they are set unconditionally on the path above, and there is no
   `continue` between the append and the amendment (the early `continue` is at `:596`, after
   both). Assert this in the test by checking a *non-transitioning* customer also has a
   non-`NULL` snapshot state.

**Validation:**
```bash
cd services/worker-service && source venv/bin/activate
pytest tests/test_usage_trend_snapshot_state.py tests/test_usage_trend_wiring.py tests/test_usage_metrics.py -v
```
**Commit:** `fix(snapshot-trend-columns): snapshot records post-classification trend state`

## 4. Edge Cases & Error Handling

- **Pre-existing snapshot rows** — stay `NULL` on both columns. No backfill (spec, out of scope).
  Downstream (`timeline-trend-event`) must treat `NULL` as unknown, never as a transition.
- **Same-day re-run** — `_write_usage_history_snapshots` skips `(org, email)` pairs that already
  have a row for `today` (`:254-268`). A same-day re-run therefore does **not** correct an
  earlier row's trend state. Acceptable (the task is daily and idempotent by design) but note it
  in the test docstring so the behavior is deliberate rather than discovered.
- **Snapshot write failure** — already caught, rolled back and logged at `:627-633` without
  undoing the trend commit. Unchanged by this aspect; it is the accepted PRD R1 divergence.
- **Multi-tenancy** — the loop is intentionally all-orgs (`:532`); `organization_id` is carried
  on every snapshot dict. No route/API surface changes here, so no `organization_id` validation
  to add.
