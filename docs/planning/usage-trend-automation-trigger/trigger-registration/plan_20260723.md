### Phase 1 — The severity helper (shared source of truth)

**Goal:** one canonical answer to "is this transition strictly worsening?"

**Files:**
- `services/backend-api/src/services/usage_trend_severity.py` (create)
- `services/backend-api/tests/test_usage_trend_severity.py` (create)

**Why its own module:** the worker needs identical logic and **cannot import backend-api code**
(PRD F3). Isolating it in a small dependency-free module means `worker-trend-evaluator`
duplicates one tiny file with a "keep in sync" header, exactly as `usage_score_service.py`
already does (`usage_score_service.py:1`). Do **not** bury this inside `automation_engine.py` —
that would force the worker to duplicate the engine.

**Steps:**
1. **RED** — write `tests/test_usage_trend_severity.py` covering the full transition matrix.
2. **GREEN** — implement:
   ```python
   # DUPLICATED: keep in sync with the worker-service copy
   # (services/worker-service/src/services/usage_trend_severity.py)

   TREND_SEVERITY = {"stable": 0, "declining": 1, "sharp_decline": 2}
   # NOTE: "insufficient_history" is deliberately absent — it means "unknown",
   # not "healthy". Its absence is what makes the baseline-seed rule work.

   def is_worsening_transition(old_state, new_state) -> bool:
       old_rank = TREND_SEVERITY.get(old_state)
       new_rank = TREND_SEVERITY.get(new_state)
       if old_rank is None or new_rank is None:
           return False
       return new_rank > old_rank
   ```
   `None`, `insufficient_history`, and any unrecognised string all fall out as `False` through
   the same `.get()` path — one rule, no special cases.

**Validation:** `pytest tests/test_usage_trend_severity.py -v`
**Commit:** `feat(trigger-registration): usage-trend severity ordering helper`

### Phase 2 — Engine checker + dispatch

**Goal:** `AutomationEngine` evaluates a `usage_trend` rule.

**Files:**
- `services/backend-api/src/services/automation_engine.py` (modify)
- `services/backend-api/tests/test_automation_engine_usage_trend.py` (create)

**Steps:**
1. **RED** — new test file modeled on `tests/test_automation_engine_churn_trigger.py`,
   asserting the spec's AC1-AC5 through `engine._check_trigger(rule, context)`:
   - fires: `stable→declining`, `stable→sharp_decline`, `declining→sharp_decline`
     (when the new state is in `cfg["states"]`)
   - does not fire: same-state, both improvements, **every** `insufficient_history` pairing in
     both directions, missing/`None` on either side

     (e.g. config `["sharp_decline"]`, transition `stable→declining`)
2. **GREEN** — add after `_trigger_churn_probability` (`automation_engine.py:326-331`):
   ```python
   def _trigger_usage_trend(self, cfg: dict, context: dict) -> bool:
       """Fire on a strictly-worsening trend transition into a configured state.

       Edge-triggered: an already-declining customer produces no transition, so
       unlike churn_probability_threshold this needs no activation-time cooldown
       seeding. Any transition touching `insufficient_history` is a baseline
       observation, not a change, and never fires (PRD M2).
       """
       states = cfg.get("states") or []
       old_state = context.get("old_trend_state")
       new_state = context.get("new_trend_state")
       if new_state not in states:
           return False
       return is_worsening_transition(old_state, new_state)
   ```
3. Add the dispatch branch at `_check_trigger` (`:208-225`), after the
   `churn_probability_threshold` line:
   ```python
   if t == "usage_trend":
       return self._trigger_usage_trend(cfg, context)
   ```
4. Add the context shape to the class docstring's trigger list (`:76-81`):
   `usage_trend: {"old_trend_state": str, "new_trend_state": str, "customer_email": str}`

   - `states: ["declining"]` → 201, and the response echoes the trigger

2. **GREEN**:
   - add `"usage_trend"` to `VALID_TRIGGER_TYPES` (`automations.py:49-55`)
   - add the config schema beside the others (`:104-118`):
     ```python
     VALID_USAGE_TREND_STATES = frozenset({"declining", "sharp_decline"})

     class UsageTrendConfig(BaseModel):
         states: List[str] = Field(..., min_length=1)

         @field_validator("states")
         @classmethod
         def validate_states(cls, v: List[str]) -> List[str]:
             invalid = [s for s in v if s not in VALID_USAGE_TREND_STATES]
             if invalid:
                 raise ValueError(
                     f"states must each be one of {sorted(VALID_USAGE_TREND_STATES)}; "
                     f"got {invalid}. 'stable' and 'insufficient_history' cannot be "
                     f"entered as a worsening transition, so a rule targeting them "
                     f"could never fire."
                 )
             return v
     ```
     The error message explains *why*, not just *what* — an operator hitting this 422 should
     not have to read the PRD to understand it.
   - add the dispatch branch in `TriggerSchema.validate_trigger` (`:190-199`):
     `elif t == "usage_trend": UsageTrendConfig(**cfg)`

## 3. Edge Cases & Error Handling

- **Missing signal never fires** — mirrors `_trigger_churn_probability`'s
  never-fire-on-`None` behavior (`automation_engine.py:329-330`). Return `False`, never raise:
  the engine swallows per-rule exceptions (`:124-128`), so raising would be silently logged and
  indistinguishable from a non-match.
- **Unknown state strings** (e.g. a future 5th trend state) fall through the `.get()` to
  `False` — fail-closed, no crash.
- **`config.states` order and duplicates** are irrelevant to evaluation; no need to normalise.
- **Multi-tenancy:** no new query surface here. Rule scoping is the existing `evaluate()`
  org filter (`automation_engine.py:108-116`).

## 4. Agent Execution Notes

- **Do NOT extend `seed_churn_cooldowns`.** It is deliberately churn-only (PRD M2). If a test
  seems to want it, the test is wrong — edge-triggering is what replaces it.
- **Do NOT add a `direction` field** to `UsageTrendConfig` for symmetry with the churn/health
  configs. Direction is implied by "worsening"; a locked-to-one-value field is noise (and the
  two existing ones exist only for historical reasons).
- The severity helper's **module location is load-bearing** — `worker-trend-evaluator` will
  duplicate that exact file. If you move it, update that aspect's spec.
- **Ask before proceeding if:** any existing automation test fails after Phase 2 — the dispatch
  addition should be purely additive, so a failure means something unexpected shares that path.
