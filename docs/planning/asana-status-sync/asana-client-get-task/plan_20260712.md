# Implementation Plan — asana-status-sync / asana-client-get-task

**Date:** 2026-07-12 · **Branch:** `feat/asana-status-sync`
**PRD:** `../prd.md` · **Spec:** `./spec.md`
**Execution:** strict TDD (RED→GREEN→REFACTOR), subagent-driven — one agent may own all three
phases, or split Phase 1 (backend) from Phases 2–3 (worker); the two services never share files.

This aspect adds the **read** capability the poller needs — nothing more. No Celery, no DB, no
routes, no models. Three deliverables:

1. **Backend** `AsanaClient.get_task(task_gid)` — extend the existing client, reuse its error
   taxonomy + fixed-host invariant, **no** `Retry-After` (backend client intentionally lacks it).
2. **Worker** `AsanaClient.get_task(task_gid)` — a NEW standalone client at
   `services/worker-service/src/clients/asana.py`, scoped to `get_task` only, mirroring the worker
   Jira client's throttle handling: on 429, read `Retry-After`, `time.sleep`, then raise
   `AsanaTransientError`. Worker-side error classes mirror the backend ones.
3. **Pure adapter** `asana_category(completed: bool) -> "done" | "new"` — no I/O, unit-testable,
   living where the future worker sync task will import it.

## 0. Project Setup / Impact Analysis

Existing codebase — no scaffolding. Services touched:

| Service | Change |
|---|---|
| `services/backend-api` | Add one method `get_task` to `src/services/asana_client.py`; extend `tests/test_asana_client.py` |
| `services/worker-service` | New `src/clients/asana.py` (client, scoped to `get_task`, with `Retry-After`); new `src/services/asana_adapter.py` (`asana_category`); new `tests/test_asana_client_worker.py` |
| `services/frontend-web` | **untouched** |
| `services/analysis-engine` | **untouched** |

**No new dependencies** (`httpx`, `pytest` already present in both services). **No new `.env` keys.**
**No DB / migration** — this aspect is independent of `model-migrations` and can be built in parallel
with it (spec §Dependencies).

**Facts fixed by the upstream Asana API (do not re-derive at build time):**
- Endpoint: `GET /tasks/{gid}` on the **fixed host** `https://app.asana.com/api/1.0`
  (backend `AsanaClient.BASE_URL`; worker client copies the same constant + `_assert_safe_host`).
- `opt_fields` is **required** — Asana omits `completed_at` and `memberships` from the default task
  representation. Exact value (order-preserved, comma-joined, no spaces):
  `completed,completed_at,memberships.section.name`.
- Response envelope wraps the payload in `{"data": {...}}` — parse `resp.json()["data"]`
  (use `.get("data", {})` defensively, matching `validate()`/`create_task` in the existing client).
- **There is NO batch endpoint.** Asana has no JQL-style `issue in (...)`; the poller does **one GET
  per linked task gid**. This method fetches a single task; batching/chunking is intentionally out of
  scope (spec §Out of scope, PRD §Nice-to-have deferred).

**Return contract (both backend and worker copies, identical):**
```python
{
    "completed": bool,          # data["completed"]
    "completed_at": str | None, # data.get("completed_at") — None for incomplete tasks
    "memberships": list,        # data.get("memberships") or []  (fetched, NOT interpreted in slice 1)
}
```
`memberships` is fetched (for the v2 section-name → `indeterminate` mapping) but slice 1 never reads
it for category — see the adapter in Phase 3.

## Phase 1 — Backend `AsanaClient.get_task`  ⟂ parallel with Phases 2–3

- **Goal:** read a single Asana task's completion state through the existing backend client,
  reusing its error taxonomy and fixed-host invariant.
- **Prereq:** none.
- **Files:**
  - `services/backend-api/src/services/asana_client.py` (modify — add one method)
  - `services/backend-api/tests/test_asana_client.py` (modify — add a `TestAsanaClientGetTask` class)
- **Steps:**
  1. **RED** — add `class TestAsanaClientGetTask` to `test_asana_client.py`, mirroring the existing
     `TestAsanaClientGetProjects`/`TestAsanaClientCreateTask` style (patch `httpx.Client`, no network,
     reuse the module-level `_make_resp` helper). Add fixtures near the other response constants:
     ```python
     GET_TASK_COMPLETED = {"data": {
         "gid": "1300000000001", "completed": True,
         "completed_at": "2026-07-10T14:03:00.000Z",
         "memberships": [{"section": {"name": "Done"}}],
     }}
     GET_TASK_INCOMPLETE = {"data": {
         "gid": "1300000000001", "completed": False,
         "completed_at": None, "memberships": [{"section": {"name": "In progress"}}],
     }}
     ```
     Assertions:
     - **Endpoint + opt_fields:** `instance.get.assert_called_once_with("/tasks/1300000000001",
       params={"opt_fields": "completed,completed_at,memberships.section.name"})`.
     - **Parsing (completed):** `result == {"completed": True, "completed_at":
       "2026-07-10T14:03:00.000Z", "memberships": [{"section": {"name": "Done"}}]}`.
     - **Parsing (incomplete → completed_at None):** on `GET_TASK_INCOMPLETE`,
       `result["completed"] is False` and `result["completed_at"] is None`.
     - **`memberships` absent → `[]`:** on `{"data": {"gid": "...", "completed": True}}`,
       `result["memberships"] == []` (never raise / never `None`).
     - **Error taxonomy** (parametrize, mirror `TestAsanaClientErrorTaxonomy`, driving `get_task`):
       401/403 → `AsanaAuthError`; 429/500/502/503 → `AsanaTransientError`; 404 →
       `AsanaNotFoundError`. These fall straight out of the shared `_handle_response`; the test just
       confirms `get_task` routes through it. **No `Retry-After` behavior is asserted here** — the
       backend client does not read it (that is the worker client's job, Phase 2).
  2. **GREEN** — implement `get_task` on `AsanaClient`, immediately after `create_task`:
     ```python
     def get_task(self, task_gid: str) -> dict:
         """Fetch a single task's completion state via
         GET /tasks/{gid}?opt_fields=completed,completed_at,memberships.section.name.

         Returns {"completed": bool, "completed_at": str|None, "memberships": list}
         parsed from resp.json()["data"]. There is no batch endpoint — the poller
         calls this once per linked task gid.

         Raises: AsanaAuthError (401/403), AsanaTransientError (429/5xx),
                 AsanaNotFoundError (404).
         """
         resp = self._get(
             f"/tasks/{task_gid}",
             params={"opt_fields": "completed,completed_at,memberships.section.name"},
         )
         data = resp.json().get("data", {})
         return {
             "completed": data.get("completed"),
             "completed_at": data.get("completed_at"),
             "memberships": data.get("memberships") or [],
         }
     ```
     No change to `_handle_response`, `_get`, `_assert_safe_host`, or `BASE_URL` — reuse as-is.
  3. **REFACTOR** — none expected; confirm docstring notes the no-batch invariant.
- **Validation:**
  `cd services/backend-api && source venv/bin/activate && pytest tests/test_asana_client.py -v`
  (all pre-existing tests still green + the new `TestAsanaClientGetTask`).
- **Commit:** `feat(asana-status-sync): AsanaClient.get_task (backend read method)`

## Phase 2 — Worker `AsanaClient` (new, get_task only, with `Retry-After`)  ⟂ parallel with Phase 1

- **Goal:** a standalone worker-owned Asana client the Celery poller can import (it cannot import
  backend-api), scoped to `get_task`, that on a 429 honors `Retry-After`, sleeps, and raises a
  transient error the Celery task will retry on — the one behavior the backend client lacks.
- **Prereq:** none. (The future `worker-sync-task` aspect depends on this, not vice versa.)
- **Files:**
  - `services/worker-service/src/clients/asana.py` (new — mirror `src/clients/jira.py`)
  - `services/worker-service/tests/test_asana_client_worker.py` (new — mirror
    `tests/test_jira_client_worker.py`)
- **Design (mirror `clients/jira.py` structure, but Bearer/fixed-host like the backend Asana client):**
  - Module-level `import time` (so tests can patch `src.clients.asana.time.sleep`).
  - Error classes, worker-side mirrors of the backend taxonomy:
    `AsanaError(Exception)`, `AsanaAuthError`, `AsanaTransientError`, `AsanaNotFoundError` — each
    subclassing `AsanaError`. (Independent definitions; the worker cannot import backend-api.)
  - `class AsanaClient`:
    - `BASE_URL = "https://app.asana.com/api/1.0"` and `_assert_safe_host(base_url)` copied verbatim
      from the backend client (scheme must be `https`; host, lowercased + `rstrip(".")`, must equal
      `app.asana.com`) — preserves the fixed-host invariant.
    - `__init__(self, api_token)` — `_assert_safe_host(self.BASE_URL)`; store `self._api_token`
      (NEVER logged / repr'd); build `httpx.Client(base_url=BASE_URL,
      headers={"Authorization": f"Bearer {api_token}"}, timeout=15.0, follow_redirects=False)`.
    - Context manager (`__enter__`/`__exit__`/`close`) and `__repr__`/`__str__` returning
      `"<AsanaClient(base_url=...)>"` (token-free) — copied from the backend client.
    - `_handle_response(resp)` — **order matters**, mirroring `clients/jira.py:96`:
      ```python
      if resp.status_code == 429:
          retry_after = int(resp.headers.get("Retry-After", "10"))
          logger.warning("asana_client: 429 rate limited; sleeping %ss", retry_after)
          time.sleep(retry_after)
          raise AsanaTransientError(f"Asana rate limited, retry after {retry_after}s")
      if resp.status_code >= 500:
          raise AsanaTransientError(f"Asana server error {resp.status_code}")
      if resp.status_code in (401, 403):
          raise AsanaAuthError(f"Asana auth error (status {resp.status_code})")
      if resp.status_code == 404:
          raise AsanaNotFoundError("Asana resource not found (status 404)")
      if resp.status_code != 200:
          raise AsanaError(f"Asana request failed with status {resp.status_code}")
      return resp
      ```
    - `_get(self, path, **kwargs)` → `self._handle_response(self._client.get(path, **kwargs))`.
    - `get_task(self, task_gid)` — **identical parsing/return to Phase 1** (same `opt_fields`, same
      `{"completed", "completed_at", "memberships"}` dict, same `.get("data", {})`).
- **Steps:**
  1. **RED** — write `tests/test_asana_client_worker.py`, mirroring `test_jira_client_worker.py`.
     Reuse its `_make_resp(status_code, json_data, headers)` helper (note: `headers` defaults `{}`).
     Test classes:
     - `TestConstruction`:
       - `base_url` kwarg == `"https://app.asana.com/api/1.0"`.
       - `headers["Authorization"] == "Bearer <token>"`.
       - `timeout == 15.0`; `follow_redirects is False`.
       - token never in `repr`/`str`.
       - context manager + direct `close()` call `instance.close()` once.
     - `TestErrorTaxonomy` (**the critical `Retry-After` cases — patch `time.sleep`**):
       - `test_429_with_retry_after_sleeps_and_raises_transient`:
         `_make_resp(429, {}, headers={"Retry-After": "5"})`; patch
         `src.clients.asana.time.sleep`; `get_task("1300000000001")` raises `AsanaTransientError`;
         `mock_sleep.assert_called_once_with(5)`.
       - `test_429_missing_retry_after_defaults_to_10`: headers `{}` →
         `mock_sleep.assert_called_once_with(10)`, raises `AsanaTransientError`.
       - `test_5xx_raises_transient_without_sleep` (parametrize 500/502/503): raises transient;
         `mock_sleep.assert_not_called()` (also patch `time.sleep` here to assert it is NOT called).
       - `test_401_403_raises_auth_error` (parametrize) → `AsanaAuthError`.
       - `test_404_raises_not_found_error` → `AsanaNotFoundError`.
       - `test_error_classes_inherit_from_asana_error`.
     - `TestGetTask`:
       - endpoint + opt_fields: `instance.get.assert_called_once_with("/tasks/1300000000001",
         params={"opt_fields": "completed,completed_at,memberships.section.name"})`.
       - parses completed / completed_at / memberships (reuse the Phase 1 `GET_TASK_COMPLETED`
         / `GET_TASK_INCOMPLETE` shapes); `memberships` absent → `[]`.

## Phase 3 — Pure adapter `asana_category`  (no I/O)  ⟂ can land with Phase 2

- **Goal:** the one pure function that maps Asana completion state to a `status_sync_core` category
  key, so the future poller does `category = asana_category(task["completed"])` before feeding
  `decide_link_update`.
- **Prereq:** none (pure function; no import of client/DB/core).
- **Location decision:** `services/worker-service/src/services/asana_adapter.py`.
  - **Why here:** the `worker-sync-task` aspect runs inside `services/worker-service` and will
    `from src.services.asana_adapter import asana_category` — the same package that already holds
    `status_sync_core.py`. This is the module the consumer imports, so it is the correct home.
  - **Why NOT inside `status_sync_core.py`:** that module is the *provider-agnostic* pure core,
    copied verbatim between backend and worker; adding an Asana-specific function would break the
    "identical copy" invariant the Jira plan established. Keep provider mapping separate from the
    generic core.
  - **Why NOT the backend / `shared/`:** no backend caller needs it in slice 1 (the backend does not
    run the reconcile), and there is no shared Python package wired into the worker's import path.
    If a backend need appears later, copy the function (it is three lines) rather than couple the
    services.
- **Files:**
  - `services/worker-service/src/services/asana_adapter.py` (new)
  - `services/worker-service/tests/test_asana_client_worker.py` (modify — add `TestAsanaCategory`,
    so the aspect's worker validation command covers it too)
- **Steps:**
  1. **RED** — add to `test_asana_client_worker.py`:
     ```python
     class TestAsanaCategory:
         def test_completed_true_maps_to_done(self):
             from src.services.asana_adapter import asana_category
             assert asana_category(True) == "done"

         def test_completed_false_maps_to_new(self):
             from src.services.asana_adapter import asana_category
             assert asana_category(False) == "new"
     ```
  2. **GREEN** — implement:
     ```python
     """Pure Asana task-state -> status_sync_core category adapter (no I/O).

     Slice 1 maps only the boolean `completed` field:
       completed True  -> "done"  (status_sync_core resolves -> resolved by default)
       completed False -> "new"
     Section-name / custom-field -> "indeterminate" is deferred to v2; `memberships`
     is fetched by AsanaClient.get_task but intentionally not consulted here yet.
     """
     from __future__ import annotations


     def asana_category(completed: bool) -> str:
         """Return the status_sync_core category key for an Asana task's completion."""
         return "done" if completed else "new"
     ```
     Returned values (`"done"`, `"new"`) are exactly the keys `status_sync_core.DEFAULT_CATEGORY_MAP`
     / `CATEGORY_RANK` already understand — no core change (PRD requirement 3).
  3. **REFACTOR** — none.
- **Validation:** covered by the Phase 2 command
  (`pytest tests/test_asana_client_worker.py -v`).
- **Commit:** `feat(asana-status-sync): pure asana_category adapter (completed -> done/new)`

## Testing Strategy

- **Zero network.** Every test patches `httpx.Client` and drives the mocked `instance.get` /
  `instance.post` — identical to the existing `test_asana_client.py` and `test_jira_client_worker.py`.
- **`Retry-After` must patch `time.sleep`.** The 429 tests patch `src.clients.asana.time.sleep`
  and assert the exact sleep argument (`5` when the header is present, `10` when absent). Without the
  patch the suite would actually sleep — treat an un-patched `time.sleep` in these tests as a bug.
  The client MUST reference `time.sleep` via a module-level `import time` (not
  `from time import sleep`) so the `src.clients.asana.time.sleep` patch target resolves.
- **5xx must NOT sleep.** The 5xx transient test also patches `time.sleep` and asserts
  `assert_not_called()` — proves `Retry-After` handling is scoped to 429 only.
- **opt_fields is asserted literally.** Both client tests assert the exact
  `params={"opt_fields": "completed,completed_at,memberships.section.name"}` dict on the mocked GET —
  a typo or reordering fails the test.
- **Parsing depth.** Assert the full returned dict on a completed task, `completed_at is None` on an
  incomplete task, and `memberships == []` when the key is absent (defensive `or []`).
- **Token-leak guard.** Both clients assert the PAT never appears in `repr()`/`str()`.
- **Host invariant.** Both clients assert `_assert_safe_host` rejects non-https, non-asana, and
  suffix-trick hosts (`app.asana.com.evil.com`).
- **Adapter purity.** `asana_category` is tested with plain booleans — no mocks, no imports beyond
  the function itself.
- **Validation commands (both must be green):**
  - `cd services/backend-api && source venv/bin/activate && pytest tests/test_asana_client.py -v`
  - `cd services/worker-service && source venv/bin/activate && pytest tests/test_asana_client_worker.py -v`

## Edge Cases & Error Handling

| Case | Behavior |
|---|---|
| Incomplete task | Asana returns `completed=false`; `completed_at` is `null` → `completed_at: None`. Adapter → `"new"`. |
| `completed_at` / `memberships` absent from payload | `.get(...)` → `None` / `or [] → []`. Never raise. |
| 429 with `Retry-After: N` (worker) | `int(N)`, `time.sleep(N)`, raise `AsanaTransientError` (Celery retries). |
| 429 without `Retry-After` (worker) | Default `10`, sleep, raise transient. |
| 429 (backend) | Raises `AsanaTransientError` immediately, **no sleep** (backend `_handle_response` unchanged). |
| 5xx (both) | Raise `AsanaTransientError`, no sleep. |
| 401 / 403 (both) | `AsanaAuthError`. Poller (later aspect) records `last_sync_status=error` without disconnecting — not this aspect's concern. |
| 404 — task deleted/moved in Asana | `AsanaNotFoundError`. The poller aspect decides skip-vs-record; the client just raises the typed error. |
| Non-200, non-taxonomy status (e.g. 400) | Worker raises base `AsanaError`; backend falls through to `resp.raise_for_status()`. |
| No batch fetch | One GET per gid — do **not** add chunking/multi-gid params; the poller loops per linked task. |
| `memberships` present but unused | Fetched for v2 section mapping; slice-1 category ignores it (adapter keys only on `completed`). |
| `BASE_URL` monkeypatched to unsafe host | `_assert_safe_host` raises `ValueError` at construction (both clients). |

## Agent Execution Notes (checkpoints & decisions)

- **Two services, no shared files.** Phase 1 touches only `backend-api`; Phases 2–3 only
  `worker-service`. Safe to split across two agents; if using one, do Phase 1 fully (RED→GREEN→commit)
  before switching services.
- **Worker client is a copy, not an import.** Do **not** attempt to import
  `backend-api`'s `AsanaClient` from the worker — the services are independently deployed. Copy the
  construction/host-assert/token-hiding boilerplate; the only *new* logic vs. the backend client is
  the 429 `Retry-After` branch in `_handle_response` (lifted from `clients/jira.py`).
- **Patch target discipline.** The client must `import time` at module scope so
  `patch("src.clients.asana.time.sleep")` works. If an agent writes `from time import sleep`, the
  429 tests will actually sleep and the `assert_called_once_with` target won't resolve — reject that.
- **`opt_fields` is load-bearing.** Asana omits `completed_at`/`memberships` without it. Keep the
  exact string `completed,completed_at,memberships.section.name`; the tests assert it verbatim.
- **Return shape parity.** Backend and worker `get_task` return the *same* dict — the poller aspect
  relies on it. If you tweak one, tweak both, and update both test files.
- **Adapter home is deliberate.** `asana_category` goes in
  `services/worker-service/src/services/asana_adapter.py` (imported by the worker-sync-task aspect),
  **not** in `status_sync_core.py` (which must stay the verbatim provider-agnostic core). Do not add
  Asana specifics to the core.
- **venv note.** If the worktree lacks per-service venvs, create/activate them per the repo's
  worktree convention before running the validation commands (the main checkout has
  `services/backend-api/venv`). Tests need only `httpx` + `pytest` (both already in each service's
  requirements) — no Redis/Postgres for this aspect.
- **Scope discipline.** Do not add: batch/multi-gid fetch, section-name interpretation, any Celery
  task, DB model, or route. Those belong to `worker-sync-task` / `model-migrations` / `backend-routes`.
