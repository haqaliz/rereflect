## 0. Project Setup / Impact Analysis

Existing codebase — no scaffolding. This aspect is the **operator control surface** for Asana inbound
status-sync: it mirrors the three Jira endpoints in
`services/backend-api/src/api/routes/jira_integration.py` onto the already-present Asana route module.

| Service | Change |
|---|---|
| `services/backend-api` | Extend `AsanaStatusResponse`; add `AsanaStatusSyncUpdateRequest` + `AsanaSyncTriggerResponse`; add `_validate_status_mapping`, `_last_status_synced_at`, `_build_status_response`, `_get_celery_app` helpers; extend `GET /status`; add `PATCH /status-sync` + `POST /sync` routes; new test file |
| `services/worker-service` | **untouched by this aspect** — `POST /sync` only `send_task(...)`s the name `src.tasks.asana_sync.sync_asana_org`; the task itself is the `worker-sync-task` aspect |
| `services/frontend-web` | **untouched by this aspect** — but the `frontend` aspect depends on the exact request/response shapes finalized here (see §Contract Freeze) |
| `services/analysis-engine` | **untouched** |

**No new dependencies** (fastapi, pydantic, sqlalchemy all present). **No new `.env` keys**
(reuses `LLM_ENCRYPTION_KEY`, `DATABASE_URL`, Redis broker config — all already read by the existing
Asana connect/test routes and `celery_client.py`).

**Hard dependency (assumed already landed — `model-migrations` aspect):**
- `AsanaIntegration.status_sync_enabled` (Boolean, NOT NULL, `server_default false`, default False)
- `AsanaIntegration.status_mapping` (JSON, nullable)
- `FeedbackAsanaTask.last_status_synced_at` (DateTime, nullable)

If any of these three columns is missing when this aspect starts, **stop and flag** — the routes read
`row.status_sync_enabled` / `row.status_mapping` and aggregate `MAX(FeedbackAsanaTask.last_status_synced_at)`,
and the test fixtures construct rows with these fields. Do not add the columns here (that is the other
aspect's file ownership: `src/models/asana_integration.py` + `alembic/`).

**Parity invariants carried from Jira (do not diverge):**
- `_validate_status_mapping` keys ∈ `{new, indeterminate, done}`, values ∈ `{new, in_review, resolved, closed}`
  → 422. Kept **byte-identical** to Jira's semantics for parity; `indeterminate` is accepted for
  forward-compat but is **inert for Asana** (the completion→category adapter only ever emits `new`/`done`).
- `PATCH` 404s on **no integration row at all** (uses `_get_org_integration`, not active-only — the
  setting lives on the row, so a row must exist first).
- `POST /sync` requires an **active** integration (400 if none), enqueues via `send_task`, and reports
  a broker/dispatch failure as a clean **502** — never a 500.
- The `api_token` (and its plaintext) is **NEVER** serialized in any response (only `token_hint`).

## Phase 4 — GREEN: `PATCH /status-sync` + `POST /sync`

- **Goal:** the toggle/mapping route and the manual-trigger route.
- **Prereq:** Phases 2–3.
- **Files:** `services/backend-api/src/api/routes/asana_integration.py` (add two routes, after the
  create-task section or in a dedicated "Inbound status-sync operator controls" block — mirror Jira's
  layout).
- **Steps:**
  1. **`PATCH /status-sync`** — mirror `jira_update_status_sync`:
     ```python
     @router.patch(
         "/status-sync",
         response_model=AsanaStatusResponse,
         dependencies=[Depends(require_admin_or_owner)],
     )
     def asana_update_status_sync(payload: AsanaStatusSyncUpdateRequest,
                                  current_org=Depends(get_current_org),
                                  db: Session = Depends(get_db)):
         row = _get_org_integration(db, current_org.id)   # any row, active or not
         if not row:
             raise HTTPException(404, "No Asana integration found. Connect Asana first.")
         _validate_status_mapping(payload.status_mapping)   # 422 on bad key/value
         row.status_sync_enabled = payload.enabled
         if payload.status_mapping is not None:
             row.status_mapping = payload.status_mapping
         row.updated_at = datetime.utcnow()
         db.commit(); db.refresh(row)
         return _build_status_response(db, current_org.id, row)
     ```
     Note ordering nuance: validate **after** the 404 check (so a missing integration is 404 even with
     a bad mapping) — matches Jira. Only assign `status_mapping` when provided (a toggle-only PATCH must
     not wipe an existing mapping).
  2. **`POST /sync`** — mirror `jira_trigger_sync`, `status_code=202`,
     `response_model=AsanaSyncTriggerResponse`:
     ```python
     integ = _get_active_integration(db, current_org.id)
     if not integ:
         raise HTTPException(400, "No active Asana integration.")
     try:
         _get_celery_app().send_task("src.tasks.asana_sync.sync_asana_org", args=[integ.id])
     except Exception as exc:  # noqa: BLE001 — never surface a 500
         logger.error("Asana status sync could not be enqueued for org %s (integration_id=%s): %s",
                      current_org.id, integ.id, exc)
         raise HTTPException(502, "Asana status sync could not be enqueued. Please try again shortly.")
     return AsanaSyncTriggerResponse(status="queued")
     ```
     `require_admin_or_owner` on both; `organization_id` scoping via `get_current_org` (cross-org can
     never reach org A's row → 400).
- **Validation:** `pytest tests/test_asana_status_sync_routes.py -v` → **all GREEN** (PATCH toggle +
  mapping persist, 404, 422 key/value, 403 member, indeterminate-accepted; POST 202/400/502, 403,
  cross-org).
- **Commit:** `feat(asana-status-sync): status-sync toggle + manual /sync routes`

## Contract Freeze (the `frontend` aspect depends on these exact shapes)

- `PATCH /api/v1/integrations/asana/status-sync`
  - **Request:** `{ "enabled": bool, "status_mapping"?: { <category>: <workflow_status> } }`
  - **Response:** `AsanaStatusResponse` (200). `404` no row · `422` bad key/value · `403` non-admin.
- `POST /api/v1/integrations/asana/sync`
  - **Request:** empty body. **Response:** `{ "status": "queued" }` (202). `400` no active integration ·
    `502` dispatch failure · `403` non-admin.
- `GET /api/v1/integrations/asana/status` — `AsanaStatusResponse` now additionally carries
  `status_sync_enabled: bool` and `last_status_synced_at: datetime | null`. No field removed/renamed.
- **Never present in any response:** `api_token` (or the plaintext token). `token_hint` only.

## Edge Cases & Error Handling

- **PATCH toggle-only** (no `status_mapping`) must **not** clear a previously-saved mapping — assign
  `status_mapping` only when the field is provided.
- **PATCH on inactive-but-existing row** → allowed (404 only when *no* row exists; the setting persists
  on the row regardless of `is_active`) — matches Jira.
- **`indeterminate` mapping key** → accepted (200) for parity/forward-compat, but inert for Asana; do
  not special-case-reject it (that would diverge from Jira and the spec).
- **POST /sync with an inactive integration** → 400 (only *active* integrations enqueue).
- **Broker/dispatch raises anything** → 502, logged, never a 500 (broad `except Exception`).
- **Cross-org** → `get_current_org` scopes every query; org B can never enqueue org A's sync.
- **Token safety** → assert-tested: no response serializes `api_token` or the raw token value.
- **Default-off** → a fresh integration reports `status_sync_enabled: false`; no route implicitly
  enables sync.
