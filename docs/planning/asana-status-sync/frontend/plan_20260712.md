**Feature:** asana-status-sync · **Aspect:** frontend · **Service:** `services/frontend-web`
**Execution:** strict TDD (Vitest). **Sequence AFTER backend-routes** (needs the final request/response shapes).

- **Steps (TDD — write the failing tests first):**
  1. **Test:** `patchAsanaStatusSync(true)` calls `apiClient.patch('/api/v1/integrations/asana/status-sync',
     { enabled: true })` and returns `response.data`; assert `result.status_sync_enabled === true`. (Mirror
     `jira.test.ts:155-178`.)
  2. **Test:** `patchAsanaStatusSync(true, { done: 'resolved' })` includes `status_mapping` in the body:
     `{ enabled: true, status_mapping: { done: 'resolved' } }`. (Mirror `jira.test.ts:180-187`.)
  3. **Test:** `triggerAsanaSync()` calls `apiClient.post('/api/v1/integrations/asana/sync')` and returns
     `{ status: 'queued' }`. (Mirror `jira.test.ts:189-194`.)
  4. **Implement** in `lib/api/asana.ts`:
     - Extend `AsanaConnectionStatus` (currently ends at `connected_at` on line ~14) with, and carry over the
       Jira doc-comment adapted to Asana:
       ```ts
       // Inbound status sync (asana-status-sync). Opt-in poller that pulls
       // Asana task completion back onto linked feedback. `last_sync_status` /
       // `last_error` above are shared with the general connection status but
       // are also updated by the status-sync poller.
       status_sync_enabled: boolean;
       last_status_synced_at: string | null;
       ```
     - Add `export interface AsanaSyncTriggerResponse { status: string; }` (next to the other response types).
     - Add module-scope exports **after** the `asanaAPI` object (matching Jira's standalone-export placement):
       ```ts
       export async function patchAsanaStatusSync(
         enabled: boolean,
         statusMapping?: Record<string, string>
       ): Promise<AsanaConnectionStatus> {
         const response = await apiClient.patch('/api/v1/integrations/asana/status-sync', {
           enabled,
           ...(statusMapping !== undefined ? { status_mapping: statusMapping } : {}),
         });
         return response.data;
       }

       export async function triggerAsanaSync(): Promise<AsanaSyncTriggerResponse> {
         const response = await apiClient.post('/api/v1/integrations/asana/sync');
         return response.data;
       }
       ```
- **Validation:** `cd services/frontend-web && npm run test -- asana` (the existing `asana.test.ts` already mocks
  `@/lib/api-client` with a `patch` fn, so no harness change needed) and `npm run lint`.

## Edge cases & error handling
- **Disconnected:** card returns `null` (double-guarded: page condition + component early return).
- **Optimistic revert:** on PATCH failure the switch snaps back to `previous` and a toast shows the server `detail`
  (or a generic fallback when `detail` isn't a string).
- **502 vs other Sync-now failures:** 502 → "background worker is unavailable" copy; any other error → server
  `detail` or generic "Could not start sync." — identical to Jira.
- **`last_error` visibility:** only rendered when `last_sync_status === 'error'`, so a stale `last_error` left over
  from a prior failed poll is hidden once a later poll succeeds.
- **Never state:** `last_status_synced_at === null` → literal "Never" (first-poll-not-yet-run and never-enabled both
  read as "Never", which is correct).
- **Theming:** only CSS variables / existing utility classes (`text-destructive`, `text-muted-foreground`,
  `border-border`, `animate-slide-up`) — no hardcoded hex, per project guidelines.
