**1b. Insert a new subsection `### Syncing Asana status back to feedback (opt-in)`** immediately **after** the Asana **"### Verify"** block (ends ~line 777, "…surfaces the existing linked task instead of duplicating it.") and **before** `### Known limitation: team-scoped projects` (~line 779). Mirror the shape of the Jira subsection at `SELF_HOSTING.md:607–633` — same bullet structure, same plain tone — but with Asana-accurate facts:

- **Off by default.** Turn it on under **Settings → Integrations → Asana** with the status-sync toggle (admin/owner). The tile shows the last sync time and any error, plus a **Sync now** button.
- **Poll-based (works behind a firewall).** A background job checks your linked tasks every ~15 minutes over the same personal access token — **no webhook or public URL required**, so it works on a self-hosted box behind NAT. (Real-time Asana webhooks are **not** part of this release.)
- **Completion-based mapping — Asana has only two states.** An Asana task is either **completed** or **not**, so Rereflect maps:
  - not completed → `new`
  - completed → `resolved`
  Default mapping is `{done: resolved, new: new}`. There is **no `in_review` / intermediate state** from Asana in this release — unlike Jira, Asana exposes no "in progress" status category to a linked task. (Section-name / custom-field mapping that could add an intermediate state is a planned v2 follow-up.)
- **Remap per-organization.** Override the mapping via `PATCH /api/v1/integrations/asana/status-sync` with `{ "enabled": true, "status_mapping": { "done": "closed" } }` (admin/owner) — e.g. resolve to `closed` instead of `resolved`. Invalid keys/values are rejected (422).
- **Non-destructive & bidirectional.** Turning it on does **not** retroactively rewrite feedback you already linked — the first poll records current task state as a baseline and only moves a feedback item when the task *changes* afterward. If a task is later **re-opened** in Asana (completed → not completed), the linked feedback reverts back toward `new`, with a timeline entry recording the change (`source=asana`). If a feedback item is linked to several Asana tasks, the most-advanced status wins.

Close with a short forward-looking line mirroring Jira's ("Real-time webhook sync … planned for a future release"), naming the retained deferrals: real-time Asana webhooks, section/custom-field → intermediate mapping, and OAuth.

> **Do not** touch the Asana **"### All features unlocked"** block (~lines 790–794) beyond confirming it still reads correctly with status-sync now present — status-sync is likewise unlocked (no plan gate), consistent with the Jira "issue creation and status-sync have no plan gate" wording at `SELF_HOSTING.md:637–639`.
