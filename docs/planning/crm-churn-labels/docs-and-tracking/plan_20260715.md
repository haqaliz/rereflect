### Phase 2 — `docs/SELF_HOSTING.md` (the load-bearing doc)

**Goal.** An operator goes from "CRM connected" to "suggestions accumulating", and is told honestly
that **review is not yet available**.
**Prerequisites.** Phase 1. Read `HubSpotChurnLabelsCard.tsx` + `SalesforceChurnLabelsCard.tsx` **in
full** — every string must match the shipped UI.
**Files.** `docs/SELF_HOSTING.md` (insert `## CRM churn-label suggestions` after the Salesforce
writeback section, ~line 690 — verify at write time).
**Steps.**
1. **Intro.** What it does: reads **lost renewal** deals/opportunities from the CRM you already
   connected and records them as **suggestions**. Not labels. State up front: **a lost renewal is not
   always a churn** (PRD R3) — which is why a human has to confirm.
2. **Ordered setup (AC-1, reduced).** (a) Connect HubSpot or Salesforce (link the existing sections;
   one CRM per org). (b) Settings → Integrations → <provider> → **CRM Churn-Label Suggestions** →
   toggle **Enable churn-label suggestions**. (c) Pick **Renewal pipelines** (HubSpot) /
   **Opportunity types** (Salesforce) from the live picker. (d) Note: **selection locks while
   enabled** (`:69`) — disable to change it. (e) Harvest runs inside the existing daily CRM sync
   (03:15 HubSpot / 03:45 Salesforce UTC) — **no new schedule**.
3. **DEFAULT-DENY (AC-2), its own subsection.** Nothing is suggested until you name pipelines/types.
   Enabled with none selected → **still nothing** (the card warns, `:131`). A deal whose
   pipeline/type is null or unrecognised produces **no suggestion, ever** — no guessing, no regex on
   pipeline names. An org that ignores this feature sees **no change anywhere**.
4. **What you get today — honest limitation, non-negotiable.** Suggestions land in the
   `churn_label_suggestions` table and appear as **two counters**: `suggestions_created` on the
   settings card and **"pending suggestions"** on Settings → AI → Readiness. **The review queue is
   not built yet, so suggestions cannot yet be confirmed and nothing reaches the calibrator.**
   Say it plainly; do not describe confirm/reject/bulk UI (§0). **Do not mention backfill** (AC-3
   void — none exists).
5. **Also: how it *will* train (no promises).** When confirmed by a human, a suggestion writes a
   `manual` churn event the calibrator trains on. Frame as design intent, not a shipped button.
6. **Verify.** Settings card stats: last harvest time, status, suggestions created, last error
   (`:236-258`).
7. **Troubleshooting.** No suggestions → pipelines/types unconfigured (most likely), or no lost
   renewals in the window. 403 → CRM scope. 429 → throttled, retried next sync. Per-run cap **200**
   (`PER_RUN_SUGGESTION_CAP`); overflow is logged with a `dropped_by_cap` count — **never silent**.
8. **No accuracy claims anywhere.** Churn prediction stays "a calibrated heuristic". No ML.
**Validation.** `bash scripts/check_docs_honesty.sh` — SELF_HOSTING now passes AC-4/AC-5. Reviewer
walks the doc against the shipped UI.

## 6. Edge cases & error handling

- **§0 goes stale.** Review-queue/backfill may merge between plan and execution → **re-run the §1

- **The gate over-matches.** "accuracy" is legitimate in `AI-TRACKING.md`'s existing AI-Accuracy rows
  → diff-scoping (Phase 1.3) handles it. Never weaken the gate to pass; narrow the scope.
- **Documenting a button that doesn't exist.** The single biggest risk. Every UI string in Phase 2
  must be greppable in `HubSpot|SalesforceChurnLabelsCard.tsx`. If you can't grep it, don't write it.
- **Copy drifts optimistic.** "Automatically detects churn" writes itself on a landing page. It is
  false twice over: default-deny, and human-confirmed. The gate blocks the phrase.
- **The 500 gate restated as fact.** Easiest slip in Phase 4. R8 caveat travels with every mention.
- **Stripe/Segment label sources.** Dead (no Stripe post-pivot) / blocked (no usage history).
  **Never promised in docs or landing.**
