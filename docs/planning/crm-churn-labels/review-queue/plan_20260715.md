### Phase 2 — Confirm + reject (the trust boundary)

**Goal.** Confirm writes a trainable label; reject resolves without one.
**Prerequisites.** Phase 1.
**Files.** `src/api/routes/churn_suggestions.py`, `tests/test_churn_suggestions_api.py`.
**Steps.**
1. RED — AC3 (one `CustomerChurnEvent`, `source=='manual'`, `marked_by_user_id ==` confirming
   user, `churned_at == suggestion.suggested_churned_at`; suggestion `status=='confirmed'` **and**
   `churn_event_id == event.id`; `CustomerHealth.probability_computed_at IS NULL`), AC4 (missing /
   invalid `reason_code` → 422, nothing written, still `pending`), AC5 both paths, AC6.
2. GREEN — private `_confirm_one(db, org, user, suggestion, reason_code, reason_text) ->
   tuple[str, Optional[str], Optional[int]]` returning `(status, reason, churn_event_id)`. Used by
   both the single and the bulk route — one write path, one collision policy.
   ```
   if suggestion.status != "pending":            -> ("skipped", "not_pending", suggestion.churn_event_id)
   # Defense 1 — pre-check (racy, hence defense 2)
   existing = _get_active_churn_event(db, org.id, suggestion.customer_email)   # churn_events.py:74-87
   if existing: resolve(existing) -> ("skipped", "already_marked", existing.id)
   event = CustomerChurnEvent(..., marked_by_user_id=user.id, source="manual")   # mirrors :350-358
   try: db.add(event); db.flush()
   except IntegrityError:
       # Defense 2 — backstop. Caller owns rollback/SAVEPOINT release.
       raise _Collision()   # resolved by the caller to ("skipped","already_marked", <pre-existing id>)
   ```
   **`_confirm_one` never commits and never rolls back** — the callers do. This is what lets bulk
   wrap it in a SAVEPOINT without the helper fighting the outer transaction.
3. `resolve(event)` (spec R-B, non-negotiable): sets `status='confirmed'`,
   `churn_event_id = event.id` (the **pre-existing** event), `reviewed_by_user_id`, `reviewed_at`.
   The suggestion **leaves the queue**. This deliberately attributes an event this action did not
   create — accepted, because leaving it `pending` re-surfaces an unactionable row every day. The
   honesty lives in the wire (`reason: "already_marked"`) and the toast, not in the DB. Put this
   paragraph in the function's docstring.
4. Single confirm route: call `_confirm_one`; on success `db.commit()` then
   `_invalidate_probability(db, org.id, email)` — **after** the commit, exactly as `:371` does
   (the helper commits internally at `:102`). On `_Collision`: `db.rollback()`, re-resolve against
   the now-visible event, `db.commit()`. Return `{id, status, churn_event_id, reason?}`.
   **Never 500. Never 409** (that is the old route's contract, not ours).
5. Reject route: `status='rejected'`, `reviewed_by_user_id`, `reviewed_at`, `churn_event_id` stays
   NULL, no event, no `_invalidate_probability`. Non-`pending` → `skipped`.
   `RejectRequest.note` is **persisted into nothing today** — the model has no note column. Do not
   silently discard it like `RecoverRequest.note` does (Finding 5 / spec Out-of-scope): accept the
   field per spec §3 and write it into `evidence["review_note"]` is **out of scope** (mutating
   harvested evidence is a lie about provenance). Instead: ignore it and **say so in the
   docstring** as a known gap, with a TODO naming a `review_note` column as the fix. Flag in §7.
**Validation.** Scoped pytest green, incl. AC5 race path via `monkeypatch.setattr(
churn_suggestions, "_get_active_churn_event", lambda *a, **k: None)` — the ONE patch the spec
permits (Dependencies: "no patching but AC5"); assert response is `skipped`, not 500, and that a
subsequent write on the same session succeeds (session usable).
**Commit.** `feat(review-queue): confirm writes CustomerChurnEvent(source='manual'); reject resolves`

### Phase 3 — Bulk confirm/reject

**Goal.** `POST /churn-suggestions/bulk`, best-effort per item, public-crud-v3 result shape.
**Prerequisites.** Phase 2 (`_confirm_one` exists).
**Files.** `src/api/routes/churn_suggestions.py`, `tests/test_churn_suggestions_api.py`.
**Steps.**
1. RED — AC7 (42 pendings, 1 collides → `{matched:42, confirmed:41, skipped:1}`, 41 events,
   `len(results)==42`, deduped input order), AC8 (duplicate email deduped & `matched` counts once;
   both/neither of `emails`|`filter` → 422; `confirm` without `reason_code` → 422 nothing written;
   a `filter` cohort touches only `pending` rows of the caller's org), plus the cap test below.
2. GREEN — `_resolve_cohort(db, org, cohort) -> list[ChurnLabelSuggestion]`:
   - `emails`: `[e.strip().lower() for e in cohort.emails]` deduped **order-preserving** via the
     `seen`-set idiom (`public_api.py:341-351, 492-497`); query this org's **pending** suggestions
     for those emails; return sorted by input position, then `id`.
   - `filter`: this org's `pending` (or `filter.status`) rows matching `provider` / `search`
     (email substring, as the GET does), ordered `suggested_churned_at DESC, id DESC`.
3. **Cap (spec R-C — non-negotiable, non-silent).** `BULK_REVIEW_CAP = 500`, module constant.
   Rationale: matches the shipped public bulk's `ids: Field(..., max_length=500)`
   (`public_api.py:366`) and the PRD's never-built 500/day figure, so the number is a house
   precedent rather than a fresh invention. Enforcement:
   - `emails` cohort: >500 deduped emails → **422** `"cohort exceeds BULK_REVIEW_CAP (500); narrow
     the selection"`. The caller chose the list; refusing is honest and fixable.
   - `filter` cohort: resolve, then `matched = <full count>` and act on the first 500. Response
     carries `matched` (the true count) — so `matched > confirmed + skipped + len(errors)` **only**
     in the capped case, and the surplus is visible, never silent. Add
     `capped: bool` + `cap: int` to `BulkReviewResponse` so the UI can say "500 of 731 processed —
     run again for the rest". ⚠️ This **extends** the spec §6 shape (which mandates `confirmed +
     skipped + errors == len(results) == matched`); the invariant holds exactly when
     `capped is False`. Additive fields only — no shipped field renamed or dropped. Flagged §7.
4. Per-item loop, mirroring `public_api.py:554-583`:
   ```
   for s in resolved:
       try:
           with db.begin_nested():          # SAVEPOINT — one collision must not roll back the other 40
               status, reason, ev_id = _confirm_one(...)   # or _reject_one(...)
           results.append(...)
       except _Collision:
           # SAVEPOINT already rolled back by the ctx manager; the pre-existing
           # event is visible again -> resolve to skipped/already_marked
       except Exception:
           logger.warning(..., exc_info=True); results.append(status="error", reason="internal_error")
   db.commit()
   ```
   Then `_invalidate_probability` once per distinct confirmed email, **after** `db.commit()`.
5. `action='reject'` → `reason_code` ignored; `confirmed` counts rejections (docstring, spec §6).
   `action='confirm'` without `reason_code` → 422 at the Pydantic layer (`model_validator`), so
   nothing is written before validation.
**Validation.** Scoped pytest green. Assert `results` order == deduped input order explicitly.
**Commit.** `feat(review-queue): bulk confirm/reject over a suggestion cohort (capped at 500)`

## 6. Edge cases & error handling

- **Collision (non-negotiable).** Dual defense, both required: pre-check `_get_active_churn_event`
  (`churn_events.py:74-87`) **and** `except IntegrityError` backstop. Pre-check alone is racy;
  catch alone leaves the session dirty mid-bulk. Resolve to `skipped` / `reason:"already_marked"`.
  **Never 500. Never a partial-bulk abort.** The collided suggestion **resolves out of the queue**
  (R-B) — deliberate mis-attribution, documented in the docstring and surfaced on the wire.
- **Non-pending target.** Confirm/reject on a `confirmed`/`rejected` row → `skipped`,
  `reason:"not_pending"`. Idempotent; no double event.
- **Cross-org id.** 404 (single) / silently absent from a resolved cohort (bulk) — a cross-org id
  must never be distinguishable from a missing one.
- **Suggestion whose `churn_event_id` target was deleted.** FK is `ON DELETE SET NULL`
  (`churn_label_suggestion.py:66-70`) → a `confirmed` row can legitimately carry a NULL
  `churn_event_id`. The UI must not assume non-null on `confirmed`.
