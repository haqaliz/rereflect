# Implementation Plan — backend-connection (2026-07-06)

**Aspect:** `asana-integration/backend-connection` · **Service:** `services/backend-api`
**Precedent to mirror:** `jira_integration` (model/client/routes) and `zendesk_integration` (2nd subdomain client). Asana = Jira minus SSRF/subdomain, plus Bearer PAT.

## 1. Setup checklist
- Service: `services/backend-api` only.
- No new packages (reuse `httpx`, `cryptography`/Fernet via `src/utils/encryption.py`).
- **DB change: yes** — one new migration adding `asana_integrations` (+ `feedback_asana_tasks`, co-created for the create-task aspect).
- Env: reuse existing `LLM_ENCRYPTION_KEY` (already required for Jira). No new keys.

## 3. File build order
1. `src/models/asana_integration.py` — models `AsanaIntegration`, `FeedbackAsanaTask`.
2. `src/models/__init__.py` — register.
3. `alembic/versions/{rev}_add_asana_integration_tables.py` — tables.
4. `src/services/asana_client.py` — `AsanaClient`, error classes.
5. `src/api/routes/asana_integration.py` — router (connect/status/disconnect/test/workspaces/projects).
6. `src/api/main.py` — include router.

## 4. Testing strategy
`tests/test_asana_models.py`, `tests/test_asana_client.py`, `tests/test_asana_connection.py` (mirror the Jira trio). Mock all HTTP (`unittest.mock` on `httpx.Client` / route-level `AsanaClient`). Command: `cd services/backend-api && pytest tests/test_asana_*.py -v`, then full `pytest tests/ -v`.

## 5. Environment & secrets
Reuse `LLM_ENCRYPTION_KEY` (already in `.env`). No new secrets. Migration applied via `alembic upgrade` against the local DB (note the multi-head caveat).

## 6. Edge cases & error handling
- Missing `LLM_ENCRYPTION_KEY` → 422, never 500 (the R6 safeguard).
- Reconnect rotates the token (upsert by org).
- Disconnect is soft; preserves `feedback_asana_tasks`.
- All queries scoped by `organization_id`.
- Token never logged, never in responses, omitted from `repr`/`str`.
