**Implement:** `lib/api/saml.ts` ‚Äî clone `lib/api/oidc.ts`:

```ts
import apiClient from '../api-client';

export interface SamlStatus {
  enabled: boolean;
  button_label: string;
}

// Response shape. The X.509 signing cert is PUBLIC (not a secret). Match the
// backend contract: it returns EITHER the full PEM (idp_x509_cert) OR a
// fingerprint (cert_fingerprint). Model both as nullable; render whichever is present.
export interface SamlConfig {
  configured: boolean;
  idp_entity_id: string | null;
  idp_sso_url: string | null;
  idp_x509_cert: string | null;      // full PEM if backend returns it
  cert_fingerprint: string | null;   // fingerprint hint if backend returns that instead
  email_attribute: string | null;
  enabled: boolean;
  allowed_email_domains: string[];
  button_label: string | null;
}

export interface SamlConfigUpdate {
  idp_entity_id: string;
  idp_sso_url: string;
  idp_x509_cert?: string;            // optional on update IF backend supports keep-on-blank; else required
  email_attribute?: string;
  enabled: boolean;
  allowed_email_domains: string[];
  button_label?: string;
}

export const getSamlStatus = async (): Promise<SamlStatus> => {
  const response = await apiClient.get('/api/v1/auth/saml/status');
  return response.data;
};
export const getSamlConfig = async (): Promise<SamlConfig> => {
  const response = await apiClient.get('/api/v1/settings/saml');
  return response.data;
};
export const putSamlConfig = async (payload: SamlConfigUpdate): Promise<SamlConfig> => {
  const response = await apiClient.put('/api/v1/settings/saml', payload);
  return response.data;
};
export const deleteSamlConfig = async (): Promise<{ success: boolean; message: string }> => {
  const response = await apiClient.delete('/api/v1/settings/saml');
  return response.data;
};
```

> **Contract lock:** before finalizing the interfaces, open the `config-model-and-crud` plan/response
> model and copy the exact field names. If it returns `cert_fingerprint` and NOT the full PEM, keep both
> fields nullable but drive the form off `cert_fingerprint` (hint) + a write-only PEM textarea (¬ß5).

## 5. Config-form cert handling detail (write-only vs prefill)

Two backend contracts are possible; the form must match whichever `config-model-and-crud` ships:

- **(A) Backend returns `cert_fingerprint` only (recommended, mirrors OIDC secret_hint):** treat the PEM
  textarea as **write-only**. Show the fingerprint in helper text; send `idp_x509_cert` in the PUT only
  when the textarea is non-empty; blank = keep existing. This is the safest default and the plan's primary
  path.
- **(B) Backend returns the full `idp_x509_cert`:** the cert is public, so prefilling the textarea is
  acceptable; always send it back on PUT. Simpler, no keep-on-blank logic.

**Decision rule:** open the sibling `config-model-and-crud` response model; if it exposes
`cert_fingerprint`, implement (A); else implement (B). The `SamlConfig` interface in `lib/api/saml.ts`
carries **both** nullable fields so the client compiles either way; the form picks the branch. Document
which branch was taken in a code comment so reviewers can trace it back to the backend contract.

## 6. Merged `sso_error` lookup behavior

- Both maps intentionally share `disabled`, `domain`, `config`, `denied` with **identical text** ‚Üí order
  of lookup doesn't matter for those.
- SAML-only codes (`signature`, `assertion`, `audience`, `recipient`, `expired`, `replay`, `unsolicited`)
  exist only in the SAML map ‚Üí must fall through to `getSamlErrorMessage`.
- OIDC-only codes (`state`, `token`, `exchange`) exist only in the OIDC map ‚Üí must resolve via
  `getSsoErrorMessage`.
- The merge in ¬ß4-Phase-D handles all three: try OIDC; if it returned the generic fallback, the code isn't
  OIDC-known, so use the SAML map (which itself falls back to a generic string for truly unknown codes).
- `unverified` differs slightly in wording between the two maps ‚Äî both are friendly and non-leaky; the
  merge resolves it via the OIDC map (OIDC-known), which is fine. If you want the SAML wording for SAML
  flows you cannot tell protocol from the code alone (the query param carries no protocol tag) ‚Äî accept
  the OIDC wording for the shared code; it's still correct.

## 7. Edge cases & correctness checks

1. **Exactly one SSO button** ‚Äî both `OidcSignInButton` and `SamlSignInButton` are mounted on `/login`,
   each self-hides unless its own status probe returns `enabled:true`. The backend M6 one-protocol guard
   guarantees at most one protocol is enabled, so at most one button ever renders. If **both** are somehow
   enabled (guard bug), both would show ‚Äî acceptable degradation, and out of this aspect's control. Verify
   the intended path with the button tests (each renders independently).
2. **Cert PEM vs fingerprint** ‚Äî see ¬ß5. The form must not display a "stored" fingerprint as if it were an

3. **Do not regress OIDC** ‚Äî the OIDC Card, its state, its tests all stay byte-identical (child-component

4. **Two "Save"/"Disconnect" buttons on the page** ‚Äî every test (old and new) must scope button queries to

7. **Fail-open probe** ‚Äî `SamlSignInButton` must render nothing (not throw) when `getSamlStatus` rejects

8. **No Suspense needed** ‚Äî the login page reads `window.location.search` directly (not

9. **`allowed_email_domains` empty = deny-all** ‚Äî preserve the OIDC semantics and warning; don't silently
   treat empty as "allow all".
