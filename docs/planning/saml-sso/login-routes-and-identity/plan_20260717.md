## 1. Scope & boundary

This aspect wires the **SP-initiated SAML login flow** into the reused OIDC identity-resolution block:

```
GET /api/v1/auth/saml/login   →  302 to IdP (AuthnRequest, RelayState, request_id persisted pending)
      … browser authenticates at IdP …
POST /api/v1/auth/saml/callback (ACS)  →  validate assertion → consume InResponseTo →
      resolve identity (subject → verified-email link → JIT member) → mint JWT →
      302 {FRONTEND_URL}/login/callback#token=<jwt>   (or  302 {FRONTEND_URL}/login?sso_error=<code>)
```

**In scope (this aspect):** two routes in `services/backend-api/src/api/routes/auth.py`, the SAML
`sso_error` constants + `SAML_ACS_PATH`, the identity-resolution reuse, router wiring confirmation in

### 2.1 Provider (`src/services/saml_provider.py`)
```python
class SamlValidationError(Exception):
    code: str   # one of: "signature" | "assertion" | "audience" | "recipient" | "expired"

@dataclass
class ValidatedAssertion:
    subject: Optional[str]          # NameID; may be None → we reject (null-subject guard)
    email: Optional[str]            # mapped email attribute / NameID emailAddress
    in_response_to: Optional[str]   # the SAML Response @InResponseTo (for replay-store consume)
    attributes: dict

class SamlProvider:
    @classmethod
    def from_config(cls, saml_config) -> "SamlProvider": ...
    def build_authn_request(self, relay_state: str) -> tuple[str, str]:   # (redirect_url, request_id)
        ...
    def validate_response(self, saml_response_b64: str, *,
                          expected_in_response_to: Optional[str],
                          acs_url: str) -> ValidatedAssertion:
        ...   # raises SamlValidationError(code=...) on any crypto/condition failure
```

**Division of responsibility (reconciles the PRD's `unsolicited|replay` codes vs. a pure provider):**
- The **provider** is DB-free. It validates signature, XSW-resistance, `Audience`, `Recipient`/
  `Destination`, `NotBefore`/`NotOnOrAfter` (±60s), `Issuer`, and that an `InResponseTo` is *present*.
  Its `SamlValidationError.code` ∈ `{signature, assertion, audience, recipient, expired}`.
- The **ACS route (this aspect)** owns `replay` and `unsolicited` — they require the DB replay store,
  which the pure provider cannot see. So the ACS calls `validate_response(..., expected_in_response_to=None)`
  (the provider returns `ValidatedAssertion.in_response_to`; a missing/empty InResponseTo is treated by
  the provider as `assertion`), then does the store consume to derive `replay` / `unsolicited` / `expired`.

  > **Seam to confirm (§9):** whether the provider signature takes `expected_in_response_to` at all, or
  > returns `in_response_to` for the route to consume. Plan assumes the latter (route-owned replay). If
  > the provider instead wants the expected value passed in, the ACS has no cookie to source it from, so
  > the route-owned-consume design stands; align the provider aspect to it.

### 2.2 Replay store (`saml_auth_requests`, owned by provider-and-replay-store)
The ACS anti-replay relies on a **DB-backed, SQLite-testable** store with one-time conditional consume.
This aspect calls two helpers (import from wherever the provider aspect exposes them — likely
`src/models/saml_auth_request.py` or `src/services/saml_replay.py`):

```python
def record_auth_request(db, *, request_id: str, organization_id: int, expires_at: datetime) -> None:
    """Insert a PENDING row at /saml/login (opportunistic cleanup of expired rows)."""

def consume_auth_request(db, request_id: str) -> str:
    """Conditional `UPDATE ... WHERE consumed_at IS NULL AND expires_at > now`.
    Returns one of: "ok" | "replay" | "unsolicited" | "expired". Race-safe (exactly-one wins)."""
```
- `"ok"` → proceed to identity resolution.
- `"replay"` → `sso_error=replay`; `"unsolicited"` → `sso_error=unsolicited`; `"expired"` →
  `sso_error=expired` (or `unsolicited` — see §7 edge cases; plan uses `expired`).

> **Seam to confirm (§9):** exact helper names + return shape. If the provider aspect exposes only the
> model, this aspect adds the two thin helpers in that model's module (not a new model).

**SAML `sso_error` codes** (superset of OIDC's; frontend `getSsoErrorMessage` extends its map in the
frontend aspect):

| Code | Emitted when |
|---|---|
| `disabled` | no enabled `SamlConfig` at `/login` or ACS |
| `config` | `SamlProvider.from_config` / `build_authn_request` raised `SsrfError`/`ValueError` |
| `state` | RelayState missing / bad HMAC / TTL-expired at ACS |
| `denied` | (optional) IdP posts a non-Success `StatusCode` — provider raises `assertion`; map to `denied` if surfaced |
| `signature` | assertion unsigned / wrong cert / XSW → `SamlValidationError("signature")` |
| `assertion` | malformed response, missing InResponseTo, bad conditions shape → `SamlValidationError("assertion")` |
| `audience` | `Audience` ≠ SP entity id |
| `recipient` | `Destination`/`Recipient` ≠ ACS URL |
| `expired` | `NotOnOrAfter`/`NotBefore` outside ±60s, OR replay-store `expired` |
| `replay` | InResponseTo already consumed |
| `unsolicited` | InResponseTo unknown to the replay store |
| `unverified` | assertion carries no email attribute (SAML has no `email_verified`; a signed email is trusted) |
| `domain` | email domain not in `allowed_email_domains` (empty list = deny-all) |
| `token` | **null/empty subject** (NameID) — the account-takeover guard |

The `SamlValidationError.code` → `sso_error` mapping is 1:1 for
`signature|assertion|audience|recipient|expired`; the ACS wraps `validate_response` in a single
`except SamlValidationError as exc: return _saml_error_redirect(exc.code)`.

### 4.1 `GET /api/v1/auth/saml/login` — SP-initiated start

```python
@router.get("/saml/login")
def saml_login(db: Session = Depends(get_db)):
    config = db.query(SamlConfig).filter(SamlConfig.enabled.is_(True)).first()
    if not config:
        return RedirectResponse(f"{_frontend_url()}/login?sso_error=disabled", status_code=302)
    try:
        provider = SamlProvider.from_config(config)
        relay_state = sign_state(hash_nonce(secrets.token_urlsafe(32)))
        redirect_url, request_id = provider.build_authn_request(relay_state)
        record_auth_request(
            db,
            request_id=request_id,
            organization_id=config.organization_id,
            expires_at=datetime.utcnow() + timedelta(seconds=STATE_TTL_SECONDS),
        )
    except (SsrfError, ValueError) as exc:
        logger.error("SAML login start failed for config %s: %s", config.id, exc)
        return RedirectResponse(f"{_frontend_url()}/login?sso_error=config", status_code=302)
    return RedirectResponse(redirect_url, status_code=302)
```
Notes:
- **No cookie** is set (unlike OIDC). The cross-site POST back to the ACS would not carry a `SameSite=Lax`
  cookie anyway; the anti-forgery controls are the signed assertion + one-time InResponseTo consume, not a
  session cookie. RelayState is a signed nonce-hash (freshness/HMAC proof only, no identity) — see §6.
- `record_auth_request` must be inside the `try` so a DB error also degrades to `sso_error=config`,
  never a 500.
- Never 500s: missing/disabled config and any provider/SSRF/value error redirect with a generic code.
- `timedelta` import added alongside `datetime`.

### 4.2 `POST /api/v1/auth/saml/callback` — Assertion Consumer Service (ACS)

**Auth-exempt, unauthenticated, form-POST.** No `Depends(get_current_user)`; there is no global auth
middleware in `main.py` (auth is per-route), so simply omitting the dependency makes it public. No CSRF
token — by design (see §6 anti-forgery rationale, which MUST be a code comment on this handler).

```python
@router.post("/saml/callback")
def saml_callback(
    SAMLResponse: str = Form(...),
    RelayState: Optional[str] = Form(None),
    db: Session = Depends(get_db),
):
    def _saml_error_redirect(code: str) -> RedirectResponse:
        # Base is ALWAYS the fixed configured FRONTEND_URL — never request/IdP-derived
        # (open-redirect guard). No user is created/linked on any error path.
        return RedirectResponse(f"{_frontend_url()}/login?sso_error={code}", status_code=302)

    # (1) RelayState: HMAC + TTL only. It carries a nonce hash, NO identity/trust.
    #     Anti-forgery = signed assertion (step 2) + one-time InResponseTo consume (step 3).
    if verify_state(RelayState) is None if RelayState else True:
        return _saml_error_redirect("state")

    # (2) Enabled config required to know the cert/entityId to validate against.
    config = db.query(SamlConfig).filter(SamlConfig.enabled.is_(True)).first()
    if not config:
        return _saml_error_redirect("disabled")

    # (3) Validate the SAML Response (signature, XSW, Audience, Recipient/Destination,
    #     NotBefore/NotOnOrAfter ±60s, Issuer, InResponseTo present). Pure crypto — no DB.
    acs_url = f"{_backend_url()}{SAML_ACS_PATH}"
    try:
        provider = SamlProvider.from_config(config)
        assertion = provider.validate_response(
            SAMLResponse, expected_in_response_to=None, acs_url=acs_url
        )
    except SamlValidationError as exc:
        logger.error("SAML assertion validation failed for config %s: code=%s", config.id, exc.code)
        return _saml_error_redirect(exc.code)
    except (SsrfError, ValueError) as exc:
        logger.error("SAML provider error for config %s: %s", config.id, exc)
        return _saml_error_redirect("config")

    # (4) Replay/unsolicited: conditional one-time consume of the InResponseTo.
    outcome = consume_auth_request(db, assertion.in_response_to)
    if outcome != "ok":
        return _saml_error_redirect(outcome)   # "replay" | "unsolicited" | "expired"

    # (5) Identity resolution — mirrors OIDC auth.py L492-552 (see §5).
    email = assertion.email
    if not email:
        return _saml_error_redirect("unverified")
    email = email.lower()
    subject = assertion.subject
    if not subject:                          # KEEP null-subject rejection (account-takeover guard)
        return _saml_error_redirect("token")

    allowed_domains = [d.lower() for d in (config.allowed_email_domains or [])]
    if email.rsplit("@", 1)[-1] not in allowed_domains:   # empty list = deny-all
        return _saml_error_redirect("domain")

    user = db.query(User).filter(User.saml_subject == subject).first()
    if not user:
        user = db.query(User).filter(func.lower(User.email) == email).first()
        if user:
            user.saml_subject = subject
            if user.auth_provider in ("email", "google", "oidc"):
                user.auth_provider = "both"
            db.commit(); db.refresh(user)
        else:
            user = User(
                email=email, password_hash=None, saml_subject=subject,
                auth_provider="saml", organization_id=config.organization_id,
                role="member", joined_at=datetime.utcnow(),
            )
            db.add(user); db.commit(); db.refresh(user)

    access_token = create_access_token({
        "user_id": user.id, "organization_id": user.organization_id, "role": user.role,
    })
    # (6) Token via URL FRAGMENT on the fixed FRONTEND_URL — reuse /login/callback (already public
    #     in both frontend allowlists; do NOT introduce a new callback path).
    return RedirectResponse(
        f"{_frontend_url()}/login/callback#token={access_token}", status_code=302
    )
```

> The `verify_state(...) is None if ... else True` line above is pseudocode shorthand; implement as an
> explicit guard: `if not RelayState or verify_state(RelayState) is None: return _saml_error_redirect("state")`.

**Ordering rationale (security-critical):** validate signature/conditions (3) **before** the replay
consume (4) so an attacker's forged/garbage POST can never consume a legitimate pending `request_id`
(a DoS on a real user's in-flight login). Consume happens only after the assertion is cryptographically
trusted. Both precede identity resolution (5).

## 5. Identity resolution (mirror of OIDC `auth.py` L492–552)

Same shape as OIDC, three swaps and one omission:
- `sub` → `assertion.subject` (SAML NameID); DB column `oidc_sub` → `saml_subject`.
- `claims.get("email")` → `assertion.email`.
- **No `email_verified` gate** — SAML has no such field; a validly-signed assertion's email is trusted
  (the IdP asserts it). Document this in a comment. The `unverified` code now means *no email present*,
  not *unverified email*.
- **Keep the null-subject rejection verbatim in intent:** `if not subject: return ...token`. Without it,
  `User.saml_subject == None` compiles to `WHERE saml_subject IS NULL`, matching every existing
  password/Google/OIDC user (all have `saml_subject IS NULL`) and `.first()` would mint a token for the
  lowest-id row (the org owner) — account takeover. This is the single most important line in the block.
- **Link precedence:** `saml_subject` match first (a returning SSO user whose IdP email changed), then
  case-insensitive email link (`func.lower(User.email) == email`), else JIT `member`.
- **`auth_provider` bump on link:** an existing `email`/`google`/`oidc` user becomes `"both"` (mirrors
  OIDC, which bumps `email`/`google` → `both`; SAML adds `oidc` to the set that gets promoted). Confirm
  the config aspect's chosen `auth_provider` vocabulary (§9) — if it uses a distinct `"saml"`/`"both"`
  convention, match it exactly.
- **Org scoping:** JIT user attaches to `config.organization_id` (never request-derived). The JWT claims
  carry `organization_id` from the resolved user, enforcing multi-tenancy downstream.

## 6. Anti-forgery rationale (gap 2 — MUST be a code comment + a test)

Put this as a docstring/comment on `saml_callback`, and assert it via `test_forged_unsigned_post_rejected`:

> The ACS is auth-exempt and CSRF-exempt **by design**. A browser returning from the IdP POSTs
> cross-site, so a `SameSite=Lax` session cookie would not be sent — a cookie-based CSRF token is
> structurally impossible here. The trust controls are: (1) the **XML signature** over the assertion,
> verified against the operator-configured IdP cert (an attacker cannot mint a valid signature), and
> (2) the **one-time `InResponseTo` consume** against the server-side replay store (a captured assertion
> cannot be replayed, and an unsolicited/IdP-initiated response with an unknown InResponseTo is rejected).
> `RelayState` is an HMAC-signed nonce hash with **no identity and no trust weight** — it only proves the
> redirect was minted by our `/saml/login` within the TTL (defense in depth). Forging or omitting it
> yields `sso_error=state`; forging the assertion yields `sso_error=signature`; replaying yields `replay`.

## 10. Testing strategy

- **Boundary-mock the crypto, real-DB the store.** Patch `src.api.routes.auth.SamlProvider` (not the
  service module) so the route's own branching is exercised; use the `db` fixture's real SQLite session
  for `saml_auth_requests` so replay/unsolicited/expired are genuine. This mirrors OIDC's pattern
  (`_StubCallbackProvider` + real `User` rows) and keeps this aspect's tests independent of `xmlsec`
  being installed in the test env.
- **No real signed assertions needed here.** Real signature/XSW/condition rejection is proven in the
  provider aspect (`test_saml_provider.py`) against real fixtures. This aspect asserts only that each
  `SamlValidationError.code` maps to the right `sso_error` and that no user is created on failure.
- **User-count invariant.** Every error-path test asserts `db.query(User).count()` is unchanged (the
  OIDC tests' discipline) — the strongest guard against a resolution-order regression.
- **JWT assertion.** Happy-path tests decode the `#token=` fragment with `decode_access_token` and assert
  `user_id/organization_id/role`, proving org-scoping end-to-end.
- **Optional integration test (flag, may live in provider aspect):** one full `/saml/login` → real
  `build_authn_request` → capture `request_id` → feed a **real signed** `SAMLResponse` fixture (throwaway
  key / Keycloak-captured) to the ACS → assert token. This is the only test that needs the provider
  aspect's shared signed-assertion fixtures; gate it behind an `xmlsec`-available marker so the suite
  stays green where the native lib is absent.

## 12. Edge cases & guards (explicit)

- **Open-redirect:** every redirect base is `_frontend_url()` (fixed env), never request/IdP/`RelayState`-
  derived. `RelayState` is never used to build a URL. Tested by the fixed-URL fragment assertion.
- **`/login/callback` reuse:** success redirects to the **existing** `/login/callback#token=` — already
  public in both frontend allowlists (`api-client.ts`, `AuthContext.tsx`). Do **not** add a SAML-specific
  callback path (spec risk note).
- **Empty allowlist = deny-all:** `allowed_email_domains or []` then membership test; `[]` denies every
  domain (matches OIDC). Explicitly tested.
- **Null/empty subject:** rejected with `token` before any `saml_subject ==` query — the account-takeover
  guard; two tests (missing-subject, and belt-and-braces the resolution never runs).
- **Org scoping:** JIT user gets `config.organization_id`; JWT claims carry the resolved user's
  `organization_id`. No cross-tenant leakage possible since only one enabled config exists per deployment
  (single-enabled guard, config aspect) and identity binds to that config's org.
- **Auth-exempt ACS:** the POST route has no `get_current_user` dependency; confirmed there is **no**
  global auth middleware in `main.py` (auth is per-route). Do not add app-level auth that would break it.
- **CSRF-exempt ACS:** intentional (§6). No CSRF middleware exists in the app today; do not add one that
  would block this route.
- **Never-500:** both routes trap `SsrfError`/`ValueError`/`SamlValidationError` and DB errors into a
  generic `sso_error=` redirect. No `HTTPException` with IdP/validation detail ever reaches the browser
  (detail stays in `logger.error`).
- **Replay consume ordering:** consume happens only *after* signature validation, so a forged POST cannot
  burn a legitimate user's pending `request_id`.
- **Expired vs unsolicited:** an expired-but-known request maps to `expired`; a never-issued id maps to
  `unsolicited`. If the store collapses both to a single outcome, mirror its choice and update the test
  (flag to align with the provider aspect's `consume_auth_request` return values).
