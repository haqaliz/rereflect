### Phase 2 — GREEN: pin the dependency in `requirements.txt`

**File:** `services/backend-api/requirements.txt` (EDIT — append a new grouped block, mirroring the
existing commented-section style)

Add, after the OIDC block (keep the SSO deps visually together):

```
# SAML 2.0 SSO (saml-sso feature — SP-initiated; validate assertions, do not hand-roll XML-DSig)
# python3-saml pulls native xmlsec (libxmlsec1/libxml2) — see services/backend-api/Dockerfile apt layer.
# xmlsec is PINNED for ABI compatibility with the Debian 12 (bookworm) base image's libxmlsec1 1.2.37:
# xmlsec >= 1.3.14 requires libxmlsec1 >= 1.3.0 and will NOT load against bookworm. Do not unpin blindly.
python3-saml==1.16.0         # OneLogin SP-side SAML — import path: onelogin.saml2.*
xmlsec==1.3.13               # Native XML-DSig/enc binding; last line compatible w/ libxmlsec1 1.2.x (bookworm)
lxml>=4.9.0,<6               # python3-saml/xmlsec XML backend (prebuilt wheels; must match xmlsec's lxml ABI)
```

**Version rationale (state assumptions, verify at install time):**
- `python3-saml==1.16.0` — the current stable OneLogin release line at plan time; pin exactly.
- `xmlsec==1.3.13` — **pinned below 1.3.14** specifically to keep the `libxmlsec1 >= 1.3.0` requirement
  from breaking on bookworm's 1.2.37. If a clean install fails to build/import, the fallback is to move the
  base image or the pin — NOT to unpin (§6).
- `lxml` — range, not exact: lxml ships manylinux wheels, so it installs without headers, and `xmlsec`
  links against whatever lxml ABI resolves. Keep `<6` to avoid a surprise major bump changing the C ABI
  `xmlsec` was built against.

**Verify (local venv, authoritative-enough for the pin):**
```bash

pip install -r requirements.txt         # must resolve in a clean venv (spec AC #3)
```

```dockerfile
    libxml2-dev \
    libxmlsec1 \
    libxmlsec1-dev \
```

## 5. Testing strategy

**The one test this aspect adds** — `services/backend-api/tests/test_saml_deps.py`:

```python
"""
Dependency smoke test for the SAML 2.0 SSO feature (saml-sso, slice 1 / deps-and-docker).

This is the RED->GREEN anchor for the whole feature: python3-saml pulls the native `xmlsec`
(libxmlsec1/libxml2) binding, which needs OS packages in the backend image + CI. If those are
missing, EVERY later SAML module fails to import and the suite dies at collection. This test
proves the imports work — before any SAML logic exists.

Fails before `python3-saml`/`xmlsec` are installed (requirements.txt) and the native libs are
present (Dockerfile). Passes after. Keep it dependency-only: no network, no config, no XML parsing.
"""


def test_onelogin_saml_auth_imports():
    # The exact entrypoint every SAML provider/route in later aspects will use.
    from onelogin.saml2.auth import OneLogin_Saml2_Auth  # noqa: F401
    from onelogin.saml2.settings import OneLogin_Saml2_Settings  # noqa: F401


def test_xmlsec_native_binding_imports():
    # The native binding that requires libxmlsec1/libxml2 system packages.
    import xmlsec

    # Touch the C extension so a broken/ABI-mismatched build fails loudly here,
    # not deep inside signature validation in a later aspect.
    assert xmlsec.__version__  # e.g. "1.3.13"


def test_lxml_backend_imports():
    from lxml import etree  # noqa: F401
```

**Why these three assertions:**
- `OneLogin_Saml2_Auth` / `OneLogin_Saml2_Settings` — the real import surface later aspects (M2/M3 provider,
  ACS route) depend on. If the package name or import path is wrong, this catches it now.
- `import xmlsec` + touching `__version__` — forces the **C extension to load**, which is the actual R1
  risk (a pure `pip install` can succeed yet fail to load if `libxmlsec1` is missing/ABI-mismatched at
  import). This is the assertion that would go red in a broken image.
- `lxml.etree` — the shared XML backend; cheap sanity that the lxml/xmlsec pairing loaded.

**Run matrix:**
| Where | Command | When | Authority |
|---|---|---|---|
| Local venv | `pytest tests/test_saml_deps.py -v` | Phase 2 | Proves the pin resolves + imports |
| Local venv | `pytest tests/ --collect-only -q` | Phase 4 | No-regression (collection) |
| Docker image | `docker run … python -m pytest tests/test_saml_deps.py -v` | Phase 3 | **Authoritative** for R1 (native libs) |

**Do NOT** add SAML XML fixtures, signature tests, or config tests here — those belong to later aspects
(`test_saml_provider.py`, `test_saml_login.py`, `test_saml_config.py`). Keep this file import-only so it
stays a fast, unambiguous canary.

## 7. Edge cases & risks (this slice)

1. **Alpine-vs-Debian → moot.** Base is `python:3.12-slim` = Debian 12 bookworm, so packages install via
   `apt-get` with the `libxmlsec1*`/`libxml2*` names above. The spec's Alpine `apk`/`xmlsec-dev` finicky
   path does not apply. (If the base image ever changes to Alpine, this is where it re-enters.)

2. **`xmlsec` ABI break vs bookworm `libxmlsec1` 1.2.37 — the real trap.** `xmlsec >= 1.3.14` requires
   `libxmlsec1 >= 1.3.0`, which bookworm does **not** ship (it has 1.2.37). An unpinned `pip install xmlsec`
   can pull a version that builds but then **fails to load** against the system lib, turning every SAML
   import red at runtime. **Mitigation:** pin `xmlsec==1.3.13` (last line compatible with 1.2.x). If a clean
   install still fails, the correct fixes, in order: (a) pick the newest `xmlsec` that still supports 1.2.x;
   (b) install `libxmlsec1` from bookworm-backports; (c) as a last resort move the base to a Debian release
   shipping `libxmlsec1 >= 1.3.0` (trixie) and re-test — **never** silently unpin.

3. **Wheel vs source build for `xmlsec`.** If PyPI serves a compatible cp312 manylinux `xmlsec` wheel, the
   `-dev` headers/`pkg-config`/`build-essential` aren't strictly needed to *install* — but the runtime libs
   (`libxml2`, `libxmlsec1`, `libxmlsec1-openssl`) still are, to *load* it. If pip falls back to the sdist
   (common when the pin doesn't match a prebuilt wheel), the build deps ARE required. We install both so the
   build is deterministic regardless of wheel availability. Trade-off: larger image. Documented follow-up
   (out of scope): a multi-stage build that compiles the wheel in a builder stage and copies only runtime
   libs into the final image — track separately, don't gold-plate here.

4. **`lxml` ABI coupling.** `xmlsec` links against the lxml C ABI; a major lxml bump can mismatch. Range-pin
   `lxml<6` and let a compatible wheel resolve. If `xmlsec` import raises an lxml symbol error, pin lxml to
   the version `xmlsec==1.3.13`'s wheel was built against and re-test.

5. **Docker daemon down in-worktree.** `docker build` may not be runnable here. Fall back to clean-venv
   verification and explicitly document "verified locally, image build deferred to CI/reviewer" (spec AC #2
   allows this). Do not mark the aspect done claiming a Docker build that didn't run.

6. **Pre-existing full-suite noise.** Repo memory: a segfault in `test_report_ws.py` and no eslint in
   landing-web are pre-existing. Scope the no-regression check to collection + OIDC/auth subset; don't chase
   unrelated red.

7. **Build context gotcha.** The image builds with context `./services` (`docker build -f backend-api/Dockerfile .`
   from `services/`), not from `backend-api/`. Using the wrong context breaks the existing
   `COPY analysis-engine/...` step — unrelated to SAML but will fail the build if you cd into the wrong dir.
