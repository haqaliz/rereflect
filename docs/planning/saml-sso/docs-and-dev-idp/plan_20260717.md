### Phase 1 — `SELF_HOSTING.md`: the SAML 2.0 section

- **Goal:** an operator can register Rereflect as a SAML SP, configure it, and test against Keycloak.
- **Prerequisites:** §0 verification complete (real ACS path, field names, error codes, SP EntityID
  derivation, email-attribute default, metadata-endpoint yes/no in hand).
- **File:** `docs/SELF_HOSTING.md`.
- **Insertion point:** immediately **after** the OIDC section (which ends at the `### All features
  unlocked` block around **L1321–1324**) and **before** `## Production notes` (~L1326). New top-level
  heading `## Single Sign-On (SAML 2.0)`.
- **Mirror the OIDC section structure** (L1210–1324), translated to SAML. Required sub-sections:

  1. **Intro paragraph** — SAML SP-initiated SSO alongside password + Google + (if present) OIDC;
     all features unlocked, no plan gate. State plainly: **SP-initiated only**.
  2. **`### 1. Register Rereflect as a SAML SP with your IdP`** — table:
     | Setting | Value |
     |---|---|
     | ACS (Assertion Consumer Service) URL | `{BACKEND_URL}/api/v1/auth/saml/callback` — **HTTP-POST binding** — e.g. `http://localhost:8000/api/v1/auth/saml/callback` locally. Must match exactly. |
     | SP Entity ID | *(document the exact value/derivation confirmed in §0.2 — e.g. the constant or `{BACKEND_URL}`-derived string the provider emits)* |
     | NameID format | `emailAddress` — **or** map an email attribute (see the email-attribute field below) |
     | Assertion signing | **The IdP MUST sign assertions.** Rereflect rejects unsigned assertions. (Response-level signing alone is not sufficient — the assertion itself must be signed.) |
     Add one line: works with Okta, Azure AD / Entra, OneLogin, ADFS, Google Workspace, Keycloak.
  3. **`### 2. Configure in the app`** — Settings → SSO (`/settings/sso`, admin/owner only) field
     table, using the **verified** field names from §0.2: IdP Entity ID, IdP SSO URL, X.509 signing
     cert (PEM), email attribute *(document only if the field shipped)*, allowed email domains,
     button label, enabled.
  4. **`### 3. Allowed email domains: empty means deny-all`** — reuse OIDC wording verbatim
     (L1251–1262): empty list = deny-all; warn against over-broad public domains.
  5. **`### One SSO protocol per deployment`** — enabling SAML requires OIDC **disabled**, and
     vice-versa. At most one of {OIDC, SAML} enabled per deployment; enabling the second is rejected
     (422). One SSO button ever shows on the login page.
  6. **`### JIT provisioning and account linking`** — mirror OIDC L1270–1278: new user → `member` in
     the config's org (no invite); existing verified-email account linked (never duplicated). Note
     SAML trust model: a validly-**signed** assertion's email is trusted (the IdP asserts it) — there
     is no `email_verified` claim in SAML, unlike OIDC. Keep the "no email → rejected" behavior.
  7. **`### Known limitations (slice 1)`** — bullet list, explicit and honest:
     - **SP-initiated only** — no IdP-initiated login.
     - **No Single Logout (SLO).**
     - **No SCIM / directory provisioning** (JIT on first login only).
     - **Signed, not encrypted** assertions (assertion encryption not supported).
     - **Single IdP** per deployment, **single signing cert** per config.
     - **Cert-rotation caveat:** because a config holds one cert, rotate by **pasting the new cert
       while the old one is still valid at the IdP** (overlap window), then cut the IdP over. There is
       no dual-cert grace period. If you get locked out, **owner email/password login still works** —
       it is the deliberate fallback.
  8. **`### Testing against a local Keycloak (SAML)`** — mirror OIDC L1287–1301:
     `docker compose --profile dev-idp up keycloak`; admin console `http://localhost:8080` (`admin`/`admin`);
     create a realm, add a **SAML client** in it, set the client's ACS/redirect to
     `http://localhost:8000/api/v1/auth/saml/callback` (POST binding), enable assertion signing, then
     copy the realm's SAML descriptor values (IdP Entity ID, SSO URL, signing certificate — from
     `http://localhost:8080/realms/<realm>/protocol/saml/descriptor`) into `/settings/sso`.
  9. **`### Troubleshooting`** — `?sso_error=<code>` table. Use **exactly** the codes §0.2 proved are
     emitted. Target set from the spec (prune/extend to match shipped):
     | Code | Meaning |
     |---|---|
     | `disabled` | No SAML configuration is currently enabled |
     | `config` | Building the AuthnRequest failed (bad IdP config / SSRF check) |
     | `signature` | The assertion's XML signature failed validation against the configured cert |
     | `assertion` | The assertion was malformed, unsigned, or failed structural checks (incl. XSW) |
     | `audience` | The `Audience` did not match the SP Entity ID |
     | `recipient` | The `Recipient`/`Destination` did not match the ACS URL |
     | `expired` | The assertion was expired (`NotOnOrAfter`) or not yet valid (`NotBefore`) |
     | `replay` | The assertion / response was already consumed (replay) |
     | `unsolicited` | The response's `InResponseTo` did not match a pending request (unsolicited) |
     | `unverified` | No usable email was present in the assertion |
     | `domain` | The email's domain is not in `allowed_email_domains` |
     | `denied` | The user was denied at the IdP / authentication failed |
     End with the OIDC section's reassurance line: every code is generic; no IdP/validation detail
     leaks to the browser.
  10. **`### All features unlocked`** — reuse the OIDC closing block verbatim (self-hosted, no gate).
- **Validation:**
  - `grep -n "auth/saml/callback" docs/SELF_HOSTING.md` matches the §0.2 path exactly.
  - Every `sso_error` code in the table is present in the §0.2 backend grep output (no invented codes).
  - No mention of IdP-initiated / SLO / SCIM / encrypted assertions / metadata-endpoint **unless
    that last one actually shipped**.
  - Markdown renders: headings nested correctly, table pipes balanced.
