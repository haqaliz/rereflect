## 1. Contract assumed from `backend-connection` (not owned by this aspect)

```
POST   /api/v1/integrations/zendesk/connect     {subdomain, email, api_token} → ZendeskConnectResponse
GET    /api/v1/integrations/zendesk/status      → ZendeskConnectionStatus
DELETE /api/v1/integrations/zendesk/disconnect  → ZendeskDisconnectResponse
POST   /api/v1/integrations/zendesk/test        → ZendeskTestResponse
```

`ZendeskConnectionStatus`: `{ connected, subdomain, email, token_hint, is_active, last_synced_at,
last_sync_status, last_error, connected_at, webhook_secret?, has_source? }`. `has_source` reflects
the backend's PRD-9a auto-provisioned `zendesk` `FeedbackSource` for the org (see
`backend-connection/spec.md`) — the frontend must not assume it's always `true`/present on older
payloads. `webhook_secret` is shown once (on `connect`, and optionally still on `status` until the
operator has seen it / a rotate action exists) — treat it as always-optional in types.

If the real backend lands with a different field name or omits `has_source`, this is a same-day
fix in `lib/api/zendesk.ts` only (isolated by the API-client layer) — no page-logic changes.

## 6. Edge cases & error handling

- `zendeskStatus` fetch failing (`Promise.allSettled` rejection) → status stays `null`,
  page/tile renders "Available"/disconnected, no crash (mirrors existing Jira/HubSpot behavior).
- `has_source` absent on older/partial payloads → treat as falsy, don't block the wizard (fail
  open to "show the integration step normally" rather than fail closed/hidden).
- Reconnect with a different subdomain after disconnect → connect form is not pre-filled from
  the previous `status` (mirrors Jira: `siteUrl`/`email`/`tokenInput` are local form state, only
  cleared on success, never hydrated from `status`).
- Webhook secret is optional/undefined on `status` refetch after the initial connect (backend may
  only return it once) — the "Set up real-time sync" block must degrade gracefully to "webhook
  secret already configured — reconnect to view it again" rather than rendering `undefined`.
- Never hardcode colors anywhere except the `#03363D` brand hex passed as a Tailwind arbitrary
  value in `className` (icon wrapper `bg-[#03363D]/10`, text `text-[#03363D]`) — everything else
  (badges, alerts, cards) uses existing CSS-variable-backed classes/components, per CLAUDE.md.
- Strict TypeScript: no `any` in the new `zendesk.ts` client; wizard's widened union checks use
  literal string comparisons already established by the `linear`/`jira` precedent (no enum
  refactor).
