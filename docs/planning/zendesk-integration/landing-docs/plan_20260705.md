## Goal

Flip the marketing surface for Zendesk from "coming soon" to a shipped/"available" integration, and
document operator setup for self-hosters — with zero changes to app/backend/worker code.

Three baked-in changes:
1. `services/landing-web/lib/integrations.ts` — flip the `zendesk` entry `status: 'coming_soon' → 'available'`,
   fill `useCases` + `setupSteps`, and rewrite `heroMessage`/`howItWorks`/`features` to shipped tone
   (connect via subdomain + agent email + API token; optional Zendesk trigger/webhook for real-time).
2. `services/landing-web/app/integrations/zendesk/page.tsx` — replace the coming-soon variant with the
   full "available" layout mirroring `app/integrations/intercom/page.tsx` (hero, how-it-works, features,
   setup steps, FAQ, CTA, animations). Remove the `Clock`/`X` coming-soon imports; reuse the existing
   landing `ZendeskIcon`.
3. `docs/SELF_HOSTING.md` — add a "Connecting Zendesk" section: create an API token, connect steps
   (subdomain / email / token), and the optional webhook setup (URL + signing secret, Zendesk
   trigger/webhook config), mirroring the existing "Connecting Jira" token-setup docs.

## Guiding principle — mirror, don't invent

Every change copies an already-shipped pattern:
- Data shape → the `jira` entry (lines ~233–268) as the reference "available" object; `JIRA_FAQS` as the
  reference for an integration-specific FAQ array.
- Page layout → `app/integrations/intercom/page.tsx` (the canonical full "available" page).
- Docs → the `## Connecting Jira` section of `docs/SELF_HOSTING.md` (token mint → connect → verify).

Keep copy **honest to shipped scope** (PRD "Out of scope"): new tickets only (no historical backfill),
one feedback item per ticket (subject + description, no per-comment), token/Basic-auth (no OAuth),
pull by default with an optional webhook for real-time. Do **not** advertise custom-field mapping,
per-comment ingestion, tag/view filters, or OAuth.

  and maps it to the requester in Customer 360. No manual CSV exports."*
- `howItWorks` (3 steps, keep the array length): rewrite to the real flow —
  1. **Connect Zendesk** — "Paste your Zendesk subdomain, agent email, and an API token to authorize
     Rereflect — no OAuth redirect required."
  2. **Tickets Flow In** — "Rereflect polls for new tickets automatically from the moment you connect.
     Want real-time? Wire an optional Zendesk trigger/webhook."
  3. **AI Extracts Insights** — "Every ticket is analyzed for sentiment, categorized, flagged for churn
     risk, and mapped to the requester in Customer 360."
- `features` (target 6, matching intercom's density; **each `icon` must be a lucide name that Phase 2
  imports** — see the icon contract below). Suggested set:
  1. Ticket Analysis — subject + description analyzed for sentiment, pain points, feature requests — `FileText`
  2. Automatic Sync — new tickets pulled in on a schedule, no CSV exports — `RefreshCw`
  3. Real-Time Webhooks — optional Zendesk trigger/webhook, HMAC-signed and verified over the raw body — `Zap`
  4. Requester Mapping — each ticket mapped to its requester by email, feeding Customer 360 / health / churn — `Users`
  5. Churn Risk Detection — AI flags frustrated customers from ticket tone before they cancel — `AlertTriangle`
  6. Exactly-Once Ingestion — de-duplicated by ticket ID across polling and webhooks — `Shield`
- `useCases` (currently `[]`; fill 2–3 mirroring jira/intercom `{persona, role, quote, icon}`). Note: the

  safe here — no Phase 2 import needed. Suggested icons: `Headphones`, `Layers`, `Heart`. Personas:
  Head of Support (ticket volume → top 3 issues), Product Manager (feature requests surfaced with sentiment),
  CSM (churn caught from ticket tone).
- `faqs`: `SHARED_FAQS` → **`ZENDESK_FAQS`**.
- `setupSteps` (currently `[]`; fill 6, mirroring jira's token flow — the page renders
  `{setupSteps.length} simple steps`):
  1. Go to Settings → Integrations.
  2. Create a Zendesk API token — "In Zendesk Admin Center → Apps and integrations → APIs → Zendesk API,

  3. Paste your credentials into Rereflect — subdomain (the `{subdomain}` in `{subdomain}.zendesk.com`),
     the agent email that owns the token, and the API token.
  4. Rereflect validates & starts ingesting — verifies the token, encrypts it at rest, and creates a
     Zendesk feedback source so new tickets flow in. Token never shown again.
  5. Optional: turn on real-time — copy the webhook URL + signing secret from Rereflect, create a Zendesk
     webhook + "Ticket is created" trigger pointing at it. Otherwise Rereflect polls automatically.
  6. Watch tickets become feedback — new tickets appear within minutes, analyzed and mapped to the requester.

## Phase 3 — Docs (`docs/SELF_HOSTING.md`)

**Reference:** the `## Connecting Jira` section (l.339–389) and its structure (intro → mint token →
connect → verify → all features unlocked).

- **TOC:** add `- [Connecting Zendesk](#connecting-zendesk)` after the `Connecting Jira` line (l.17).
- **New section `## Connecting Zendesk`** inserted after the Jira section, before `## Production notes`:
  - **Intro** — unlike Jira (outbound issue creation), Zendesk is an **inbound feedback source**: new
    tickets become feedback items, analyzed like any other, and enriched into Customer 360 by the
    requester's email. Note the shipped scope: new tickets only (no historical backfill), one item per
    ticket (subject + description), de-duplicated by ticket ID; pull by default, webhook optional; Basic
    auth (agent-email + API token) against the Zendesk REST API v2 — no OAuth.
  - **### 1. Create a Zendesk API token** — Admin Center → Apps and integrations → APIs → Zendesk API →
    enable token access → add API token → copy (shown once).
  - **### 2. Connect from the app** — mirror Jira's "not an env var, pasted + encrypted per org, needs
    `LLM_ENCRYPTION_KEY`" note. Table of fields: **Subdomain** (`{subdomain}` in `{subdomain}.zendesk.com`),
    **Agent email** (owns the token), **API token**. Settings → Integrations → Zendesk (admin/owner only).
    State the SSRF safeguard (host must resolve to a public `*.zendesk.com` address; loopback/private
    rejected), that the token is never returned, and that connecting auto-provisions a Zendesk feedback
    source so tickets start flowing (PRD 9a option a).
  - **### 3. Optional: real-time via webhook** — copy the webhook URL (`POST /api/v1/webhooks/zendesk/events`)
    and signing secret shown on connect; in Zendesk Admin Center → Apps and integrations → **Webhooks**,
    create a webhook pointing at that URL with the secret, then a **Trigger** on "Ticket is created" that
    notifies the webhook. Signature scheme (per PRD R4): `X-Zendesk-Webhook-Signature` =
    base64(HMAC-SHA256(signing_secret, `timestamp + raw_body`)) with `X-Zendesk-Webhook-Signature-Timestamp`,
    verified over the raw body. Without the webhook, Rereflect polls for new tickets automatically.
  - **### Verify** — Settings → Integrations → Zendesk shows connection status, connected account, and a
    Test Connection action; new tickets appear as feedback mapped to the requester within minutes.
  - **### All features unlocked** — same OSS/no-gate boilerplate as the Jira section.

## Risks & notes

- **R1 — Empty "Coming Soon" section.** Zendesk is the last `coming_soon` entry; flipping it leaves the
  index's Coming Soon block heading-only. Mitigate with the Phase 4 guard, or accept as cosmetic.
- **R2 — Docs vs shipped UX (sequencing).** This aspect is "best done last"; the SELF_HOSTING route/label/
  secret details are proposed contracts — reconcile with the shipped frontend/backend before finalizing.
- **R3 — Icon import mismatch.** `features[].icon` (Phase 1) and `featureIconMap` (Phase 2) must be kept in
  lockstep; a mismatch is a silent blank icon and/or an unused-import lint failure.
- **R4 — Honesty guardrails.** Do not let copy over-promise beyond PRD scope (no custom-field mapping,
  per-comment, tag/view filters, OAuth, or historical backfill). `useCases` icons are page-unused (safe).
- **R5 — No app/backend edits.** This aspect is copy/docs only; any real connect/webhook behavior lives in
  the frontend/backend/worker aspects and is out of scope here.
