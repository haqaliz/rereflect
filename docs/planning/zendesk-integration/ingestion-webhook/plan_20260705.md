## Interface contract with ingestion-core (must match ‚Äî confirm before Phase 4)

This route's only job is to authenticate the request and hand off to the shared core via
`queue_source_event`. The exact call this aspect makes is:

```python
queue_source_event(
    source_type="zendesk",
    external_event_id=str(ticket_id),          # dedup key = ticket id, per ingestion-core spec
    event_type="ticket.created",               # provisional ‚Äî see note below
    event_data=payload.get("ticket", {}),       # raw ticket sub-object from the webhook body
    provider_context={"subdomain": subdomain},  # consumed by _find_matching_sources' zendesk branch
)
```

- `event_type="ticket.created"` is this plan's proposed literal for the trigger the operator wires
  up in Zendesk (new-ticket trigger). If **ingestion-core**'s `ZendeskAdapter.check_triggers`
  landed with a different literal (e.g. `"ticket_created"` or a generic `"ticket"`), update this
  route's literal to match at Phase 4 ‚Äî do not change the wiring shape (still one string constant
  passed straight through).
- `event_data` is the `ticket` sub-object exactly as received in the webhook JSON body (see
  **Webhook payload contract** below) ‚Äî this must be shaped so `ZendeskAdapter.extract_content`
  can read `subject`/`description`/`id`/`status`/`requester_email`/`tags` from it directly,
  mirroring Intercom's route passing `payload.get("data", {})` straight through (L314).
- `provider_context={"subdomain": subdomain}` is exactly what `_find_matching_sources`' zendesk
  branch expects per the ingestion-core spec ("matching `provider_context["subdomain"]`").

### Webhook payload contract (this aspect's assumption ‚Äî document for the operator in landing-docs)

The operator configures their Zendesk trigger's webhook body (Zendesk lets you author an arbitrary
JSON body with placeholders). This plan pins the expected shape to:

```json
{
  "subdomain": "acmeco",
  "ticket": {
    "id": 35436,
    "subject": "Can't log in",
    "description": "I get an error when I try to log in.",
    "status": "new",
    "requester_email": "jane@customer.example.com",
    "tags": ["login"]
  }
}
```

- `subdomain` is a top-level convenience field (simplest for the operator to hardcode literally in
  the Zendesk trigger's JSON body, since it's static per account ‚Äî no placeholder needed).
- Fallback resolution (in case the operator's trigger body omits the top-level `subdomain`, or a
  future connect-page snippet only offers `{{ticket.url}}`): parse the host label from
  `payload["ticket"]["url"]` (Zendesk ticket URLs are `https://{subdomain}.zendesk.com/...`), then
  fall back to an `X-Zendesk-Subdomain` request header if present. See `_resolve_zendesk_subdomain`
  in Phase 1.

## Phase 1 ‚Äî Pure helpers: subdomain resolution + signature verification (RED)

- **Goal:** two small, framework-free functions, each fully unit-testable without `TestClient` or a
  DB session.
- **Files:** `tests/test_zendesk_webhook.py` (new); implementation added to `source_webhooks.py` in
  this phase (helpers only ‚Äî no route yet).
- **RED ‚Äî write tests first:**
  1. `_resolve_zendesk_subdomain(payload: dict, headers: Mapping) -> str | None`:
     - Returns `payload["subdomain"]` when present (lower-cased).
     - Falls back to parsing the host label from `payload["ticket"]["url"]`
       (`https://acmeco.zendesk.com/...` ‚Üí `"acmeco"`) when `subdomain` is absent.
     - Falls back to the `X-Zendesk-Subdomain` header when neither payload field is present.
     - Returns `None` when none of the three are resolvable (malformed/empty payload).
     - Case-insensitive: `"ACMECO"` and `"acmeco"` resolve to the same normalized value.
  2. `_verify_zendesk_signature(body: bytes, timestamp: str, signature: str, secret: str) -> bool`:
     - Valid case: `signature == base64.b64encode(hmac.new(secret.encode(), timestamp.encode() +
       body, hashlib.sha256).digest()).decode()` ‚Üí `True`.
     - Tampered body (same signature, one byte changed in `body`) ‚Üí `False`.
     - Tampered/missing timestamp (signature computed with a different timestamp) ‚Üí `False`.
     - Empty/`None` `secret` ‚Üí `False` (**fail-closed** ‚Äî see Security Note below; this
       deliberately does **not** mirror `verify_intercom_signature`'s "no secret configured ‚Üí
       return True" bypass, because Zendesk's secret is a required per-org value, not an optional
       global env var).
     - Uses `hmac.compare_digest` (not `==`) for the final comparison ‚Äî assert via a test that
       patches `hmac.compare_digest` and confirms it is called (timing-attack resistance,
       mirroring the existing helpers' pattern).
- **Validation:** `cd services/backend-api && pytest tests/test_zendesk_webhook.py -v` (Phase 1
  tests green; route tests not yet written).
- **Commit:** `test(zendesk-webhook): subdomain resolution + HMAC signature helpers (RED)` then
  `feat(zendesk-webhook): implement _resolve_zendesk_subdomain + _verify_zendesk_signature (GREEN)`.

**Security note (fail-closed on missing secret):** `zendesk_integrations.webhook_secret` is
nullable (an operator can connect via BYOK token without opting into the real-time webhook). If
`webhook_secret` is `None`/empty for an otherwise-matched integration, treat the request exactly
like an invalid signature (401) ‚Äî **not** like an unknown subdomain (200) and **not** like
Intercom's dev-friendly "no secret ‚Üí allow" behavior. Silently accepting unsigned payloads for an
org that never opted into the webhook would let anyone who guesses/enumerates a subdomain inject
fake tickets. This is called out explicitly because it is the one place this aspect intentionally
diverges from the Intercom template it otherwise mirrors closely.

## Phase 2 ‚Äî Route skeleton: raw-body read, JSON parse, subdomain/integration resolution (RED ‚Üí GREEN)

- **Goal:** wire the route up through "do we even know which org this is for", with **no**
  signature check yet (that's Phase 3) so each concern gets its own RED/GREEN cycle.
- **Prerequisites:** Phase 1.
- **Files:** `source_webhooks.py` (route added, incomplete); `tests/test_zendesk_webhook.py`.
- **Steps:**
  1. `@router.post("/zendesk/events")` `async def handle_zendesk_webhook(request: Request, db:
     Session = Depends(get_db))` ‚Äî takes `db` (unlike `handle_intercom_webhook`, which needs no DB
     access; this route does, mirroring `handle_generic_webhook`'s signature at L165-170).
  2. **Read raw body FIRST, before any parsing:** `body = await request.body()` ‚Äî this exact bytes
     object is both (a) what gets JSON-parsed and (b) what gets HMAC-verified in Phase 3. This
     ordering is load-bearing: the signature is defined over the *raw* bytes Zendesk sent, and
     `json.dumps(json.loads(body))` is **not** guaranteed to round-trip byte-for-byte (key order,
     whitespace, unicode escaping), so the verification must never re-serialize the parsed dict.
  3. Parse: `payload = json.loads(body)`; invalid JSON ‚Üí `HTTPException(400, "Invalid JSON")`
     (mirrors Intercom/Slack's `json.JSONDecodeError` handling).
  4. `subdomain = _resolve_zendesk_subdomain(payload, request.headers)`; if `None` ‚Üí
     `logger.info(...)`; return `{"status": "ignored", "reason": "missing_subdomain"}`, HTTP 200,
     **no** `queue_source_event` call.
  5. `integration = db.query(ZendeskIntegration).filter(ZendeskIntegration.subdomain ==
     subdomain, ZendeskIntegration.is_active.is_(True)).first()`. If `None` ‚Üí log at `info` level
     (include the subdomain for operator debugging, but do not leak in the response body); return
     `{"status": "ignored", "reason": "unknown_subdomain"}`, HTTP 200, **no** queueing.
- **RED ‚Äî tests:**
  1. `test_webhook_ignores_missing_subdomain` ‚Äî payload with no `subdomain`/`ticket.url` and no
     header ‚Üí 200, `{"status": "ignored", "reason": "missing_subdomain"}`, `queue_source_event`
     not called.
  2. `test_webhook_ignores_unknown_subdomain` ‚Äî well-formed payload with `subdomain: "nosuchorg"`
     but no matching `ZendeskIntegration` row in the test DB ‚Üí 200, `reason: "unknown_subdomain"`,
     not called.
  3. `test_webhook_rejects_invalid_json` ‚Äî non-JSON body ‚Üí 400.
- **Validation:** `pytest tests/test_zendesk_webhook.py -v`.
- **Commit:** `feat(zendesk-webhook): route skeleton ‚Äî raw body, subdomain resolution, integration lookup (GREEN)`

## Phase 3 ‚Äî Signature enforcement wired into the route (RED ‚Üí GREEN)

- **Goal:** once an `integration` row is found (Phase 2 step 5), require a valid signature before
  doing anything else.
- **Prerequisites:** Phase 2.
- **Files:** `source_webhooks.py`; `tests/test_zendesk_webhook.py`.
- **Steps (inserted between Phase 2's integration lookup and the eventual queueing in Phase 4):**
  1. `timestamp = request.headers.get("X-Zendesk-Webhook-Signature-Timestamp", "")`;
     `signature = request.headers.get("X-Zendesk-Webhook-Signature", "")`.
  2. If `not integration.webhook_secret` ‚Üí `raise HTTPException(401, "Webhook not configured for
     this account")` (fail-closed per the Phase 1 security note ‚Äî do this check before attempting
     to decrypt, since there's nothing to decrypt).
  3. Decrypt: `try: secret = decrypt_api_key(integration.webhook_secret) except Exception:
     logger.error(...); raise HTTPException(401, "Invalid webhook configuration")` ‚Äî **never** let
     a corrupt/rotated-key `InvalidToken` bubble into an unhandled 500 (mirrors the "missing
     LLM_ENCRYPTION_KEY ‚Üí 422 never 500" discipline from the connection aspect, adapted to 401
     here since this is inbound auth, not a user-facing connect call).
  4. `if not signature or not timestamp or not _verify_zendesk_signature(body, timestamp,
     signature, secret): raise HTTPException(401, "Invalid signature")`.
- **RED ‚Äî tests:**
  1. `test_webhook_accepts_valid_signature` (combined with Phase 4's success-path test ‚Äî see
     below).
  2. `test_webhook_rejects_missing_signature_header` ‚Äî valid subdomain/integration, no
     `X-Zendesk-Webhook-Signature` header at all ‚Üí 401, `queue_source_event` not called.
  3. `test_webhook_rejects_missing_timestamp_header` ‚Üí 401.
  4. `test_webhook_rejects_invalid_signature` ‚Äî wrong secret used to sign ‚Üí 401.
  5. `test_webhook_rejects_tampered_body` ‚Äî valid signature computed over body A, request sent
     with body B ‚Üí 401 (proves verification is over the actual raw bytes received, not a
     re-derived value).
  6. `test_webhook_rejects_when_webhook_secret_not_set` ‚Äî integration exists (BYOK-only connect,
     `webhook_secret=None`) ‚Üí 401, not 200 (this is the fail-closed case from Phase 1 ‚Äî explicitly
     assert it is 401, **not** the 200 "unknown_subdomain"/"no_active_source" no-op shape, so a
     future refactor can't accidentally merge these two very different cases).
  7. `test_webhook_never_500s_on_corrupt_encrypted_secret` ‚Äî patch `decrypt_api_key` to raise
     `cryptography.fernet.InvalidToken` ‚Üí 401, not 500.
- **Validation:** `pytest tests/test_zendesk_webhook.py -v`.
- **Commit:** `feat(zendesk-webhook): enforce HMAC signature against per-org webhook_secret (GREEN)`

## Phase 4 ‚Äî Active-source lookup, quick dedup pre-check, queue (RED ‚Üí GREEN)

- **Goal:** the success path ‚Äî everything after "signature is valid."
- **Prerequisites:** Phases 1-3.
- **Files:** `source_webhooks.py`; `tests/test_zendesk_webhook.py`.
- **Steps:**
  1. `source = db.query(FeedbackSource).filter(FeedbackSource.organization_id ==
     integration.organization_id, FeedbackSource.source_type == "zendesk",
     FeedbackSource.is_active.is_(True)).first()`. If `None` ‚Üí log at `warning` level (per PRD
     9a/9b this should not normally happen, since **backend-connection** auto-provisions the
     source on connect ‚Äî its absence here means the source was later deactivated or the
     auto-provision step regressed; log loudly so it's observable); return `{"status": "ignored",
     "reason": "no_active_source"}`, HTTP 200, **no** queueing.
  2. `ticket = payload.get("ticket") or {}`; `ticket_id = ticket.get("id")`. If falsy ‚Üí log
     `warning`; return `{"status": "ignored", "reason": "missing_ticket_id"}`, HTTP 200, no
     queueing (defensive ‚Äî not in the acceptance list verbatim, but required by PRD 9b's "never a
     silent drop" ‚Äî this makes the drop *observable*, not silent).
  3. **Quick synchronous dedup pre-check** (mirrors `handle_slack_webhook`'s pattern, L139-146):
     `existing = db.query(FeedbackSourceEvent).filter(FeedbackSourceEvent.source_id == source.id,
     FeedbackSourceEvent.external_event_id == str(ticket_id)).first()`. If found ‚Üí return
     `{"status": "duplicate"}`, HTTP 200, **no** re-queueing. This is a fast-path optimization on
     top of (not a replacement for) the ingestion-core's own `FeedbackSourceEvent` dedup ‚Äî the
     authoritative dedup still lives downstream in `process_source_event` / the unique constraint
     `uq_source_event (source_id, external_event_id)`, so out-of-order/duplicate webhook
     redeliveries remain safe even if this pre-check is ever bypassed or races.
  4. Else: build `provider_context = {"subdomain": subdomain}`; call
     `queue_source_event(source_type="zendesk", external_event_id=str(ticket_id),
     event_type="ticket.created", event_data=ticket, provider_context=provider_context)`; return
     `{"status": "queued", "task_id": task_id}`, HTTP 200.
  5. Wrap the `queue_source_event` call in the same `try/except Exception ‚Üí logger.error +
     HTTPException(500, ...)` shape Intercom uses (L309-323) ‚Äî this is the one legitimate 500 path
     (Celery broker unreachable), matching the existing sibling webhook handlers' behavior exactly
     rather than inventing a new contract for broker failures.
- **RED ‚Äî tests (mirroring Intercom's `TestIntercomWebhook` class, L240-403):**
  1. `test_webhook_valid_signature_new_ticket_queues_event` ‚Äî full happy path: seed an active
     `ZendeskIntegration` (subdomain `acmeco`, decrypted secret known) and an active `zendesk`
     `FeedbackSource` for the same org; POST a valid, correctly-signed new-ticket payload; assert
     `200`, `{"status": "queued", ...}`, and `mock_queue.assert_called_once_with(source_type=
     "zendesk", external_event_id="35436", event_type="ticket.created", event_data=ticket_dict,
     provider_context={"subdomain": "acmeco"})`.
  2. `test_webhook_no_active_source_is_noop` ‚Äî integration exists + signature valid, but **no**
     active `zendesk` `FeedbackSource` for that org ‚Üí 200, `reason: "no_active_source"`,
     `queue_source_event` not called, nothing created (assert no new `FeedbackItem`/
     `FeedbackSourceEvent` row exists after the call).
  3. `test_webhook_dedupes_ticket_already_ingested_via_pull` ‚Äî seed a `FeedbackSourceEvent` row
     with `source_id=<the zendesk source>.id`, `external_event_id="35436"` (simulating a prior pull
     ingestion of the same ticket); POST the same ticket via the webhook with a valid signature ‚Üí
     200, `{"status": "duplicate"}`, `queue_source_event` **not** called, and the pre-seeded
     `FeedbackSourceEvent` row count is unchanged (still exactly one row for that ticket ‚Äî no
     duplicate `FeedbackItem`/event created).
  4. `test_webhook_missing_ticket_id_is_noop` ‚Äî valid signature, valid source, payload's `ticket`
     has no `id` ‚Üí 200, `reason: "missing_ticket_id"`, not queued.
  5. `test_webhook_broker_failure_returns_500` ‚Äî patch `queue_source_event` to raise (simulating a
     Celery/Redis outage) ‚Üí 500, matching the Intercom sibling's existing contract.
- **Validation:** `pytest tests/test_zendesk_webhook.py -v` ‚Äî full file green, including Phases 1-3.
- **Commit:** `feat(zendesk-webhook): active-source check + quick dedup + queue_source_event (GREEN)`

## Edge cases & errors

| Condition | Response | `queue_source_event` called? | Notes |
|---|---|---|---|
| Malformed JSON body | 400 | No | Mirrors Intercom/Slack |
| No `subdomain` resolvable (payload + headers) | 200 `ignored/missing_subdomain` | No | Not in spec's acceptance list verbatim but required by 9b |
| Subdomain resolved, no matching active `ZendeskIntegration` | 200 `ignored/unknown_subdomain` | No | Spec AC3 (first half) |
