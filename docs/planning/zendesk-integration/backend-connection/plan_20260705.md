**No plan/feature gating.** Per CLAUDE.md's OSS pivot and the Jira/Linear
precedent in `feedback_sources.py` (no `has_feature` branch for `jira`),
`zendesk` gets **no** `has_feature` check — it's a BYOK, own-auth source
type, unlocked on every plan including Free. `TestFreePlanUnlocked`-equivalent
coverage is mandatory (mirrors Jira's).

**No new link/model table.** Unlike Jira (which has `FeedbackJiraIssue` for
the outbound create-issue aspect), this aspect is inbound-only — there is no
create-issue feature, so there is no second table. Just `ZendeskIntegration`.

**Env/secrets:** no new environment variable. Reuses `LLM_ENCRYPTION_KEY`
(Fernet key, `services/backend-api/src/utils/encryption.py`) for **both**
`api_token` and `webhook_secret` encryption — `encrypt_api_key`/
`decrypt_api_key`/`get_key_hint` are generic over any secret string, so no
new encryption helper is needed.

**Locked design decision (resolves PRD OQ1 — webhook secret UX):**
- On the **first** successful connect for an org, generate a new
  `webhook_secret` via `secrets.token_urlsafe(32)`, encrypt it, store it.
- On **reconnect** (row already exists), **preserve** the existing
  `webhook_secret` — do not regenerate it. Rotating it silently would break
  an already-configured Zendesk trigger/webhook, whereas rotating
  `api_token` on reconnect (mirroring Jira) is expected/safe since the
  operator is explicitly re-pasting a token.
  - Rationale flagged to user: this is a judgment call resolving an
    explicitly-open PRD question (OQ1); if you'd rather always rotate the
    webhook secret on reconnect (forcing the operator to re-paste it into
    Zendesk each time), say so before Phase 3 and this plan's connect logic
    flips to "always regenerate."
- `POST /connect` response **always** includes the current plaintext
  `webhook_secret` (freshly generated or the preserved one, decrypted) —
  this is the "display once" contract: it is never included in
  `GET /status` (mirrors `api_token` never being returned there).
- A "regenerate webhook secret" action is explicitly deferred (v2 / out of
  scope for this aspect) — the column design supports adding it later
  without a migration.

**Source auto-provision (PRD 9a, locked to option (a)):** on a successful
`connect` (initial or reconnect), if the org has no existing `FeedbackSource`
row with `source_type == "zendesk"`, auto-create one:
```python
FeedbackSource(
    organization_id=current_org.id,
    integration_id=None,          # zendesk is own-auth, like jira/linear
    source_type="zendesk",
    name="Zendesk",
    provider_config={"subdomain": subdomain},
    triggers={},
    field_mapping={},
    auto_import=True,
)
```
`provider_config.subdomain` is the join key the (separate, out-of-scope)
`source_events._find_matching_sources` `zendesk` branch will use later — this
aspect only needs to make sure it's populated correctly and idempotently
(never a second source auto-created on repeat reconnects).
`GET /status` gains a `has_feedback_source: bool` field so the operator/UI
can tell "connected, source exists" from "connected, no source yet" (should
auto-provision ever be disabled/fail in some edge case).

## 4. Testing strategy

- **Mocking boundary:** `ZendeskClient` is patched at
  `src.api.routes.zendesk_integration.ZendeskClient` in route tests (never
  real HTTP) — same pattern as Jira/Linear/HubSpot/Salesforce. `httpx.Client`
  is patched directly in `test_zendesk_client.py` (client-level unit tests,
  no route involved).
- **DNS mocking is request-scoped, not autouse.** Copy Jira's `_dns()`
  contextmanager + its docstring rationale verbatim (patching
  `socket.getaddrinfo` for a whole test/session hijacks the TestClient
  app-lifespan's real DNS calls on startup and causes multi-minute hangs).
- **`LLM_ENCRYPTION_KEY`:** every test that exercises `encrypt_api_key`/
  `decrypt_api_key` wraps the call in
  `patch.dict(os.environ, {"LLM_ENCRYPTION_KEY": TEST_FERNET_KEY})` using
  the same throwaway 32-byte base64 test key Jira's suite uses
  (`TEST_FERNET_KEY = "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="`) —
  copy the constant, don't invent a new one (keeps a single documented
  "this is a test-only key" convention across integration test files).
- **RBAC coverage:** all four routes get an explicit member-gets-403 test —
  non-negotiable per the Jira/HubSpot/Salesforce precedent and this repo's
  RBAC section in CLAUDE.md (`require_admin_or_owner`).
- **Free-plan coverage:** one end-to-end `TestFreePlanUnlocked` test proves
  zero gating, matching the OSS "all features unlocked" positioning in the
  PRD header.
- **Model tests never touch the route/client layers**; client tests never
  touch the DB; route tests never make real HTTP calls or real DNS lookups.
  Keep that separation — it's what makes the suite fast and mockable, and
  it is why the parallel Jira/Linear/HubSpot suites stay fast today.
- **Full-suite gate:** every phase's validation step ends with the
  project-wide `pytest tests/ -v`, not just the new file — `main.py` and
  `feedback_sources.py` are shared, high-traffic files; a passing new file
  in isolation is not sufficient evidence of a green branch.

## 5. Env / secrets

- `LLM_ENCRYPTION_KEY` — **no new variable**; this aspect is additive-only
  on top of the existing Fernet key already documented in
  `services/backend-api/src/utils/encryption.py` and `.env.example`. Both
  `api_token` and `webhook_secret` are encrypted through the same
  `encrypt_api_key`/`decrypt_api_key` helpers.
- No Zendesk-side app registration/OAuth client id/secret is needed — this
  is a BYOK (operator-pasted API token) flow, not OAuth, so there is
  nothing to add to `.env.example` / Railway / `docs/SELF_HOSTING.md` for
  this aspect specifically (the SELF_HOSTING.md update for "how to
  generate a Zendesk API token + set up a webhook trigger" is explicitly
  listed under PRD requirement 11, a **separate** docs aspect — not this
  plan's file list).
- **R6 parity (never-500 on missing key):** exactly like Jira, `POST
  /connect` must catch `ValueError` from `encrypt_api_key` and return 422,
  never crash the request — this applies to encrypting `api_token` **and**
  (new, since this aspect introduces a second encrypted secret) whichever
  encrypt call for `webhook_secret` runs. Both must be wrapped; a bare
  `encrypt_api_key(payload.api_token)` succeeding while an unwrapped
  `encrypt_api_key(webhook_secret)` call three lines later raises an
  uncaught `ValueError` would be a 500-on-missing-key regression hiding
  behind the first call's `try/except` — the GREEN implementation must
  wrap **both** calls (or one shared `try` block covering both `encrypt_api_key`
  invocations), and Phase 3's RED test suite must actually exercise the
  order where the *first* call succeeds and a later one fails, not just
  "the first encrypt call raises" (which Jira's test happens to cover
  trivially since Jira only has one secret).

## 6. Edge cases

- **SSRF (two-layer):** route `_assert_host_not_ssrf` (DNS-resolution-based,
  the primary gate on untrusted operator input) + `ZendeskClient._assert_safe_subdomain`
  (string-level, defense-in-depth, unit-testable without network I/O) — both
  layers must be independently covered by tests (route-level SSRF tests in
  `test_zendesk_connection.py`; client-level in `test_zendesk_client.py`'s
  `TestZendeskClientSSRFGuard`), exactly mirroring Jira's two-layer split.
- **Suffix-trick hosts** (`acme.zendesk.com.evil.com`) must be rejected by
  `_normalize_subdomain` before DNS is even attempted — covered explicitly
  (this is the one edge case the Jira suite covers only at the client layer
  via `_assert_safe_site_url`'s test; this plan adds an equivalent
  route-layer test too, since `_normalize_subdomain`'s stripping logic is
  exactly where a suffix-trick host could slip through if implemented
  naively with a plain `.endswith()` check without first splitting off a
  single trailing `.zendesk.com` label).
- **Missing `LLM_ENCRYPTION_KEY` → 422, never 500** — covered for both
  secrets (see Section 5's R6 note); must not leak a stack trace or a bare
  `ValueError` past the route boundary.
- **Reconnect rotates `api_token`, preserves `webhook_secret`** — the
  locked design decision in Section 1; both halves need explicit assertions
  (token_hint changes; webhook_secret does not) in the same test, since
  testing only one half wouldn't catch a naive "regenerate everything on
  every connect" implementation.
- **Org-scoping:** every query (`_get_active_integration`,
  `_get_org_integration`, the auto-provision existence check) filters on
  `organization_id == current_org.id` sourced from the JWT via
  `get_current_org` — never trust a client-supplied org id (there isn't
  one in these payloads, which is itself the correct design — flag if any
  future edit to this route ever accepts an org id in the request body).
- **Auto-provision idempotency:** connecting twice (Phase 3's dedicated
  test), and connecting when a `zendesk` source already exists from a
  manual wizard creation, must both result in exactly one `zendesk`
  `FeedbackSource` row — never a second, and never overwriting/renaming an
  operator-customized existing one (only its *existence* is checked, not
  its `name`/`triggers`/`field_mapping`, which the operator may have
  already edited).
- **Never a silent drop / crash on `test`/`status` for a not-yet-connected
  org:** `status` returns `connected: false` (200), `test` returns 400 (not
  404/500) when there's no active integration — matches Jira exactly and
  is required for the frontend connect page (a separate aspect) to render
  a sane "not connected" state without special-casing error types.
- **Disconnect does not touch the auto-provisioned `FeedbackSource`** —
  covered explicitly; connection lifecycle and source lifecycle are
  intentionally decoupled per PRD 9a ("connecting stores credentials only").
