## Contracts locked in this aspect

These are not fully pinned by the PRD/spec text and must be decided here because both
`ingestion-pull` and `ingestion-webhook` depend on them:

1. **Normalized `event_data` shape** (both entry points must produce this before calling
   `process_source_event`):
   ```python
   {
       "ticket": {
           "id": 4521,
           "subject": "Billing issue",
           "description": "<p>My invoice is wrong</p>",   # may be missing/empty/None
           "status": "open",
           "tags": ["billing", "urgent"],
           "requester_email": "jane@example.com",           # may be missing/None
       },
       "subdomain": "acmeco",
   }
   ```
   `event_type` passed to `process_source_event`/`check_triggers` is the literal string
   `"ticket.created"` for new-ticket events (the only granularity in scope; PRD explicitly
   restricts pull to "new tickets only").
   `provider_context` passed to `process_source_event` is `{"subdomain": "acmeco"}`.

2. **`access_token` packing for Basic auth.** Zendesk auth is `("{email}/token", api_token)`, not a
   bearer token, so the single `access_token: Optional[str]` parameter on `fetch_context` carries
   `"{email}:{api_token}"` (colon-delimited, split once). This is a Zendesk-adapter-local decision —
   it does not change the `BaseSourceAdapter` signature. `ingestion-pull`/`ingestion-webhook` (when
   they call `fetch_context` directly, or whichever aspect wires `access_token` into
   `_process_event_for_source`) must build this string from the stored `ZendeskIntegration`
   `email`/decrypted `api_token`. Malformed/missing separator → treat as no token (`{}`).

3. **`customer_email` plumbing is generic, not Zendesk-only.** `extract_content()` may now return an
   optional top-level `customer_email` key (sibling to `text`/`metadata`) in its return dict.
   `_process_event_for_source` reads `content.get("customer_email")` (defaults `None`) and sets it on
   the created `FeedbackItem`. This is purely additive — no existing adapter (`Intercom`, `Slack`,
   `Email`, `Webhook`) returns this key today, so their created `FeedbackItem`s are unaffected
   (`customer_email` stays `None` as it does today). Only the `auto_import=True` path can carry it;
   `PendingFeedback` has no `customer_email` column, so the manual-review path silently drops it
   (documented in Edge cases, not a regression — no adapter populates it there today either).

4. **Pre-existing bug found, must be fixed here: `_log_event`'s `external_message_id` is wrong for
   every non-Slack/webhook adapter.** `_process_event_for_source` computes the correct dedup key via
   `adapter_event_id, message_id = adapter.get_external_ids(event_data)` and dedupes the *lookup*
   against it, but the `FeedbackSourceEvent` row it *writes* (`_log_event`) independently
   re-derives `message_id` via `event_data.get("ts") or event_data.get("item", {}).get("ts") or
   event_data.get("content_hash")` — a Slack/webhook-shaped guess that is `None` for Intercom's
   `data.item.id` shape and would be `None` for Zendesk's `ticket.id` shape too. The result: the
   stored `external_message_id` never matches the dedup query's `message_id` on the next event, so
   dedup silently never fires for these adapters. No existing test exercises
   `process_source_event`/`_log_event` end-to-end (confirmed: no test file references `_log_event`,
   `external_message_id`, or `source_events` outside `source_events.py` itself), so this has been
   latent and untested. Fixing it is required to satisfy this aspect's acceptance criterion ("a
   second identical event is deduped via `FeedbackSourceEvent`") and is a **strict superset fix**:
   add an optional `message_id=None` parameter to `_log_event`, pass the already-computed
   `message_id` from all three call sites inside `_process_event_for_source` where it's known
   (`empty_text`, `processed`/`feedback_created`, `pending`/`pending_created`), and fall back to the
   legacy derivation only when `message_id` is not supplied (the one `no_trigger_match` call site,
   where `get_external_ids` hasn't been called yet — unchanged, "ignored" rows aren't in the dedup
   filter anyway). No behavior change for any status other than the two affected columns; no test
   currently pins the old (broken) value.

     ```python
     def check_triggers(self, event_type, event_data, triggers):
         if triggers.get("new_ticket") and event_type == self.TICKET_CREATED:
             return "new_ticket"
         keywords = triggers.get("keywords", [])
         if keywords:
             body = self._get_body_text(event_data)
             if body:
                 body_lower = body.lower()
                 for keyword in keywords:
                     if keyword.lower() in body_lower:
                         return f"keyword:{keyword}"
         return None

         ticket = event_data.get("ticket", {})
         subject = strip_html(ticket.get("subject") or "")
         description = strip_html(ticket.get("description") or "")
         return f"{subject} {description}".strip()
     ```

       returns `{}` (or whatever was gathered before the exception), never raises.
  2. GREEN:
     ```python
     def fetch_context(self, event_data, access_token, field_mapping):
         if not access_token or ":" not in access_token:
             return {}
         email, api_token = access_token.split(":", 1)
         ticket = event_data.get("ticket", {})
         subdomain = event_data.get("subdomain")
         ticket_id = ticket.get("id")
         context = {}
         try:
             with httpx.Client(timeout=10, auth=(f"{email}/token", api_token)) as client:
                 resp = client.get(f"https://{subdomain}.zendesk.com/api/v2/tickets/{ticket_id}")
                 ticket_data = resp.json().get("ticket", {})
                 context["ticket_url"] = f"https://{subdomain}.zendesk.com/agent/tickets/{ticket_id}"
                 requester_id = ticket_data.get("requester_id")
                 if requester_id:
                     resp2 = client.get(f"https://{subdomain}.zendesk.com/api/v2/users/{requester_id}")
                     user_data = resp2.json().get("user", {})
                     context["requester_name"] = user_data.get("name")
                     context["requester_email"] = user_data.get("email")
         except Exception as e:
             logger.error(f"Failed to fetch Zendesk context: {e}")
         return context
     ```

## Edge cases / error handling
- Missing `description` → subject-only text, never a dangling `"\n\n"` (verified in Phase 2).
- Missing `requester_email` → item still ingested with `customer_email=None` (PRD 9b — never
  dropped for lack of an email).
- `PendingFeedback` (manual-review / `auto_import=False`) path has no `customer_email` column —
  Zendesk feedback routed there loses the requester mapping until approved into a `FeedbackItem`.
  Out of scope to add a column here (schema change belongs to a review-flow aspect if ever needed);
  note it and move on — PRD's default recommendation is auto-provision + `auto_import=True`.
- Unmatched subdomain (no `ZendeskIntegration` row, or row exists but `is_active=False`) → empty
  source list → `{"status": "no_sources", ...}`, never a 500/crash (Phase 7, test 4).
- `fetch_context` malformed/missing `access_token`, or any `httpx` exception → returns `{}` or a
  partial dict, never raises (mirrors Intercom's `except Exception` catch-all).
- `_log_event`'s `message_id` fix must not alter `status` semantics or any other column — only
  `external_message_id` on the "processed"/"pending"/"empty_text" (ignored) rows; the
  `no_trigger_match` row keeps the legacy (best-effort) derivation since no adapter call has run yet.
- Duplicate detection is a ticket-id equality check (`message_id`), independent of
  `external_event_id` — this is required for PRD's "exactly-once across repeated syncs / redelivered
  webhooks" goal, since a webhook redelivery gets a *new* delivery id but the *same* ticket id, and a
  pull-cursor overlap likewise re-observes the same ticket id under a new synthetic event id.
