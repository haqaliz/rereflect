## 1. Context & Goals

This aspect wires a sixth health-score component (`crm`) that encodes commercial
renewal-proximity risk drawn from the `crm_enrichment` table written by the
`hubspot-sync` aspect. It mirrors the shipped `usage_component` (M4.2) exactly: opt-in
at weight 0% by default so no existing scores change until an operator explicitly raises
`health_weight_crm`.

**Headline acceptance criterion (stated in spec):** with `health_weight_crm = 0`, every
existing health test must still pass — the regression suite is the single most important
gate.

**What this aspect does NOT do:** pull or store CRM data (that is `hubspot-sync`), render
a profile card (that is `crm-profile-and-timeline`), or manage the HubSpot connection
(that is `hubspot-connection`). It only reads the `crm_enrichment` table and adds the
weighted component to `compute_health_score`.

**Worker-mirror decision (R4):** the worker does not read weights itself; only
`services/backend-api` computes health scores. However, to keep the ORM mirror
consistent (avoiding the current drift where `health_weight_usage` is already missing),
this aspect adds **both** `health_weight_usage` and `health_weight_crm` to the worker's
`OrgAIConfig`, and adds `usage_component` + `crm_component` to the worker's
`CustomerHealth` and `CustomerHealthHistory`. This is a catch-up plus addition — one PR,
no future drift.

```python
# This is a documented heuristic, NOT a trained model. Operators opt in by

CRM_RENEWAL_DAYS_WARN   = 30    # ≤ this many days to renewal → risk territory
CRM_RENEWAL_DAYS_HIGH   = 14    # ≤ this many days → higher risk
CRM_RENEWAL_DAYS_CRIT   = 7     # ≤ this many days → critical risk
```

Rationale: a near renewal on an at-risk account is a churn signal; an operator who
enables the weight can surface these customers. The scores are low (below neutral) to
represent risk, clamped to 0–100. No ARR influence in v1 — that requires a separate
heuristic model and is deferred.

### 4.4 `_compute_crm_component` (new function, health_score_service.py)

```python
def _compute_crm_component(
    db: Session, org_id: int, customer_email: str, now: datetime
) -> float:
    """
    Read renewal_date from crm_enrichment for (org, email) and return a
    0–100 float encoding renewal-proximity risk (lower = more risk).

    Renewal-proximity heuristic (documented constants at module top):
      - No row / no renewal_date → 50.0 (neutral, same as no data)
      - Renewal date already passed → 50.0 (not actionable in v1)
      - days_to_renewal ≤ CRM_RENEWAL_DAYS_CRIT (7)  → CRM_SCORE_CRIT  (15.0)
      - days_to_renewal ≤ CRM_RENEWAL_DAYS_HIGH (14) → CRM_SCORE_HIGH  (25.0)
      - days_to_renewal ≤ CRM_RENEWAL_DAYS_WARN (30) → CRM_SCORE_WARN  (35.0)
      - Otherwise                                     → CRM_SCORE_NEUTRAL (50.0)

    This is a HEURISTIC, not a trained model. See module-level constants for
    threshold/score values. Operators set health_weight_crm > 0 to opt in.

    Contract: this function NEVER raises. It uses a SAVEPOINT so a missing
    crm_enrichment table cannot abort the outer SQLAlchemy transaction.

    Args:
        now: Computation timestamp (passed by caller for consistency with
             other _compute_* functions; used for days_to_renewal arithmetic).
    """
    from sqlalchemy import text
    try:
        sp = db.begin_nested()  # SAVEPOINT — isolates this read from the outer transaction
        row = db.execute(
            text(
                "SELECT renewal_date FROM crm_enrichment "
                "WHERE organization_id = :org_id AND customer_email = :email "
                "LIMIT 1"
            ),
            {"org_id": org_id, "email": customer_email},
        ).fetchone()
        sp.commit()
        if row is not None and row[0] is not None:
            renewal_date = row[0]
            # Normalize to datetime if stored as date
            if not isinstance(renewal_date, datetime):
                renewal_date = datetime(
                    renewal_date.year, renewal_date.month, renewal_date.day
                )
            days_to_renewal = (renewal_date - now).days
            if days_to_renewal < 0:
                return CRM_SCORE_NEUTRAL        # past renewal — not actionable
            if days_to_renewal <= CRM_RENEWAL_DAYS_CRIT:
                return max(0.0, min(100.0, CRM_SCORE_CRIT))
            if days_to_renewal <= CRM_RENEWAL_DAYS_HIGH:
                return max(0.0, min(100.0, CRM_SCORE_HIGH))
            if days_to_renewal <= CRM_RENEWAL_DAYS_WARN:
                return max(0.0, min(100.0, CRM_SCORE_WARN))
    except Exception:
        try:
            sp.rollback()
        except Exception:
            pass
    return CRM_SCORE_NEUTRAL
```

```python
class TestCrmComponentSavepointIsolation:
    """
    _compute_crm_component must use db.begin_nested() (SAVEPOINT) so a
    missing crm_enrichment table cannot abort the outer SQLAlchemy transaction
    and cause PendingRollbackError on subsequent queries.
    Mirrors TestUsageComponentSavepointIsolation in test_health_usage_component.py.
    """

    def test_begin_nested_called_during_crm_read(self, db, test_organization):
        """begin_nested() must be invoked during the CRM component read."""
        with patch.object(db, "begin_nested", wraps=db.begin_nested) as spy:
            result = _compute_crm_component(
                db, test_organization.id,
                "savepoint_crm@example.com", datetime.utcnow()
            )
        assert result == pytest.approx(50.0)  # missing table → neutral fallback
        assert spy.call_count >= 1, (
            "begin_nested must be called to SAVEPOINT-isolate the CRM read. "
            "Without it, a PostgreSQL SQL error aborts the transaction and the "
            "next db.query() raises PendingRollbackError (HTTP 500)."
        )

    def test_session_still_usable_after_simulated_crm_error(self, db, test_organization):
        """After a simulated execute error the session must still be usable."""
        from sqlalchemy import text as sql_text
        original_execute = db.execute
        first_call = [True]

        def failing_execute(statement, *args, **kwargs):
            if first_call[0]:
                first_call[0] = False
                raise Exception("simulated: table crm_enrichment does not exist")
            return original_execute(statement, *args, **kwargs)

        with patch.object(db, "execute", side_effect=failing_execute):
            result = _compute_crm_component(
                db, test_organization.id,
                "session_crm@example.com", datetime.utcnow()
            )
        assert result == pytest.approx(50.0)
        val = db.execute(sql_text("SELECT 1")).scalar()
        assert val == 1, "Session must be usable after SAVEPOINT-isolated CRM error"
```

```python
class TestCrmComponentRenewalProximityScoring:
    """
    Renewal-proximity heuristic: closer renewal → lower (more risky) score.
    Uses crm_enrichment_table fixture for an actual DB row.
    All assertions are deterministic against the documented module constants.
    """

    def _insert_renewal(self, db, org_id, email, days_from_now):
        """Insert a crm_enrichment row with renewal_date = now + days_from_now."""
        from sqlalchemy import text
        from datetime import date, timedelta
        renewal = (date.today() + timedelta(days=days_from_now)).isoformat()
        db.execute(text(
            "INSERT INTO crm_enrichment "
            "(organization_id, customer_email, renewal_date) "
            "VALUES (:o, :e, :r)"
        ), {"o": org_id, "e": email, "r": renewal})
        db.commit()

    def test_no_renewal_returns_neutral(
        self, db, test_organization, crm_enrichment_table
    ):
        """Customer with no row returns 50.0 (neutral)."""
```

## 7. Open Questions & Risks

### R-1: Exact `down_revision` for the migration

**Status:** Unresolved at plan time; hubspot-sync migration does not yet exist.
**Mitigation:** placeholder in migration file; mandatory step before merging — run
`alembic history` and set `down_revision` to the hubspot-sync revision.

### R-2: Float vs Integer for crm_component

**Decision:** `Float` per the pinned contract. The renewal-proximity heuristic naturally
produces continuous values (e.g., 15.0, 25.0, 35.0, 50.0) and using Float avoids a
second migration if intermediate values are ever needed. The existing `usage_component`
is `Integer`; this is an intentional divergence per the contract.

### R-3: Worker model drift beyond this aspect

**Decision (see §4.8):** this PR also adds `health_weight_usage` (missing since
`z5a6b7c8d9e0`) and `usage_component`/`confidence_score` to the worker mirror. These
are catch-up additions. Future aspects must add worker mirror updates alongside any
backend column additions — or a CI check comparing backend and worker model field sets
should be introduced (deferred).

### R-4: 6-weight sum validation for callers that send 5 fields

**Status:** Resolved in design — `crm` defaults to `0` in `HealthWeightsUpdate`, so
existing callers that send only 5 fields continue to work as long as those 5 sum to
100. A caller that sends `{"churn":35,"sentiment":25,"resolution":25,"frequency":15,"usage":10}`
(total 110) was always invalid and remains 422. No breaking change.

### R-5: Score semantics of past renewal dates

**Decision:** `days_to_renewal < 0` → return `CRM_SCORE_NEUTRAL` (50.0). A past renewal
date means the contract may have already renewed or expired — neither signal is actionable
from the enrichment snapshot alone. This is documented in the function docstring and is
revisable in a later aspect without a schema change.

### R-6: Existing test isolation

`test_health_usage_component.py` and `test_health_weights.py` do not import anything
from this aspect. Adding `"crm": 0.0` to `WEIGHTS` and `health_weight_crm=0` to
`OrgAIConfig` is purely additive. The snapshot value `47` in
`TestScoreStabilityCharacterization` must remain identical because `crm_weight = 0`
makes the CRM term contribute `0 * anything = 0` to the weighted sum.

### R-7: _get_org_weights 6-key sum invariant

The existing `TestGetOrgWeightsDefaultsBeforeUsageColumn.test_sum_equals_one` test
(file: `test_health_usage_component.py`, line 152) asserts `sum(weights.values()) ==
1.0`. After adding `"crm": 0.0` to `WEIGHTS`, the sum becomes
`0.35 + 0.25 + 0.25 + 0.15 + 0.0 + 0.0 = 1.0` — still passes. No change to that test.

The `test_returns_five_keys_with_usage_default` assertion (`len == 5`, line 145) will
fail after adding the `crm` key. **Action required:** update that test to `== 6` keys
and `{"crm_risk", ..., "crm"}` in the same commit as Phase 4. Document this as an
intended snapshot update, not a regression.
