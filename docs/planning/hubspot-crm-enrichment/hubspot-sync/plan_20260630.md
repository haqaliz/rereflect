## 1. Context and Scope

This aspect delivers the data-movement layer of HubSpot CRM enrichment. After the `hubspot-connection` aspect establishes the `hubspot_integrations` table and token encryption, this aspect:

- Creates the `crm_enrichment` table (second chained Alembic migration, `down_revision` = hubspot-connection revision; see §3).
- Implements the HubSpot HTTP client in the worker (`services/worker-service/src/clients/hubspot.py`), pulling Contacts, Companies, and Deals.
- Implements the sync task file (`services/worker-service/src/tasks/hubspot_sync.py`) with a Celery-free core function `_sync_org`, a fan-out task `sync_all_hubspot`, and a per-org retryable task `sync_hubspot_org`.
- Adds both tasks to `services/worker-service/src/celery_app.py`'s `include` list and `beat_schedule`.
- Adds the manual trigger endpoint `POST /api/v1/integrations/hubspot/sync` to the HubSpot router created by `hubspot-connection` (`services/backend-api/src/api/routes/hubspot_integration.py`).
- Mirrors `CrmEnrichment` in the worker's no-FK model file (`services/worker-service/src/models/__init__.py`).

Out of scope here: health component math (→ `crm-health-component`), profile card and timeline events (→ `crm-profile-and-timeline`), the connection UI (→ `hubspot-connection`).

## 3. Architecture Decisions

### 3.1 Python-level upsert (no ON CONFLICT)

Mirroring `usage_metrics.py:92-132`, the upsert is done with a `SELECT` then insert-or-update. This keeps tests running on SQLite (which does not support `INSERT ... ON CONFLICT DO UPDATE` in the same way). The pattern:

```python
row = db.query(CrmEnrichment).filter_by(
    organization_id=org_id, customer_email=email
).first()
if row is None:
    row = CrmEnrichment(organization_id=org_id, customer_email=email, created_at=now)
    db.add(row)
# always update fields
row.company_name = ...
row.last_synced_at = now
db.flush()
```

### 3.2 Renewal proxy heuristic

Among all open deals associated with a HubSpot contact (via company), filter to those with `dealstage != "closedwon"` and `dealstage != "closedlost"` and `closedate` is not null. Pick the one with the highest `amount`. Store its `closedate` as `renewal_date`, its `dealname` as `deal_name`, `dealstage` as `deal_stage`, `amount` as `deal_amount`, and its HubSpot deal ID as `hubspot_deal_id`. When no qualifying deal exists, all four fields are null. This is documented as a heuristic, not a guarantee.

### 3.3 Per-run work cap

To protect against enormous HubSpot portals, cap the sync at a configurable maximum contact page count (default: 100 pages × 100 contacts/page = 10,000 contacts). When the cap is reached, `logger.warning("hubspot_sync: per-run cap reached for org=%s; %s contacts pulled, stopping pagination", org_id, count)` — never silent. The cap constant is module-level so tests can override it.

### 3.4 Worker model drift (R4 mitigation)

Decision: the worker task (`_sync_org`) writes `CrmEnrichment` rows directly using the worker mirror model. It does **not** read `OrgAIConfig.health_weight_crm` — it only calls `update_customer_health(org_id, email, db)` (the backend-api health service handles the weight). This means `health_weight_crm` never needs to be in the worker's `OrgAIConfig` mirror for this aspect. The only columns that must be kept in sync between the two model files are the `crm_enrichment` table columns. A comment in `services/worker-service/src/models/__init__.py` above the `CrmEnrichment` mirror will read: `# R4: Keep in sync with services/backend-api/src/models/crm_enrichment.py`.

### 3.5 HubSpot client structure

`services/worker-service/src/clients/hubspot.py` — a plain class (no ABC), mirroring the approach of `services/worker-service/src/adapters/intercom.py` for httpx usage but following the simpler pull-API pattern (no webhook dispatch).

```
class HubSpotClient:
    BASE_URL = "https://api.hubapi.com"
    PAGE_SIZE = 100
    PER_RUN_PAGE_CAP = 100  # module-level constant

    def __init__(self, access_token: str, arr_property_name: str = "annualrevenue"):
        self._token = access_token
        self._arr_property = arr_property_name
        self._client = httpx.Client(
            base_url=self.BASE_URL,
            headers={"Authorization": f"Bearer {access_token}"},
            timeout=15.0,
        )

    def close(self):
        self._client.close()

    def __enter__ / __exit__: standard context manager

    def list_contacts(self) -> list[dict]:
        # GET /crm/v3/objects/contacts with paginaton + 429 handling
        # yields accumulated list; logs cap warning if hit

    def get_company(self, company_id: str) -> dict | None:
        # GET /crm/v3/objects/companies/{id}

    def get_open_deals_for_company(self, company_id: str) -> list[dict]:
        # GET /crm/v3/objects/deals?associations=companies&...
        # or use associations API; filter open deals only
```

The client raises a custom `HubSpotTransientError(Exception)` on 429 (after honoring `Retry-After` sleep) and on 5xx. The Celery task catches `HubSpotTransientError` and calls `self.retry(exc=...)`. All other exceptions are logged and re-raised (to let Celery decide).

**429 handling pattern:**

```python
if resp.status_code == 429:
    retry_after = int(resp.headers.get("Retry-After", "10"))
    logger.warning("HubSpot 429 for org; sleeping %ss", retry_after)
    time.sleep(retry_after)
    raise HubSpotTransientError(f"rate limited, retry after {retry_after}s")
```

### 3.6 Manual trigger endpoint

Added to the existing `services/backend-api/src/api/routes/hubspot_integration.py` router (created by `hubspot-connection`). Endpoint: `POST /api/v1/integrations/hubspot/sync`. Dependencies: `require_admin_or_owner`, `require_feature("hubspot_integration")`. Enqueues `sync_hubspot_org.delay(integration.id)` and returns `{"status": "queued", "integration_id": integration.id}`. Does not wait for completion.

## 6. Risks and Mitigations

### R4 — Worker/Backend Model Drift (flagged in PRD)

**Risk:** The `crm_enrichment` table columns are defined in both `services/backend-api/src/models/crm_enrichment.py` and `services/worker-service/src/models/__init__.py`. Any column added to one must be manually added to the other; there is no shared package enforcing consistency. The existing `OrgAIConfig` worker mirror at `src/models/__init__.py:577-595` already lacks `health_weight_usage` (which the backend model gained in `z5a6b7c8d9e0`).

**Mitigation for this aspect:**
1. The worker task (`_sync_org`) reads only `CrmEnrichment` columns it writes itself. It never reads `OrgAIConfig` weight fields.
2. A prominent comment is placed above the `CrmEnrichment` worker mirror: `# R4: Keep in sync with services/backend-api/src/models/crm_enrichment.py — no shared package exists.`
3. A test (`test_worker_and_backend_crm_enrichment_columns_match`) in `tests/test_hubspot_sync.py` imports column names from both model files and asserts they are identical. This fails at CI time if someone adds a column to one file but not the other.
4. The same test pattern should eventually be applied to `OrgAIConfig` and other drifted mirrors (tracked as follow-up in `TRACKING.md`).

**Worker's HubSpotIntegration mirror:** The fan-out task `sync_all_hubspot` queries `HubSpotIntegration` rows. The `hubspot-connection` aspect must add this mirror to `services/worker-service/src/models/__init__.py`. If it does not, this aspect adds it. Coordinate before implementation starts.

### R2 — Per-run Work Cap (from PRD)

Large HubSpot portals can have tens of thousands of contacts. The cap (`PER_RUN_PAGE_CAP = 100`, i.e. 10,000 contacts) is logged with a `WARNING` when hit, never silently skipped. The test `test_list_contacts_respects_per_run_cap` verifies both that pagination stops and that the warning is emitted. Operators needing full coverage can adjust the cap via a worker env var in a future iteration.

### R3 — Token Never Logged

The `HubSpotClient.__init__` stores the token in `self._token` but it must never appear in log output. The plan mandates: log messages in `hubspot_sync.py` and `hubspot.py` must use `integration_id` and `org_id` for identification, never `access_token` or `self._token`.

### R6 — `LLM_ENCRYPTION_KEY` Hard Dependency

The `get_decrypted_token()` helper (from `hubspot-connection`) raises `ValueError` when `LLM_ENCRYPTION_KEY` is unset (per `webhook_delivery.py:41-47`). The `sync_hubspot_org` task calls this at runtime. If the env var is missing, the task will raise before making any HTTP call. This is acceptable for the sync task (the operator would have already seen a 4xx when connecting), but the error must be logged clearly: `logger.error("hubspot_sync: LLM_ENCRYPTION_KEY unset for org=%s — cannot decrypt token; skipping", org_id)` and the task should return `{"status": "error", "reason": "missing_encryption_key"}` rather than crashing with an unhandled ValueError (which would cause unnecessary Celery retries on a non-transient configuration error).

```python
@shared_task(name="src.tasks.hubspot_sync.sync_all_hubspot")
def sync_all_hubspot() -> dict:
    from src.models import HubSpotIntegration
    with get_db_session() as db:
        integrations = db.query(HubSpotIntegration).filter(
            HubSpotIntegration.is_active == True
        ).all()
        if not integrations:
            return {"status": "no_integrations", "queued": 0}
        queued = 0
        for integ in integrations:
            try:
                sync_hubspot_org.delay(integ.id)
                queued += 1
            except Exception as exc:
                logger.error("sync_all_hubspot: failed to enqueue org=%s: %s", integ.organization_id, exc)
        return {"status": "queued", "queued": queued}


@shared_task(
    bind=True,
    max_retries=3,
    default_retry_delay=30,
    name="src.tasks.hubspot_sync.sync_hubspot_org",
)
def sync_hubspot_org(self, integration_id: int) -> dict:
    from src.models import HubSpotIntegration
    from src.clients.hubspot import HubSpotClient, HubSpotTransientError

    with get_db_session() as db:
        integ = db.query(HubSpotIntegration).filter(
            HubSpotIntegration.id == integration_id
        ).first()
        if not integ:
            return {"status": "not_found", "integration_id": integration_id}
        if not integ.is_active:
            return {"status": "inactive", "integration_id": integration_id}

        try:
            token = get_decrypted_token(integ)
        except ValueError as exc:
            logger.error("hubspot_sync: LLM_ENCRYPTION_KEY unset for org=%s — skipping", integ.organization_id)
            integ.last_sync_status = "error"
            integ.last_error = "missing_encryption_key"
            db.commit()
            return {"status": "error", "reason": "missing_encryption_key"}

        try:
            with HubSpotClient(token, arr_property_name=integ.arr_property_name or "annualrevenue") as client:
                result = _sync_org(integ.organization_id, db, client)
            now = datetime.utcnow()
            integ.last_synced_at = now
            integ.last_sync_status = "success"
            integ.last_error = None
            integ.contacts_synced = result["contacts_synced"]
            integ.contacts_matched = result["contacts_matched"]
            db.commit()
            return {"status": "success", **result}
        except HubSpotTransientError as exc:
            logger.warning("hubspot_sync: transient error for org=%s: %s", integ.organization_id, exc)
            integ.last_sync_status = "retrying"
            integ.last_error = str(exc)
            db.commit()
            raise self.retry(exc=exc)
        except Exception as exc:
            logger.error("hubspot_sync: unhandled error for org=%s: %s", integ.organization_id, exc, exc_info=True)
            integ.last_sync_status = "error"
            integ.last_error = str(exc)[:500]
            db.commit()
            raise
```
