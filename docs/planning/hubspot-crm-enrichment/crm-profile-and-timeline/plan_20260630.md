## Pinned shared contract (do not deviate)

**Serializer** (`customer_profile_serializer.py`): read the `(org, email)` `crm_enrichment`
row and emit these EXACT `Optional` fields (`None` when the row/value is absent):

```
crm_company_name, crm_lifecycle_stage, crm_arr, crm_renewal_date,
crm_deal_name, crm_deal_stage, crm_deal_amount
```

`CustomerProfileResponse` (customers.py) and `PublicCustomerProfile360` (public_api.py)
both gain the same 7 `Optional` fields — they fill automatically because both call the
shared serializer.

**Timeline** (`customer_timeline_service.py`): `_fetch_crm_events(db, org_id, email)` returns
`TimelineEvent`s with `source="hubspot"`, types limited to what a **single enrichment
snapshot** can derive:

- `crm_contact_synced` — anchored at `last_synced_at`.
- `crm_renewal_upcoming` — emitted when `renewal_date` is within the look-ahead window.

`crm_deal_stage_changed` is **DEFERRED to v2** — a single current-state snapshot cannot
detect a stage *change* (no per-sync history table in v1). Stated honestly here; do not
fake it.

One `extend()` call is added to `build_timeline` next to the churn `extend()` (the
Python-cursor-filtered group). **Sort/cursor logic is NOT touched.** New `Optional` payload
fields are added to BOTH the `TimelineEvent` dataclass and the `ActivityEvent` Pydantic model
and mapped in `_timeline_event_to_activity`.

**Frontend:** add `crm_contact_synced` + `crm_renewal_upcoming` literals (+ optional CRM
payload fields) to `ActivityEvent['type']` in `lib/api/customers.ts`; matching `eventIconMap`
entries (Lucide icon + `var(--chart-N)` via `color-mix(in oklch, …)` — **never** a hardcoded
color; HubSpot brand orange is allowed only on the Settings integration tile, never the
timeline). New `CrmCompanyCard` mirrors `UsageActivityCard` (self-contained `useQuery`
`retry:false` `staleTime` 5 min, three-state body skeleton/empty/data, shadcn `Card`, theme CSS
vars), placed near the top of the Overview tab, reading CRM fields off the profile response.

**Two independent tracks.** The BACKEND track and FRONTEND track share only the wire
contract above. At execution time they are two parallel agents (see §7).

### Shared dependency: the `CrmEnrichment` ORM model (read-only here)

Both this aspect's tests and the serializer/timeline reads need the
`src/models/crm_enrichment.py` model to exist (so SQLite test DBs `create_all` the table and
fixtures can insert rows). Because `hubspot-sync` is built in a parallel worktree, **agree on
one byte-identical model definition** so the merge is a no-op (two classes mapping the same
`__tablename__` in one `Base.metadata` otherwise raises "Table already defined").

Agreed definition (mirrors `customer_usage.py`; create it ONLY if `hubspot-sync` has not
already landed it — coordinate, do not duplicate-define):

```python
# services/backend-api/src/models/crm_enrichment.py
from datetime import datetime, timezone
from sqlalchemy import (Column, DateTime, Float, ForeignKey, Index, Integer,
                        String, UniqueConstraint)
from .base import Base

class CrmEnrichment(Base):
    __tablename__ = "crm_enrichment"
    id = Column(Integer, primary_key=True, index=True)
    organization_id = Column(Integer, ForeignKey("organizations.id", ondelete="CASCADE"),
                             nullable=False, index=True)
    customer_email = Column(String(255), nullable=False, index=True)
    company_name = Column(String(512), nullable=True)
    lifecycle_stage = Column(String(64), nullable=True)
    arr = Column(Float, nullable=True)
    renewal_date = Column(DateTime, nullable=True)
    deal_name = Column(String(512), nullable=True)
    deal_stage = Column(String(128), nullable=True)
    deal_amount = Column(Float, nullable=True)
    hubspot_contact_id = Column(String(64), nullable=True)
    hubspot_company_id = Column(String(64), nullable=True)
    hubspot_deal_id = Column(String(64), nullable=True)
    last_synced_at = Column(DateTime, nullable=True)
    created_at = Column(DateTime, nullable=False, default=lambda: datetime.now(timezone.utc))
    updated_at = Column(DateTime, nullable=False, default=lambda: datetime.now(timezone.utc),
                        onupdate=lambda: datetime.now(timezone.utc))
    __table_args__ = (
        UniqueConstraint("organization_id", "customer_email", name="uq_crm_enrichment_org_email"),
        Index("ix_crm_enrichment_org_email", "organization_id", "customer_email"),
    )
```

All serializer/timeline reads of this model use a **guarded lazy import** (`try: from
src.models.crm_enrichment import CrmEnrichment except ImportError: return …`) so this aspect
never hard-breaks if the model lands later.

     **no SQL cursor filter** — `build_timeline`'s Python `_after_cursor` handles it):
     ```python
     def _fetch_crm_events(db, org_id, email):
         try:
             from src.models.crm_enrichment import CrmEnrichment
         except ImportError:
             return []
         row = db.query(CrmEnrichment).filter(
             CrmEnrichment.organization_id == org_id,
             CrmEnrichment.customer_email == email,
         ).first()
         if not row:
             return []
         events = []
         if row.last_synced_at:
             events.append(TimelineEvent(
                 type="crm_contact_synced",
                 timestamp=_to_naive_utc(row.last_synced_at),
                 description="CRM contact synced from HubSpot"
                             + (f" — {row.company_name}" if row.company_name else ""),
                 source="hubspot", source_id=row.id,
                 company_name=row.company_name,
             ))
         if row.renewal_date and row.last_synced_at:
             rd = _to_naive_utc(row.renewal_date)
             now = datetime.utcnow()
             if now <= rd <= now + timedelta(days=CRM_RENEWAL_WINDOW_DAYS):
                 events.append(TimelineEvent(
                     type="crm_renewal_upcoming",
                     # anchored at detection time (last_synced_at), NOT the future
                     # renewal_date — keeps the timeline past-dated & cursor-safe.
                     timestamp=_to_naive_utc(row.last_synced_at),
                     description=f"Renewal upcoming on {rd.date().isoformat()}",
                     source="hubspot", source_id=row.id,
                     renewal_date=rd, deal_stage=row.deal_stage,
                     arr=float(row.arr) if row.arr is not None else None,
                 ))
         return events
     ```
     - **Timestamp rationale (honest):** `crm_renewal_upcoming` is anchored at `last_synced_at`
       (when we *detected* the upcoming renewal), not the future `renewal_date`. Anchoring on a
       future date would force the event permanently to the top and break the "reverse-chron of
       past activity" + cursor invariants. The renewal date rides the `renewal_date` payload +
       description. Both CRM events share `source_id = row.id`; their composite cursor key stays
       unique because `type` differs (`crm_contact_synced` < `crm_renewal_upcoming`).

## 6. Edge Cases & Error Handling

- **No `crm_enrichment` row** → serializer emits 7 `None`; `_fetch_crm_events` returns `[]`;
  `CrmCompanyCard` shows empty state. No errors.
- **`CrmEnrichment` model not yet merged** (parallel dev) → guarded lazy import returns
  `None` / `[]`; backend stays green.
- **`renewal_date` null / in the past / beyond `CRM_RENEWAL_WINDOW_DAYS`** → no
  `crm_renewal_upcoming` event (only `crm_contact_synced`).
- **`last_synced_at` null** → no CRM events emitted (both anchor on it).
- **Timezone:** all CRM timestamps pass through `_to_naive_utc` before sorting (mixed
  aware/naive safe, like every other source).
- **Cursor integrity:** CRM events are Python-cursor-filtered by `build_timeline._after_cursor`
  (like churn) — `build_timeline` sort/cursor code is untouched; the two CRM events share
  `source_id` but differ by `type`, keeping the composite cursor key unique.
- **Multi-tenancy:** every read filters `organization_id == org_id AND customer_email == email`.
- **Currency / dates on FE:** ARR/deal amount currency-formatted; `null` → render `—`; renewal
  within 30 days tinted `var(--chart-1)` (attention, not alarm).
- **Public timeline parity:** without the `PublicActivityEvent` field additions (B3 step 6) the
  CRM payload fields are silently dropped from `/api/public/v1/.../timeline` — included to avoid
  drift; the type/description/timestamp parity test still passes either way.
