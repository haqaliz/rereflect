## 0. Impact analysis (verified against source)

- **Model:** `src/models/custom_category.py` — `CustomCategory{organization_id, name(≤100), description?,
  category_type ∈ pain_point|feature_request|urgency|general, is_active}`. No unique constraint (dedup in route).
- **Internal CRUD to mirror:** `src/api/routes/categories.py` (`/api/v1/categories/custom`):
  - list (org-scoped, `?category_type=`, order by name), create (201, 409 dup on `(org,type,name)`),
    patch (name/description/is_active; type not editable; 409 on rename collision), delete (204 hard delete).
  - **Reuse strategy:** extract the create/update/delete/list bodies into a small **service module**
    `src/services/custom_category_service.py` so internal + public share one implementation. Characterization-lock
    the internal route first (M3), then refactor it onto the service, then build the public route on the same service.
- **Delete/rename warning:** `AutomationRule` (`src/models/automation_rule.py`) with
  `trigger_type == "feedback_category_match"` stores the matched category in `trigger_config` (JSON). The
  warning = "this name is referenced by N active automation rule(s)". **Determine the exact `trigger_config`
  key** holding the category value from the automation engine's `feedback_category_match` handler (dig pointed
  at `services/worker-service/.../analysis.py:173-181` and/or the automations route) before implementing the
  lookup — do not guess the key.
- **Auth:** `verify_api_key` + `require_scope("read")` (GET) / `require_scope("write")` (POST/PATCH/DELETE).
- **Route placement:** add to the public router (`src/api/routes/public_api.py` or a new
  `src/api/public/categories.py` mounted under the same `/api/public/v1` prefix). Static collection routes
  before any parametric `/{id}` — here paths are `/categories` and `/categories/{id}` (no collision, but keep order).

## 2. Implementation phases

### Phase 1 — Extract shared service (characterization-locked)
**Goal:** One implementation behind both internal + public routes, internal behavior unchanged.
**Files:** `src/services/custom_category_service.py` (new); `src/api/routes/categories.py` (refactor to call it);
`tests/test_categories.py` (regression guard).
**Steps (TDD):**
1. RED/GUARD: ensure `tests/test_categories.py` covers list/create(201,409)/patch(409,404)/delete(204). Add any
   missing characterization tests **before** refactoring.
2. GREEN: create `custom_category_service.py` with `list_categories(db, org_id, category_type=None)`,
   `create_category(db, org_id, name, description, category_type) -> CustomCategory (raises Conflict/…)`,
   `update_category(db, org_id, category_id, *, name, description, is_active)`,
   `delete_category(db, org_id, category_id) -> deleted CustomCategory`. Raise small typed errors
   (`DuplicateCategoryError`, `CategoryNotFoundError`) so each router maps to its own HTTP codes.
3. REFACTOR: point `categories.py` at the service; internal tests stay green (byte-identical behavior).
**Validation:** `pytest tests/test_categories.py -v`.
**Commit:** `refactor(categories): extract custom_category_service (char-locked internal CRUD)`

### Phase 2 — Rule-reference lookup helper
**Goal:** Cheap, org-scoped "is this category name referenced by a feedback_category_match rule?" check.
**Files:** `src/services/custom_category_service.py` (add `rules_referencing_category(db, org_id, name) -> list[str]`
returning rule names); `tests/test_custom_category_service.py`.
**Steps (TDD):**
1. Determine the `trigger_config` key from the automation engine (see §0) — write it into the helper.
2. RED: with an active `feedback_category_match` rule whose config targets "Billing", the helper returns
   that rule's name; with no such rule, returns `[]`; inactive rules excluded (decide: include inactive? →
   only active, since only active rules fire).
3. GREEN: implement (query `AutomationRule` by org + trigger_type, filter in Python on the config value).
**Validation:** `pytest tests/test_custom_category_service.py -v`.
**Commit:** `feat(categories): rules_referencing_category helper for delete/rename warning`

### Phase 3 — Public read + create
**Goal:** `GET /api/public/v1/categories` (read) + `POST /api/public/v1/categories` (write).
**Files:** public router; public request/response models with `extra="forbid"`
(`PublicCustomCategoryCreate`, `PublicCustomCategoryResponse`).
**Steps (TDD):**
1. RED: GET needs `read` (403 without), lists org-scoped, `?category_type=` filters; POST needs `write`
   (403 without), 201 on valid, 409 on dup `(org,type,name)`, 422 on bad `category_type` / unknown field
   (`extra="forbid"`) / name length; cross-org isolation (GET only returns caller org's rows).
2. GREEN: implement via `custom_category_service`.
**Validation:** `pytest tests/test_public_api_categories.py -v -k "list or create"`.
**Commit:** `feat(public-categories): GET + POST /api/public/v1/categories`

### Phase 4 — Public update + delete (+ warning header)
**Goal:** `PATCH /categories/{id}` + `DELETE /categories/{id}` with `X-Rereflect-Warning`.
**Files:** public router; `PublicCustomCategoryUpdate` (`extra="forbid"`, `{name?, description?, is_active?}`).
**Steps (TDD):**
1. RED: PATCH `write`-gated; 404 other-org/missing; 409 on rename collision; `category_type` in body → 422
   (not an allowed field). DELETE `write`-gated; 204; other-org/missing → 404; when the name is referenced
   by an active `feedback_category_match` rule, the 204 response carries header
   `X-Rereflect-Warning: category '<name>' is referenced by N automation rule(s): ...` (assert header present
   only when referenced, absent otherwise).
2. GREEN: implement. For DELETE, compute the warning **before** delete (need the name + rule check), then
   delete via service, then set the response header on the 204 (`Response.headers[...]`).
**Validation:** `pytest tests/test_public_api_categories.py -v`.
**Commit:** `feat(public-categories): PATCH + DELETE /api/public/v1/categories (+ rule-ref warning header)`

## 6. Edge cases & error handling

- Org scoping on every query; other-org id → 404 (not 403).
- `category_type` immutable on PATCH (reject in body via `extra="forbid"` — it's simply not a field on the
  update model, so it 422s).
- 409 semantics identical to internal (same message shape is fine; assert on status not exact string).
- Warning is **non-blocking** — delete always succeeds; header is advisory. No cascade, no rewrite of
  orphaned feedback strings or rules (documented behavior).
- `X-Rereflect-Warning` only set when ≥1 active rule references the name.
