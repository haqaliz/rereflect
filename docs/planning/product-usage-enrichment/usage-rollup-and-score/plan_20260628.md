## Phase 1 — `customer_usage` model + migration

## Phase 2 — `compute_usage_score` (RED→GREEN, pure function)

- **Files:** `services/backend-api/src/services/usage_score_service.py` (new); `tests/test_usage_score.py` (new).
- **Steps:** `compute_usage_score(rollup, now) -> int` blends:
  - **Recency** from `last_active_at`: ≤2d→100, ≤7d→80, ≤14d→60, ≤30d→40, ≤60d→20, else 5; `None`→neutral.
  - **Frequency** from `active_days_30d`: ≥20→100 … 0→low (define bands).
  - **Breadth** from `distinct_feature_count`: ≥5→100, scaled down.
  - Weighted blend (e.g. recency .5 / frequency .3 / breadth .2) → clamp 0-100.
  - **No rollup / all-None ⇒ 50.**
- **Tests (RED first):** recent+frequent+broad → >70; stale (>30d) → <40; empty → 50; monotonic in recency.
- **Validation:** `pytest tests/test_usage_score.py -v`.
- **Commit:** `feat(usage): recency+frequency+breadth usage score (neutral-50 fallback)`

## Phase 3 — Celery task `process_usage_event` (GREEN)

- **Files:** `services/worker-service/src/tasks/usage_metrics.py` (new); register in beat/include list if tasks are explicitly imported.
- **Steps:**
  1. `@shared_task(bind=True, max_retries=3, default_retry_delay=30) def process_usage_event(self, org_id, customer_email, event_type, event_name, occurred_at_iso, external_event_id, properties)`.
  2. Full idempotent dedup on `(org_id, external_event_id)`.
  3. Upsert `customer_usage` for `(org, email)`: bump `events_total`, advance `last_active_at`, recompute 7d/30d counts/active-days from `usage_event` rows, add `event_name` to `distinct_features`.
  4. `usage_score = compute_usage_score(rollup, now)`; save.
  5. Call `update_customer_health(org_id, customer_email, db)` once (recompute health incl. usage component).
- **Tests:** upsert increments + last_active advance; idempotent re-process; `update_customer_health` called once (mock).
- **Validation:** worker tests `cd services/worker-service && pytest tests/ -v`.
- **Commit:** `feat(usage): process_usage_event task — rollup upsert, score, health recompute`

## Phase 4 — Scheduled recompute (recency decay)

- **Files:** `services/worker-service/src/tasks/usage_metrics.py` (add `recompute_usage_scores`); beat schedule (where churn calibration's Monday beat lives, `worker-service/src/.../scheduler` or celery beat config).
- **Steps:** periodic (daily) task iterating orgs' `customer_usage` rows, recomputing `usage_score` (so recency ages even with no new events) and calling `update_customer_health` when the score moves ≥2.
- **Tests:** a rollup with `last_active_at` aged past thresholds → score drops on recompute with no new events.
- **Validation:** worker tests green; beat entry registered.
- **Commit:** `feat(usage): daily scheduled usage-score recompute (recency decay)`

## Phase 5 — Read API `GET /customers/{email}/usage`

## Edge cases & errors

- Event with email not seen before → create rollup (`first_seen_at`).
- Re-processed `messageId` → no double count.
- `customer_usage` read by `health-component` already defended (missing row → 50).
- Counts derived from `usage_event` within windows — ensure timezone-aware comparisons.
