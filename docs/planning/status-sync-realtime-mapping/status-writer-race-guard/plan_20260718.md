### Phase 2 — Race-safe conditional UPDATE (RED → GREEN)
- **Goal:** `apply_status_change_worker` applies via a guarded conditional UPDATE
  and writes an event only when it actually changed the row.
- **Files:** `services/worker-service/src/services/status_writer.py`;
  `tests/test_status_writer.py`.
- **Steps:**
  1. RED: add a regression test simulating a stale apply — set up a feedback at
     `new`; simulate a concurrent update to `resolved` (write directly), then call
     `apply_status_change_worker(..., new_status='in_review')` where the caller's
     observed `old_status` was `new`. Expect: **zero** events, returns False,
     final status stays `resolved`. Also a happy-path test: observed old matches
     current → exactly one event, status updated. Run → the stale test fails
     (current code clobbers to `in_review` + writes an event).
  2. GREEN: rewrite the applier to accept the caller-observed `old_status` and run
     a conditional Core `text()` UPDATE mirroring Zendesk's `_apply_zendesk_status`:
     `UPDATE feedback_items SET workflow_status=:new WHERE id=:id AND
     organization_id=:org AND workflow_status=:old` → if `rowcount == 1` write the
     single `FeedbackWorkflowEvent`, else return False and write nothing. Preserve
     signature keywords (`actor_label`, `metadata`, `organization_id`),
     `actor_id=None`, `event_type="status_changed"`, no outbound webhook.
  3. Update `jira_sync.py` / `asana_sync.py` call sites to pass the observed prior
     status (from the link's `*_status_category`/last-observed → they already read
     the current feedback status before applying; thread it through as
     `old_status`).
  4. REFACTOR: keep the SQL in one helper so the backend-api webhook port can
     mirror it verbatim.
- **Validation:**
  `cd services/worker-service && pytest tests/test_status_writer.py tests/test_jira_sync_task.py tests/test_asana_sync_task.py -v`
  (characterization stays green; new race test green).
- **Commit:** `fix(status-sync): race-safe conditional UPDATE in apply_status_change_worker`

## 6. Edge Cases & Error Handling
- Row already at target (no-op): conditional UPDATE matches 0 rows when old≠current
  or old==new → zero events. Ensure the equal-status short-circuit still returns
  False without an event.
- ORM identity-map staleness: prefer Core `text()` UPDATE + explicit refresh/expire
  of the ORM object if later read (Zendesk pattern) to avoid reading stale
  `workflow_status`.
- Multi-tenancy: `organization_id` in the WHERE clause (defense in depth).
