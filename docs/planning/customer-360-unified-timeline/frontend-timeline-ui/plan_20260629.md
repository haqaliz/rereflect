## Locked upstream contract (from aspect 1)

- `GET /api/v1/customers/{email}/timeline?before=<cursor>&limit=<n>` →
  `{ events: ActivityEvent[]; next_cursor: string | null }`.
- `ActivityEvent` adds optional `risk_level`, `reason_code`, `feature_name`, `source`, `gap_days` to
  the existing `type, description, timestamp, feedback_id, old_score, new_score`.
- **10 event `type` strings (exact):** `feedback_created`, `status_changed`, `health_score_changed`,
  `llm_analysis_generated`, `action_completed`, `churned`, `churn_recovered`, `usage_first_seen`,
  `usage_feature_adopted`, `usage_reactivated`.

### Phase 3 — Paginated `CustomerTimeline` card

- **Validation:** Vitest test: initial fetch no cursor; "Load more" calls with `before=<cursor>` and

- **Commit:** `feat(timeline-ui): paginated Customer Timeline card with load-more on profile`

## Acceptance criteria (from spec)

1. Each new event type renders with its own icon + tint + readable description.
2. `getTimeline` called without cursor initially; "Load more" uses `before=next_cursor`, appends, no dups.
3. "Load more" hidden/disabled when `next_cursor` null.
4. Loading → skeleton; empty → empty state; both match conventions.
5. Usage/churn interleaved in correct time order (backend-driven).
6. `npm run lint` + `npm run test` pass; CSS vars only (no hardcoded colors).

## Edge cases

- `next_cursor` null on first page → no "Load more", show events (or empty state if none).
- Email with special chars → `encodeURIComponent`.
- Re-render/refetch loops → stable React Query keys (`['customer-timeline', email]`).
- Theme: verify icons/tints in both light and `.dark`.
