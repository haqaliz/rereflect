## 2. Implementation phases (TDD)

### Phase 0 ‚Äî Characterization test for `/activity` (RED‚ÜíGREEN, no behavior change)

- **Goal:** lock the current `/activity` external contract before refactoring.
- **Files:** `tests/test_customer_timeline_endpoint.py` (new) ‚Äî or extend `tests/test_customer_profile.py`.
- **Steps:**
  1. Seed an org + customer with: 2 feedbacks, 1 workflow status_changed, 2 health-history rows,
     `llm_analyzed_at` set, 1 completed action.
  2. Assert `GET /api/v1/customers/{email}/activity` ‚Üí 200, `events` length ‚â§ 10, descending by
     `timestamp`, and that each existing `type` appears. Snapshot the field shape.
- **Validation:** `pytest tests/test_customer_timeline_endpoint.py -v` green against current code.
- **Commit:** `test(timeline): characterize existing /activity contract before refactor`

### Phase 1 ‚Äî Timeline service: port existing 5 sources (RED‚ÜíGREEN‚ÜíREFACTOR)

- **Goal:** `build_timeline(db, org_id, email, before=None, limit=20)` reproduces today's 5 event
  types, returns `(events, next_cursor)`.
- **Files:** `src/services/customer_timeline_service.py`, `tests/test_customer_timeline_service.py`.
- **Steps:**
  1. RED: write service tests asserting all 5 ported types are produced for a seeded fixture, in
     `(timestamp DESC, type ASC, source_id ASC)` order.
  2. GREEN: implement per-source queries, each `ORDER BY <ts> DESC LIMIT limit` and, when `before`
     is set, `WHERE <ts> < before`. `source_id` = the row id (feedback id, history id, churn id;
     synthetic stable id for the singleton llm/usage_first_seen events, e.g. `0`).
  3. Build a `TimelineEvent` dataclass/dict with `type, timestamp, description, source, source_id`
     + optional payload (`feedback_id, old_score, new_score, risk_level, reason_code, feature_name`).
  4. K-way merge: concatenate all source lists, sort by the composite key, truncate to `limit`.
  5. `next_cursor`: if exactly `limit` events returned, set cursor = encode(last event's
     `timestamp` [+ tiebreak]); else `None`.
- **Validation:** `pytest tests/test_customer_timeline_service.py -v`.
- **Commit:** `feat(timeline): shared customer_timeline_service porting the 5 activity sources`

### Phase 2 ‚Äî Add churn events (`churned`, `churn_recovered`)

- **Goal:** churn appears in the merged stream.
- **Steps:** RED test (seed a `customer_churn_events` row with `churned_at` + `reason_code`, and one
  with `recovered_at`) ‚Üí GREEN: query `customer_churn_events` for org+email; emit `churned` at
  `churned_at` (description includes `reason_code`) and, when `recovered_at` is set, a separate
  `churn_recovered` event at `recovered_at`.
- **Validation:** service tests green.
- **Commit:** `feat(timeline): churned + churn_recovered timeline events`

### Phase 3 ‚Äî Notable usage events (`usage_first_seen`, `usage_reactivated`, `usage_feature_adopted`)

- **Goal:** synthesize bounded, non-flooding usage events from `usage_events` + `customer_usage`.
- **Steps:**
  1. RED tests:
     - `usage_first_seen` once, at `customer_usage.first_seen_at` (skip if no row/null).
     - `usage_reactivated` only when an event's `occurred_at` follows a ‚â• `DORMANCY_DAYS` gap from
       the previous event; back-to-back events ‚Üí none; multiple reactivations ‚Üí one each.
     - `usage_feature_adopted` once per distinct `event_name`, at its earliest `occurred_at`.
  2. GREEN: load this customer's `usage_events` ordered by `occurred_at ASC`, bounded to
     `USAGE_SCAN_WINDOW_DAYS` (and, when paging, also `occurred_at < before` is **not** applied to
     the scan input for gap/first-occurrence correctness ‚Äî instead compute all notable events once,
     then filter the resulting synthetic events by the `before`/`limit` window like any other source).
     *Document this clearly: derivation runs on the full (bounded) history; pagination filters the
     derived events, not the raw scan.*
  3. Merge synthetic usage events into the same stream + ordering.
- **Validation:** service tests green; add a "usage doesn't flood" test (1000 raw events of the same
  `event_name` with no gaps ‚Üí exactly 1 `usage_first_seen` + 1 `usage_feature_adopted`, 0 reactivations).
- **Commit:** `feat(timeline): notable usage events (first-seen, reactivation, feature-adopted)`

### Phase 4 ‚Äî Cursor correctness (R4)

- **Goal:** paging never skips/repeats events sharing a `timestamp`.
- **Steps:** RED test: seed ‚â• `limit`+5 events including ‚â•2 sharing an identical timestamp across
  different sources; page through with `limit=5`, collect all events, assert the concatenation equals
  the unpaged full list with **no dup, no gap**. GREEN: ensure the cursor encodes the composite
  tiebreak `(timestamp, type, source_id)` and every per-source query + the merge apply it identically;
  the `before` predicate becomes "strictly before the composite cursor."
- **Validation:** service tests green.
- **Commit:** `test(timeline): cursor pagination stable across equal timestamps`

### Phase 5 ‚Äî Wire endpoints: new `/timeline` + refactor `/activity`

- **Goal:** expose the service; `/activity` delegates with `limit=10`.
- **Files:** `src/api/routes/customers.py`.
- **Steps:**
  1. Extend `ActivityEvent` schema with optional `risk_level: str | None`, `reason_code: str | None`,
     `feature_name: str | None`, `source: str | None`. (Keep existing fields; additive only.)
  2. Add `TimelineResponse(BaseModel){ events: List[ActivityEvent]; next_cursor: Optional[str] }`.
  3. Add `GET /{email}/timeline` with `before: Optional[str] = Query(None)`,
     `limit: int = Query(20, ge=1, le=100)`, `current_org`, `db`; parse/validate `before` (ISO; 422
     on bad value); call `build_timeline`; return `TimelineResponse`. Same `require_feature(...)`
     dependency as siblings (no-op in OSS) for consistency.
  4. Refactor `get_customer_activity` to `events, _ = build_timeline(db, org.id, email, limit=10)`
     and return `CustomerActivityResponse(events=events)`.
- **Validation:** `pytest tests/test_customer_timeline_endpoint.py tests/test_customer_profile.py -v`
  ‚Äî Phase-0 characterization still green (now also showing usage/churn where seeded); new endpoint
  tests (pagination, 422 on bad cursor, cross-org isolation) green.
- **Commit:** `feat(timeline): GET /customers/{email}/timeline + /activity delegates to service`

## 6. Edge cases & error handling

- Customer with **no** health row (`customer_health` missing): still return feedback/usage/churn
  events; don't 500. (Today's `/activity` guards `if health:` ‚Äî preserve.)
- Customer with **no** usage row / no usage events: emit zero usage events, no error.
- `before` cursor in the future / malformed: 422 with a clear message.
- `limit` out of range: 422 via `Query(ge=1, le=100)`.
- **Multi-tenancy:** every source query filters `organization_id == current_org.id`; cross-org test
  asserts another org's churn/usage/feedback never leak.
- Equal timestamps across sources: deterministic tiebreak (Phase 4).
- Naive vs aware datetimes: source columns are naive UTC (`datetime.utcnow`) except `usage_events`
  uses `datetime.now(timezone.utc)` and `received_at` aware ‚Äî **normalize to naive UTC** in the
  service before comparing/sorting (explicit helper; add a test mixing both).
