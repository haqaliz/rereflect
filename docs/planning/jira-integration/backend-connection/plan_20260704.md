## Phase 2 — `JiraClient` (class + error taxonomy + `validate`)

- **Goal:** the REST client shell with Basic auth and the `validate()` method (create/list methods are
  added in the `backend-create-issue` aspect).
- **Prerequisites:** none (independent of Phase 1; can be parallelized).
- **Files:** create `src/services/jira_client.py`.
- **RED — `tests/test_jira_client.py` (mock `httpx`):**
  1. constructs with `(site_url, email, api_token)`; base URL = `https://{site}.atlassian.net/rest/api/3`.
  2. uses HTTP Basic auth (`httpx.Client(..., auth=(email, api_token))` — assert the `auth` tuple).
  3. `validate()` happy path (mock `GET /myself` → 200 `{accountId, displayName, emailAddress}`) returns
     an object/dict exposing `account_id` + `display_name`.
  4. 401/403 → raises `JiraAuthError`; 429 or 5xx → `JiraTransientError`; 404 → `JiraNotFoundError`.
  5. the token is never present in `repr`/logs (assert `str(client)` excludes it — mirror HubSpot client R3).
- **GREEN:**
  - Define `JiraError(Exception)` + `JiraAuthError`, `JiraTransientError`, `JiraNotFoundError` subclasses.
  - `class JiraClient`: `__init__(self, site_url, email, api_token)`; store creds privately; build
    `httpx.Client(base_url=f"{site_url}/rest/api/3", auth=(email, api_token), timeout=15.0)`.
    Context-manager (`__enter__/__exit__/close`).
  - `_get(path)/_post(path, json)` helpers that map status → the error taxonomy (401/403→auth,
    429/5xx→transient, 404→notfound, else raise_for_status).
  - `validate(self)` → `GET /myself`.
  - Assume `site_url` arrives already canonicalized (`https://{site}.atlassian.net`) — normalization
    lives in the route (Phase 3), not the client.
- **Validation:** `pytest tests/test_jira_client.py -v` green.
- **Commit:** `feat(jira): JiraClient (Basic auth) + validate + error taxonomy`

## Phase 3 — Connect / status / disconnect / test routes

- **Goal:** the connection lifecycle endpoints under `/api/v1/integrations/jira`.
- **Prerequisites:** Phases 1 & 2.
- **Files:** create `src/api/routes/jira_integration.py`.
- **RED — `tests/test_jira_connection.py` (patch `src.api.routes.jira_integration.JiraClient`):**
  1. `POST /connect` valid token → `200 {connected:true, site_url, email, token_hint, account_id,
     display_name}`; **`api_token` NOT in the response body**; DB row's stored `api_token != plaintext`
     (encrypted).
  2. invalid token (client raises `JiraAuthError` in `validate`) → **422** with a clear message; no row persisted.
  3. `LLM_ENCRYPTION_KEY` unset (monkeypatch `encrypt_api_key` to raise `ValueError`, or unset env) → **422**, not 500.
  4. **site_url normalization**: inputs `acme`, `acme.atlassian.net`, `https://acme.atlassian.net`,
     `https://acme.atlassian.net/` all persist canonical `https://acme.atlassian.net`; a non-atlassian
     host (`https://acme.example.com`) → **422** (Cloud-only slice).
  4b. **SSRF**: a `*.atlassian.net` host that resolves to a private/loopback/link-local address
     (mock `socket.getaddrinfo` → e.g. `169.254.169.254` / `10.0.0.1` / `127.0.0.1`) → **422**;
     a public-resolving host passes. (Primary SSRF gate; the client re-asserts scheme+suffix.)
  5. **reconnect reuses the row**: second `POST /connect` for the same org updates the same row id,
     rotates `api_token` + `token_hint`, sets `is_active=true` (no second row; unique constraint holds).
  6. `GET /status` connected → full status; not-connected → `{connected:false}`.
  7. `DELETE /disconnect` → `is_active=false`; a pre-seeded `FeedbackJiraIssue` row still exists after (preserved).
  8. `POST /test` valid → `{success:true}`; token now invalid → `{success:false}` (200 body, **never 500**).
  9. **unlocked**: all of the above pass for an org on the **free** plan.
- **GREEN:**
  - `router = APIRouter(prefix="/api/v1/integrations/jira", tags=["jira"])`.
  - Pydantic: `JiraConnectRequest{site_url, email, api_token}` (all `min_length=1`),
    `JiraConnectResponse` / `JiraStatusResponse` (no token field), `JiraTestResponse{success, message}`.
  - `_normalize_site_url(raw)`: strip scheme/whitespace/trailing slash, allow bare subdomain, require the
    host to end in `.atlassian.net` (else `HTTPException(422)`), return `https://{host}`.
  - **SSRF gate (primary — MEDIUM finding):** after normalizing, resolve the host and **reject any
    address in loopback / RFC1918-private / link-local ranges** before constructing `JiraClient`, e.g.:
    ```python
    import ipaddress, socket
    for info in socket.getaddrinfo(host, 443):
        addr = ipaddress.ip_address(info[4][0])
        if addr.is_loopback or addr.is_private or addr.is_link_local:
            raise HTTPException(422, "site_url resolves to a disallowed address")
    ```
    This is the real gate on untrusted operator input (a `*.atlassian.net` host whose DNS points at
    `169.254.169.254` / internal ranges). The `JiraClient` already re-asserts https + host-suffix as
    defense-in-depth (see commit `174a406`), but the DNS/IP check lives here to keep the client
    unit-testable. **Test it with `patch("...jira_integration.socket.getaddrinfo")`** returning a private
    IP → assert 422, and a public IP → passes.
  - `POST /connect` [`Depends(require_admin_or_owner)`]:
    normalize → `JiraClient(...).validate()` (map `JiraAuthError`→422, transient→502) →
    `encrypt_api_key(api_token)` wrapped in `try/except ValueError → 422` → `get_key_hint` → upsert by
    `organization_id` (reuse existing row if present) → return metadata only.
  - `GET /status`, `DELETE /disconnect` (soft; **no CRM purge call**), `POST /test`
    (decrypt → `validate()` → success/failure, catch everything, never 500).
  - `get_decrypted_token(integration)` module-level helper → `decrypt_api_key(integration.api_token)`
    (exported for `backend-create-issue`).
  - `_get_active_integration(db, org_id)` + `_require_active_integration(...)` helpers (mirror Linear).
- **Validation:** `pytest tests/test_jira_connection.py -v` green.
- **Commit:** `feat(jira): connect/status/disconnect/test routes (token-paste, encrypted)`

  module (never hit the network). Follow the Linear/HubSpot fixture style (`db`, `client` TestClient,
  `test_organization`, `test_user`).

- **Full-suite regression** — `pytest tests/ -v` stays green (esp. `plans.py` and model-import touchpoints).
- **Manual (optional, end of aspect)** — with a real Atlassian API token: `POST /connect` against a test
  Jira site → `GET /status` shows connected → `POST /test` succeeds → `DELETE /disconnect`.
- Run all backend tests from `services/backend-api` with the venv active.

## Edge Cases & Error Handling

- Missing/blank `site_url|email|api_token` → 422 (Pydantic `min_length=1`).
- Non-`atlassian.net` host → 422 (Cloud-only).
- Invalid/expired token at connect → 422; transient Jira 5xx/429 at connect → 502.
- `LLM_ENCRYPTION_KEY` unset → 422 (never 500) — the HubSpot "R6 safeguard".
- `POST /test` must swallow all exceptions into `{success:false}` — never 500.
- Every route scopes by `organization_id` from JWT; one active row per org (unique constraint).
- Jira is **not** a CRM — do not call `crm_integration_common.another_crm_active` / `purge_crm_enrichment`.
