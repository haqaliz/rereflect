## Architecture decisions (locked)

1. **Extend the existing AutomationEngine** (backend) with a `churn_probability_threshold`
   trigger + a `run_playbook` action + `off|shadow|active` `mode`. Health/`risk_level`
   auto-runs flow through the backend engine (already dispatched from
   `health_score_service`).
2. **Worker churn-probability seam is an ISOLATED focused evaluator**, NOT a full-engine
   mirror. The worker cannot import backend-api (confirmed: `clients/asana.py:5`,
   `email_parser.py:6`). Mirroring the whole engine would silently activate the currently
   **dead** worker dispatch of `feedback_category_match`/`sentiment_pattern`
   (`analysis.py:174-186` imports a nonexistent `src.services.automation_engine` →
   ImportError swallowed). To honour PRD G5 (no regressions), we add a small
   `automation_churn_trigger.py` in the worker that evaluates **only**
   `churn_probability_threshold`, reusing the same Redis cooldown scheme + the mirrored
   models. The run_playbook execution slice (insert `ChurnPlaybookExecution` + enqueue) is
   duplicated per-service — the established mirror tax (cf. `playbook_engine.py`,
   `status_sync_core.py`).
3. **`mode` is the single source of truth** for evaluation; `is_active` becomes a derived
   alias (`is_active == (mode != 'off')`), write-through both ways in model/schema.
4. **Activation semantics = conservative** (user-approved): on `shadow → active`, seed each
   above-threshold customer's cooldown so only *new* crossings fire; shadow shows the
   would-run count as the preview. (Implemented in Phase 7.)
5. **Trigger-source label:** reuse the pre-declared `ChurnPlaybookExecution.triggered_by
   = "auto_probability"` for all rule-driven runs (no new enum value; simplest).
6. **No plan-tier gating** (OSS unlocked); the auto path adds no `require_feature`.
7. **Redis required** in the self-host baseline (already the Celery broker) — documented in
   `SELF_HOSTING.md`; cooldown correctness depends on it (PRD Risk #2, option a).

## Testing Strategy

- **Backend/worker:** `pytest tests/ -v -k "automation or playbook or timeline or probability"`
  per service (scope runs — avoid the pre-existing full-suite `test_report_ws.py` segfault).
- **Frontend:** `npm run test` + `npm run lint`.
- **Integration/manual:** `./start-all.sh` (Redis + worker + backend + frontend). Seed a
  playbook + a customer with high `churn_probability`; create a `churn_probability_threshold`
  rule in **shadow** → confirm a shadow execution logged, **no** playbook run; flip to
  **active** → confirm existing at-risk customer is cooldown-seeded (no immediate run), then
  trigger a *new* crossing (re-run probability update) → confirm one queued
  `ChurnPlaybookExecution` + a `playbook_auto_run` timeline entry.

## Edge Cases & Error Handling

- Playbook missing / other-org / inactive → action error, no execution, no crash.
- Redis down → cooldown disabled (documented); 60-min playbook-engine rate-limit backstops.
- Customer with null `churn_probability` → trigger returns False.
- `mode='off'` rule never selected; `shadow` never executes actions.
- Concurrent recompute + health seam for the same customer → per-(rule,customer) cooldown
  (shared key scheme) prevents double-fire across both seams.
- All queries `organization_id`-scoped; `run_playbook` re-verifies playbook org ownership.
