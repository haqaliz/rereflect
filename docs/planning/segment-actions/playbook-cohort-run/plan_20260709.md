### Phase 2 — Extend RunBatchFilters + resolution
- **Files:** `src/schemas/churn_playbook.py` (`RunBatchFilters`), `src/api/routes/playbooks.py`
  (`_apply_run_batch_filters` L128-157, `run_playbook_batch` L450-515), tests.
- **Steps:**
  1. RED tests: run-batch with `emails` → queues per resolved customer (skip unknown); with `segment` →
     whole-cohort org-scoped; `segment`+probability → AND; invalid segment → 422; back-compat (no new
     fields) → unchanged.
  2. Extend `RunBatchFilters`: add `emails: Optional[list[str]]`, `segment: Optional[str]` (validate against
     `SEGMENT_SLUGS`). Keep probability fields.
  3. Extend `_apply_run_batch_filters`: if `emails`/`segment` present, build the customer query via the
     shared `cohort_service` helpers (`_apply_customer_filters` for segment; email list for emails),
     intersected with the playbook's probability band **only when probability fields are also given**
     (AND semantics). If only probability fields → unchanged path. Org-scope throughout.
  4. Keep per-customer `ChurnPlaybookExecution` creation + `app.send_task(...)` unchanged.
- **Validation:** `pytest tests/ -v -k run_batch`.
- **Commit:** `feat(segment-actions): run playbook on cohort (emails/segment) via run-batch`

### Phase 3 — Queue-safety cap + affected-count
- **Files:** `src/api/routes/playbooks.py`, tests.
- **Steps:**
  1. RED tests: cohort > 500 → 422 (`"cohort of N exceeds batch cap of 500; narrow the filter"`), nothing
     queued; `?count_only=true` (or a `count` in response) → returns `{matched}` without queuing.
  2. Compute resolved count up front; if `> RUN_BATCH_MAX_CUSTOMERS` → `HTTPException(422, ...)` before any
     execution rows created. Ensure the **daily-limit check accounts for full cohort size up front** (not
     per-iteration).
  3. Add `matched` to the run-batch response; support `count_only` dry-run for the UI preview.
  4. Docs: update run-batch entry in `docs/API.md` + docstring (new filters, cap, count_only).
- **Validation:** `pytest tests/ -v -k run_batch`.
- **Commit:** `feat(segment-actions): run-batch cohort cap (500) + affected-count preview`

## 5. Env & secrets
- None new. Celery/Redis for full integration (dispatch is mocked in unit tests).

## 6. Edge cases
- Both segment + probability → AND (documented).
- Empty resolved cohort → 0 queued, `matched:0` (not an error).
- Cohort > cap → 422, atomic (nothing queued).
- Daily-limit interaction: if cohort would exceed remaining daily allowance → existing limit error path
  (confirm it triggers on full cohort size, not mid-loop).
- Invalid segment slug → 422.

## 7. Agent notes

- Import cohort resolution from `bulk-actions-api`'s shared module; do not re-implement segment filtering.
- Checkpoint: confirm AND-vs-OR semantics for segment+probability with the integrator if ambiguous (plan
  assumes AND).
