## 3. Implementation phases (TDD)

### Phase 1 — Migration: provider/dim columns + length-based backfill
- **Files:** new `alembic/versions/<rev>_add_embedding_provider_dim_to_mappings.py`;
  `models/query_template_mapping.py`.
- **Steps:**
  1. **RED:** `tests/test_mapping_provider_columns.py` — after `Base.metadata` / migration,
     a `QueryTemplateMapping` has `embedding_provider` and `embedding_dimension` attributes;
     a row with a 1536-len embedding backfills to `('openai', 1536)`; a row with a null/short
     vector backfills to stale (provider `None` / dimension `None`).
  2. **GREEN:** add columns to the model (`embedding_provider = Column(String(50), nullable=True)`,
     `embedding_dimension = Column(Integer, nullable=True)`). Write the migration: `add_column`
     both (nullable), then a data migration using `len(question_embedding)`:
     `UPDATE ... SET embedding_provider='openai', embedding_dimension=1536 WHERE
     json_array_length(question_embedding) = 1536` (use a portable length check; for JSON
     column, load + len in a Python loop if SQL length fn is unavailable). Leave non-1536 /
     null rows with null provider+dim. Optional index `(embedding_provider, embedding_dimension)`.
     `downgrade` drops index + columns.
  3. **VERIFY:** `alembic upgrade head` then `alembic downgrade -1` round-trips on a scratch DB.
- **Validation:** `pytest tests/test_mapping_provider_columns.py -v`; migration up/down clean.
- **Commit:** `feat(copilot): provider/dim columns on query_template_mappings + length-based backfill`

### Phase 2 — Matcher routes through the embedding layer + skips mismatched rows
- **Files:** `services/backend-api/src/services/copilot/template_matcher.py`;
  `tests/test_template_matcher.py` (update existing 15).
- **Steps:**
  1. **RED:** update/extend `test_template_matcher.py`:
     - `generate_embedding` no longer hardcodes 1536; parameterize fixtures for 768 and 1536.
     - `find_match(question, org_id, db, embedder=...)` uses the injected embedder.
     - **Skip rule:** given mappings tagged `('openai',1536)` and an active 768 embedder, those
       rows are excluded from cosine compare (no exception, no false match); and vice versa.
     - Match still returns when provider/dim agree and cosine ≥ 0.85.
  2. **VERIFY RED.**
  3. **GREEN:** remove the global `_EMBEDDING_DIMS = 1536` and the hardcoded
     `_call_embedding_api` OpenAI client. `find_match` accepts the resolved embedder (passed in
     by `copilot_ws`); `generate_embedding` delegates to `embedder.embed`; validation compares
     against `embedder.dimension` not a constant. In the cosine loop, **filter mappings** to
     those whose `embedding_provider == active.provider and embedding_dimension == active.dimension`.
  4. **REFACTOR:** keep `normalize_question` / `cosine_similarity` untouched.
- **Validation:** `pytest tests/test_template_matcher.py -v`.
- **Commit:** `feat(copilot): template matcher uses pluggable embeddings + provider-aware skip`

## 5. Edge cases & error handling
- Mixed-provider mappings coexisting (post-switch, pre-reseed): matcher must silently skip the
  wrong-provider set — never cosine across dims (would crash on `zip` mismatch or give garbage).
- Backfill of a row whose JSON is `null` → stale, excluded.
- Seeding when embedding endpoint down → log, continue boot.
- All paths `organization_id`-scoped; system templates remain `organization_id IS NULL`.
