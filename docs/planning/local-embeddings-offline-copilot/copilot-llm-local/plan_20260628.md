> Consumer aspect #2. Makes the Copilot's **answer generation** (NL→SQL, analysis, reports) run on
> the local-capable provider so a keyless local-LLM org gets an end-to-end working Copilot, with
> the existing SQL safety enforced for any provider and an honest weak-model UX.

## 1. Scope & impact
- **Service:** `services/backend-api`.
- **DB change:** none.
- **Existing files modified:** `api/routes/copilot_ws.py` (the LLM-provider gate at L509-543 and
  the generation calls), the LLM-calling generators (`SQLGenerator` and the analysis/report
  generators it sits beside under `src/services/copilot/`), and the SQL safety validator (verify

- **New deps:** none (reuses the local-capable provider pattern already in the codebase).

## 2. Current behaviour to change (confirmed)
`copilot_ws.py` L509-543:
- Resolves `provider`/`model` from `OrgAIConfig`, then `api_key = resolve_org_byok_key(...)`.
- L533-541: **if `not api_key` → sends an error and returns.** ← the hard gate that blocks every
  keyless local org from using the Copilot at all.
- L580-589: `SQLGenerator().generate(question, org_id, plan, context, api_key=api_key, model=model)`
  — generation is OpenAI-BYOK-keyed; needs a local-capable path (base_url, keyless).

## 3. Implementation phases (TDD)

### Phase 1 — Provider resolution that accepts local/keyless (replace the hard gate)
- **Files:** `copilot_ws.py` (L509-543); a small helper, e.g.
  `src/services/copilot/llm_resolver.py` (mirrors the analysis pipeline's selection — local via
  `OrgAIConfig.base_url` + `_LOCAL_PROVIDERS={"ollama","openai_compatible"}`, else cloud BYOK).
- **Steps:**
  1. **RED:** `tests/test_copilot_llm_resolution.py`:
     - org with `default_provider="openai_compatible"` + `base_url` set + no key → resolver
       yields a usable generation config (provider, model, base_url, api_key=None), **no error**.
     - org with `default_provider="openai"` + valid BYOK key → yields key-based config (unchanged).
     - org with no key AND no base_url → yields a sentinel meaning "unconfigured" (handled in P3),
       NOT a crash.
  2. **VERIFY RED.**
  3. **GREEN:** implement the resolver; replace the L533-541 `if not api_key: return error` block
     so it only short-circuits when truly unconfigured (no key AND no local base_url). For local
     providers, proceed with `api_key=None` + `base_url`.
  4. **REFACTOR.**
- **Validation:** `pytest tests/test_copilot_llm_resolution.py -v`.
- **Commit:** `feat(copilot): resolve local-capable LLM provider for generation (drop hard BYOK gate)`

### Phase 2 — Generators accept base_url / keyless local
- **Files:** `SQLGenerator` + the analysis/report generators under `src/services/copilot/`.
- **Steps:**
  1. **RED:** generator tests — `generate(..., base_url=..., api_key=None, model=...)` builds an
     OpenAI-compatible client against `base_url` (mocked) and returns SQL; cloud path unchanged.
     Add a test that the **prompt has a plain fallback** when the provider may not support
     OpenAI-only features (JSON mode / function calling) — R2.
  2. **VERIFY RED.**
  3. **GREEN:** thread `base_url` through the generators; construct the client like the worker's
     `OpenAICompatibleProvider` (dummy key when keyless). Where generation relied on JSON-mode /
     function-calling, add a plain-text-prompt + parse fallback gated on provider capability.
  4. **REFACTOR:** factor the client construction into one helper shared by the generators.
- **Validation:** `pytest tests/ -v` (generator tests).
- **Commit:** `feat(copilot): NL→SQL/analysis/report generation supports local/OpenAI-compatible endpoints`

### Phase 3 — Provider-agnostic SQL safety + honest weak-model UX
- **Files:** the SQL safety validator (locate under `src/services/copilot/`; the read-only,
  org-scoped, ≤3-join, schema-whitelist, 5s-timeout, row-limit checks), `copilot_ws.py` (the
  failure/empty branches around L596-612), and the WS error/`unconfigured` messaging.
- **Steps:**
  1. **RED:** `tests/test_copilot_sql_safety_provider_agnostic.py`:
     - a malicious/unsafe SQL string (e.g. `DROP`, cross-org, 5-join, subquery) — **as if emitted
       by a local model** — is rejected by the validator regardless of provider.
     - generation that returns invalid/unsafe SQL → user receives the honest "couldn't turn that
       into a safe query with the current model" message (hint: stronger model), **no 500**.
     - fully-unconfigured org → honest "configure a model in Settings → AI" message, no crash.
     - quality guardrail: a fixed list of ~10 canned questions against a mocked local model →
       ≥70% produce valid, org-scoped, safety-passing SQL; the rest hit the graceful path.
  2. **VERIFY RED.**
  3. **GREEN:** ensure the validator runs unconditionally on every generated SQL (it likely
     already does inside `SQLExecutor`/generator — make it explicit and covered). Add the
     weak-model + unconfigured messages in `copilot_ws.py`. Confirm execution failures already
     `db.rollback()` (they do at L612) and surface a friendly message rather than an error type.
  4. **REFACTOR.**
- **Validation:** `pytest tests/ -v` (safety + ws tests).
- **Commit:** `feat(copilot): provider-agnostic SQL safety + honest weak-model/unconfigured UX`

  4. **REFACTOR.**

## 5. Edge cases & error handling
- Unconfigured org (no key, no base_url) → honest message, never 500.
- Local model emits unsafe/invalid SQL → validator rejects → graceful "couldn't answer safely".
- Local model ignores JSON-mode → plain-prompt fallback parse; if parse fails → graceful path.
- Local endpoint unreachable mid-request → caught, friendly error, `db.rollback()`.
- All SQL execution remains org-scoped (`params={"org_id": org.id}`); validator enforces it.
