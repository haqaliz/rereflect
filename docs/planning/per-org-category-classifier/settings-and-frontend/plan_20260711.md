## Contract already provided (verified by reading, not assumed)

- `GET /api/v1/settings/ai/classifier/accuracy?classifier_type=<type>` and
  `POST /api/v1/settings/ai/classifier/rollback?classifier_type=<type>` — both default `"sentiment"`
  when the param is omitted (`classifier_accuracy.py:34,123,148`). Passing `classifier_type=category` for
  an org with no active category model returns the existing honest empty state
  (`has_model=false, label_count=0, is_ready=false, min_labels=20, history=[]` — `classifier_accuracy.py:90-100`),
  which `ClassifierAccuracyCard` already renders as "No model yet" (`:154-158` today). **No new empty-state
  branch is needed** — the category card gets this for free once it sends the right query param.
- `ClassifierAccuracyResponse.classifier_type: str` already exists in the response schema
  (`lib/api/classifier-accuracy.ts:25`) — currently unused by the frontend; safe to read for a future
  cross-check but not required for this slice.
- `category_classifier_mode` will exist on `AISettings`/`AISettingsUpdate` once `data-and-config` merges
  (same shape as `classifier_mode` today: `off|shadow|auto`, default `"off"`, validated + sklearn-guarded
  server-side). This aspect treats it as an opaque `string` field, mirroring how `classifier_mode` is
  already typed on the frontend (`lib/api/ai-settings.ts:18,29`) — no new validation logic needed
  client-side; 422s surface via `err.response.data.detail` exactly like the existing sentiment toggle.

## Decisions (locked for this plan)

- **`classifierType` prop, not a full config object.** `ClassifierAccuracyCard` takes
  `classifierType?: string` (default `'sentiment'`), matching the query-param name and the response
  field name 1:1 — no new enum type introduced, consistent with how `classifier_type` is already just
  `str` end-to-end on the backend.
- **Copy source of truth:** a local `TYPE_COPY` map inside `ClassifierAccuracyCard.tsx` (not exported),
  keyed by `classifierType`, so per-type strings live next to the component that renders them:
  ```ts
  const TYPE_COPY: Record<string, { label: string; trainedOn: string; note?: string }> = {
    sentiment: {
      label: 'Sentiment',
      trainedOn: "your team's sentiment corrections",
    },
    category: {
      label: 'Category',
      trainedOn:
        "your team's category corrections; promoted only when it beats the keyword " +
        "categorizer on your held-out data",
      note: 'Evaluated on labels the keyword categorizer can produce.',
    },
  };
  const copy = TYPE_COPY[classifierType] ?? { label: classifierType, trainedOn: `your team's ${classifierType} corrections` };
  ```
  Title becomes `` `${copy.label} Corrections Classifier Accuracy` ``; subtitle becomes
  `` `Per-org TF-IDF + logistic regression, trained on ${copy.trainedOn}. We recommend shadow mode until this history is substantial.` ``
  and, only when `copy.note` is set, a second `<p>` renders it under the subtitle (category only — keeps
  the sentiment card's exact prior text unaffected in substance, only interpolated).
- **Sentiment behavior is unchanged in substance.** Default `classifierType='sentiment'` reproduces the
  exact current title ("Corrections Classifier Accuracy" with "Sentiment" prefix — text changes from
  static to interpolated but conveys the same meaning; no existing test asserts the literal previous
  string, verified against `ClassifierAccuracyCard.test.tsx` — it only regexes on `/no model/i`, count
  patterns, percentages, and decision labels, never the header copy). If this is judged too risky, the
  fallback is to special-case `sentiment` to the byte-identical original strings and only vary copy for
  non-sentiment types — call this out explicitly in the PR description either way.
- **Empty/not-ready states are reused, not re-implemented.** The existing `!data.has_model`,
  `!data.is_ready`, `error`, and `loading` branches (`ClassifierAccuracyCard.tsx:147-168`) already produce
  the correct "no category model yet" UX once `classifierType='category'` flows through — per the Contract
  section above, the backend already returns the honest empty shape. Only the interpolated copy inside
  those branches changes (e.g. "accumulate at least N **category** corrections").
- **No new AIReadinessCard change.** The PRD's should-have "readiness readout notes category-correction
  volume" is explicitly **not** in this aspect's `spec.md` scope — leave `AIReadinessCard.tsx` untouched.
- **`AISettingsGeneral` category card copy** is category-specific (not a copy-paste of the sentiment
  block): "Learns from your team's category corrections on pain-point/feature-request items.
  **Off** disables it. **Shadow** observes and scores in the background without changing stored
  categories — recommended until you have accumulated a substantial number of corrections. **Auto** lets
  the trained model override the stored category once it beats the keyword categorizer, and only when its
  predicted label is unambiguous (belongs to exactly one built-in category type)." — last clause is
  disclosure-only copy about the predict-seam's routing rule (`prd.md` requirement 6); this aspect does
  not implement that logic, only describes it honestly.
- **Independent Selects, independent handlers.** `handleClassifierMode` (sentiment) and a new
  `handleCategoryClassifierMode` are separate functions each calling `aiSettingsAPI.update` with only
  their own field, mirroring the existing early-return-on-no-op guard. This directly satisfies the PRD's
  "Independent control" success metric and the spec's acceptance criterion ("changing the category select
  ... does not touch `classifier_mode`").

## Edge cases

- `category_classifier_mode` absent from the `AISettings` object returned by an un-migrated backend (i.e.
  this aspect ships/tests against a mock before `data-and-config` merges) → `settings.category_classifier_mode`
  is `undefined`; the `Select`'s `value` prop would be `undefined`. Guard with
  `value={settings.category_classifier_mode ?? 'off'}` in the JSX (defensive, mirrors the backend's own
  `getattr(..., 'off')` fallback) so the UI never crashes even if wired against a stale mock in CI before
  the dependency lands.
- Two `ClassifierAccuracyCard` instances with different `classifierType` mount in the same page —
  `useCallback`/`useEffect` deps must include `classifierType` (Phase 2) so each instance's `load()` closes
  over its own type and neither card accidentally fetches the other's data (this would only manifest as a
  bug if the deps array were left empty, given each is a separate component instance with its own state —
  still worth the explicit dep for correctness/lint-cleanliness under `react-hooks/exhaustive-deps`).
- Rollback button on the category card, clicked when there's no active category model → backend 404s
  (`classifier_accuracy.py:165-169`); `handleRollback`'s existing catch block already surfaces
  `err?.response?.data?.detail` — verify the 404 detail string ("No active 'category' classifier model to
  roll back...") reads sensibly in the card's error line (no code change needed, just confirm during
  manual verification).
- `CORRECTION_TYPE_LABELS['category']` already exists (`page.tsx:50`) — the "Corrections by Type" summary
  block above the two accuracy cards already lists category corrections correctly; do not duplicate that
  work here.
