### Phase C — GREEN: re-derive windows in the daily pass

**Goal:** `recompute_usage_scores` re-derives the windowed fields against the run's `now`,
without disturbing lifetime aggregates.

**Files**
- `services/worker-service/src/tasks/usage_metrics.py`

**Steps**

1. Extend `_compute_rollup_from_events` (`:39-89`) to also emit `active_days_14d`
   (`cutoff_14d = now - timedelta(days=14)`, same distinct-calendar-day logic as 7d/30d).
   Keep this the **single** source of truth for window derivation — the event path and the
   daily path must both go through it. Do not write a second implementation.
2. Add a new helper, e.g. `_rederive_windows(db, org_id, customer_email, now) -> dict`, that:
   - reads `UsageEvent` rows for `(org_id, customer_email)` **bounded to
     `occurred_at >= now - 30 days`** (the widest window in play), and
   - returns **only** the windowed keys: `active_days_7d/14d/30d`, `login_count_7d/30d`.
   It must **not** return or write `events_total`, `first_seen_at`, `last_active_at`, or
   `distinct_features` — see §6.1.
3. In `recompute_usage_scores` (`:306-355`), for each row: call `_rederive_windows`, assign the
   returned windowed fields onto the row, then compute `compute_usage_score(row, now=now)` from
   the **updated** row.
4. Preserve existing behaviour exactly: still write back only on change, still call
   `_call_update_health` when `abs(new_score - old_score) >= _HEALTH_RECOMPUTE_DELTA` (`:31`,

   `usage_score` alone; a row whose window fields changed but whose banded score did not must
   still be persisted, or the re-derivation silently no-ops.
5. Keep the existing event path (`_do_process_usage_event`) working: it continues to use the
   full-history read for lifetime aggregates. Its `_Proxy` (`:213-222`) needs no change unless

## 5. Edge Cases & Error Handling

- **No `usage_event` rows in window** → all window fields `0` (not NULL, not neutral). A
  customer with a rollup but no recent events is genuinely at zero activity.
- **Rollup row with no events at all** (possible via direct DB state) → windows `0`;
  `compute_usage_score` still returns neutral 50 when its three inputs are all None. Do not
  conflate "zero activity" with "no data".
- **Events with NULL `occurred_at`** — the column is nullable
  (`usage_event.py`); existing window logic already filters on `e.occurred_at and ...`. Preserve
  that guard in the bounded read.
- **Timezone:** the pipeline uses naive `datetime.utcnow()` throughout. Stay consistent; do not
  introduce tz-aware datetimes in this aspect.
- **Multi-tenancy:** the bounded read must filter on **both** `organization_id` and
  `customer_email`. The daily task iterates rows that already carry both — pass them through,
  never query by email alone.
- **Large orgs:** the daily pass now issues a bounded query per rollup row. See §6.2.

## 6. Agent Execution Notes

### 6.1 The bounded-read trap — the single most important instruction

`events_total`, `first_seen_at`, and `distinct_features` are **lifetime** aggregates.
`_compute_rollup_from_events` derives them from *all* events passed to it. If the daily pass
feeds it a 30-day-bounded event list and then writes the full returned dict back, those three
fields are **silently truncated** — `events_total` collapses to the 30-day count,
`first_seen_at` jumps forward, and long-tail features vanish from `distinct_features`.

This corrupts data with no error, and `first_seen_at` is unrecoverable once overwritten.

Two acceptable shapes; pick one and be explicit:
- **(preferred)** `_rederive_windows` returns *only* the five windowed keys, and the caller
  assigns only those; or
- the caller uses the full dict but explicitly deletes/ignores the three lifetime keys.

`test_lifetime_aggregates_survive_recompute` (Phase A) exists precisely to catch this and must
be written **before** Phase C.

### 6.2 Known inherited cost (D2, out of scope)

The daily pass now performs one bounded `usage_event` query per rollup row. `usage_event` has
**no retention** despite the documented 90-day policy, and the event path already re-reads a
customer's entire history on every event. Bounding the daily read to 30 days is a deliberate
mitigation, not a fix. **Do not fix D2 here** — it is explicitly out of scope and belongs on its
own branch. If the per-row query proves slow on a large fixture, note it and continue; a
batched group-by rewrite is a valid follow-up but is not this aspect.

### 6.3 Duplicated modules

`usage_score_service.py` and `segment_service.py` exist in **both** `backend-api` and
`worker-service` with explicit "keep in sync" headers. This aspect should not need to change
either — but if it does, change both and test both. Same for the `CustomerUsage` model, which
has a worker mirror.
