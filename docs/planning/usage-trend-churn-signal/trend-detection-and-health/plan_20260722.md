The highest-risk aspect. The single boundary that matters more than all others:
**do not let usage reach `churn_risk_component` / `churn_probability` / the isotonic model.** AC 10
is the executable form of that boundary and must exist even though it asserts absence.

## Phases

### Phase A — RED: pure trend core (no I/O)

`services/backend-api/tests/test_usage_trend_core.py` + a worker-side mirror test. Cover the two
pure functions before writing them:

- `classify_usage_trend(current_active_days_14d, baseline_active_days_14d, baseline_age_days) -> (state, pct)`
- `apply_trend_penalty(usage_component, trend_state) -> int`

ACs: 3 (small-number guard, incl. baseline 0 / None — no div-by-zero), 4 (state thresholds &
pct values), 5 (bounded penalty property test over full 0-100), 6 (double-count bound, arithmetic
in a comment), 7 (`stable`/`insufficient_history` are free), 14 (distribution sanity: ≥50 flat
customers → ≥95% `stable`, 0 `sharp_decline`).

### Phase B — GREEN: trend core in usage_score_service.py (BOTH copies)

Add the two functions + the named constants (spec's indicative set) to
`backend-api/src/services/usage_score_service.py` AND
`worker-service/src/services/usage_score_service.py` — byte-identical, honouring the
"NO magic numbers" convention. Tie-break at 12-vs-16 (AC 2) must be deterministic and stated in a
comment (e.g. "on equal distance, prefer the OLDER snapshot"). AC 15: a test asserts the two
copies stay textually in sync.

### Phase C — Schema: trend columns + migration

`customer_usage` gains `usage_trend_state` (String, NOT NULL, server_default
`'insufficient_history'`) and `usage_trend_pct` (Float, nullable). Worker mirror updated too.
Migration additive; round-trip via SQLite proxy; single head preserved. (Do NOT re-add
`active_days_14d`.)

### Phase D — Wiring in the daily pass

In `recompute_usage_scores`, AFTER re-windowing AND after today's snapshot write:
- Look up the nearest in-band snapshot (12-16d, nearest to 14) from `customer_usage_history` per
  customer — **prefer one batched query** over per-row (the `(org,email,snapshot_date)` index from
  aspect 2 must exist). State which in a comment.
- `classify_usage_trend(...)`, persist `usage_trend_state`/`usage_trend_pct`.
- **Extend the health-refresh trigger:** call `_call_update_health` when the score moved ≥ 2 pts
  **OR** the trend state changed (AC 12) — else the penalty never reaches health until an unrelated
  event.
- Idempotent with a fixed `now` (AC 13).
- Do NOT mutate stored `usage_score` (AC 11); do NOT touch churn (AC 10).

### Phase E — Health penalty (usage component only)

In `_compute_usage_component` (`health_score_service.py:65-105`): extend the SAVEPOINT SELECT to
also read `usage_trend_state`, return `apply_trend_penalty(usage_score, trend_state)` clamped
`[0,100]`. Penalty applies **only when a real row was read** — the neutral `50` fallback stays
unpenalised (AC 8). AC 9: byte-stability at weight 0 across all four states, extending aspect 1's
characterization fixture.

### Phase F — API surface (AC 16)

`usage_trend_state: Optional[str]` + `usage_trend_pct: Optional[float]` on
`CustomerProfileResponse` (`customers.py:106-145`), filled by a `_read_usage_trend_fields(record, db)`
helper in `customer_profile_serializer.py` mirroring `_read_crm_fields` — never raises, all-None on
failure, endpoint never 500s when the rollup table is absent. **Freeze this contract early** so
aspect 4 can build against it.

## The boundary — read before touching any worker task

`churn_probability` = `_predict_with_interval(health.churn_risk_component, model)`
(`probability_updater.py:75`). `churn_risk_component` is feedback-only
(`health_score_service.py:191`). Usage must never feed it. Concretely:
- Do NOT edit `churn_calibration.py` or `churn_calibrator.py`.
- Edit `probability_updater.py` ONLY if the trend-triggered health refresh strictly requires it,
  and NEVER in a way that changes the input to `_predict_with_interval`.
- AC 10 asserts a `stable → sharp_decline` transition with no new feedback leaves
  `churn_risk_component`, `churn_probability`, `_low/_high`, `calibration_model_id`,
  `time_to_churn_bucket` all unchanged.
