### Phase C — Daily batched write (GREEN for snapshot tests)

In `recompute_usage_scores` (`usage_metrics.py:307-355`), AFTER the existing per-row loop:

- Build one `CustomerUsageHistory` mapping per scanned row for today's UTC date, from the row's
  post-recompute values (AC 2 — after the score update).
- **Same-day idempotency, dialect-neutral:** one query for today's existing
  `(org,email)` snapshot keys, OR one bounded `DELETE` of today's rows for the scanned set, before
  a single batched `add_all`/`bulk_save_objects`. NO `ON CONFLICT`. (AC 3)
- **Commit unconditionally.** The existing commit is `if updated:` — a steady population would
  otherwise never snapshot and the trend would never warm up. Resolve per the spec: make the
  snapshot write + commit run on every execution.
- **Partial-failure = two transactions.** Wrap the snapshot write so a failure is caught, logged
  with `logger.error` (NOT swallowed silently — D3 is the cautionary tale), and does NOT roll back
  the already-committed score updates. State the choice in a comment.
- Batch/chunk (e.g. 1000/flush) so statement count does not scale per-customer (AC 7).

### Phase D — Prune task + beat (GREEN for prune tests)

- `purge_old_usage_history` in `usage_metrics.py`: `@shared_task(name="...")`, module-level
  `USAGE_HISTORY_RETENTION_DAYS: int = 180`, cutoff `date.today() - timedelta(days=180)` (UTC),
  single `.delete(synchronize_session=False)`, `db.commit()`,
  `{"status": "complete", "deleted": N}`. (AC 9)
- Register in `celery_app.py` weekly purge block (`:179-215`) at a free slot — NOT
  02:15/02:30/03:00/03:15/03:30/03:45/04:00/04:15. Suggest Sun 02:45. One-line comment matching
  neighbours. (AC 10)
