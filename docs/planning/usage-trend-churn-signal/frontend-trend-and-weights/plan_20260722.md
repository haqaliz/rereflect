## Part B — D4 weights fix (do first, split-able)

### The one real design decision
`HealthWeightsEditor` loads weights via GET; on failure it currently swallows the error into
`console.error` and leaves `DEFAULT_WEIGHTS` (crm:0) in state
(`HealthWeightsEditor.tsx:41-49`). That is exactly the zeroing shape we're removing. **Decision:
on a failed load, block Save and surface the failure** (disabled Save + visible error), rather
than letting a save proceed from a defaulted-crm state. The safer of the two options the spec
offers; it cannot silently zero crm.

### Steps (TDD — extend `__tests__/settings/HealthWeightsEditor.test.tsx` first)
1. `HealthWeights` (`lib/api/categories.ts:24-29`) gains `usage` and `crm`; `updateHealthWeights`
   (`:65-68`) sends the six-key object. (AC 8, 9)
2. `HealthWeightsEditor`: add `usage` to `DEFAULT_WEIGHTS` (:10-15), `WEIGHT_LABELS` (:17-22),
   `WEIGHT_DESCRIPTIONS` (:24-29) — the `Object.keys(weights)` render loop (:93) picks it up.
   `crm` is carried in state and in the payload but is NOT rendered as an input (AC 9, out-of-scope
   to surface it). (AC 10, 14)
3. Extend the sum computation (:51-53) to all SIX keys so the indicator matches the backend's
   `validate_sum_is_100`. (AC 11, 12)
4. "Reset to Default" → six-key object `churn35/sentiment25/resolution25/frequency15/usage0/crm0`
   summing to 100. (AC 13)
5. Failed GET → block Save + surface error (the decision above).
6. Correct stale copy: `app/(dashboard)/settings/usage-events/page.tsx:208-216` →
   Settings → AI → Health Score Weights (`/settings/ai`); comment at `ComponentProgressBars.tsx:32`
   and `lib/api/categories.ts:34`. (AC 15)
7. Sum-error copy may need to name the hidden CRM allocation (spec risk) so "currently 90" isn't
   confusing when the 5 visible fields look like they sum right.

## Part A — Trend on the Usage Activity card (after aspect 3)

### Steps (TDD — extend `__tests__/customers/CustomerProfilePage.test.tsx`)
1. `UsageRollup` (`lib/api/customers.ts:155-169`) gains `usage_trend_state?:
   'insufficient_history'|'stable'|'declining'|'sharp_decline'` and `usage_trend_pct?: number|null`
   — optional, tolerating older backends / pre-migration rows (AC 6).
2. Render trend in the inline `UsageActivityCard` (`page.tsx:476-543`): labelled state + direction
   affordance + signed pct when non-null. `sharp_decline` visually stronger than `declining`
   (distinct label/test id). (AC 1,2,3)
3. `insufficient_history` → explicit "Warming up" state naming the ~2-week warm-up (NOT a precise
   countdown — no snapshot history client-side), no percentage. (AC 4)
4. Precedence: no-usage empty state (`hasUsage===false`, :498) WINS; warm-up only when the customer
   has usage but no computable trend. (AC 5)
5. Read trend from the card's own `days=30` query, so switching the embedded timeline to 60/90d
   does NOT change the trend. (AC 7)
6. Theme CSS variables for state colours — no hardcoded colours.
7. Do NOT invent `stable` from a missing state; render only what the API returns. (AC 6)
