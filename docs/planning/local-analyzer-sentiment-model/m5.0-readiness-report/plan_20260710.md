## 3. Phase 1 — Backend: RED tests (`tests/test_ai_readiness.py`)

- **Goal:** a fully failing (RED) test file that pins every acceptance criterion from `spec.md` against
  the not-yet-built endpoint, plus the trivial threshold-constants module the tests need to import exact
  values from (not the endpoint logic itself — safe to create ahead of the route).
- **Prereqs:** none (first phase).
- **Files:**
  - create `src/config/readiness_thresholds.py` (constants only — see Steps 1).
  - create `tests/test_ai_readiness.py`.
- **Steps:**
  1. **Constants first (not RED, just data):** `src/config/readiness_thresholds.py`:
     ```python
     # Activation thresholds for M5.2 (per-org corrections flywheel) and M5.3 (per-org churn ML).
     # See AI-TRACKING.md:313-317 (M5.0) and docs/planning/local-analyzer-sentiment-model/m5.0-readiness-report/spec.md.
     CHURN_LABEL_TARGET = 500          # AI-TRACKING.md:317 — M5.3 exit criterion, verbatim
     CORRECTION_VOLUME_TARGET = 200    # v1 proposal for M5.2 — unvalidated, tune after a real training run
     ```
  2. Test-helper scaffolding (mirror `test_churn_accuracy_api.py`): `_make_org(db, name)`,
     `_make_user(db, org, role="admin")`, `_headers(user)`.
  3. Seed helpers local to this test module: `_make_feedback(db, org, n=1)`,
     `_make_correction(db, org, correction_type, user=None)`, `_make_churn_event(db, org, *, reason_code,
     source, recovered=False, email=None)` — thin wrappers constructing `FeedbackItem`/`AICorrection`/
     `CustomerChurnEvent` rows directly (no service-layer calls needed — this is a read endpoint, seed data
     directly).
  4. **AC1** `test_empty_org_returns_zeros` — new org, no seed data, `GET` → 200, every count/dict is
     zero/empty, both `*_ready` flags `False`, `correction_volume_target == CORRECTION_VOLUME_TARGET`,
     `churn_label_target == CHURN_LABEL_TARGET` (imported from the constants module, not hardcoded numbers,
     so the test doesn't silently drift from the source of truth).
  5. **AC2** `test_mixed_data_returns_exact_counts` — seed 7 feedback items; 3 `sentiment` + 2 `category` +
     1 `churn_risk` corrections; 4 churn events: 2 `price`/`manual`, 1 `competitor`/`csv_import`, 1

  6. **AC3** `test_cross_org_isolation` — org A and org B each seeded with distinct, non-overlapping data;

  7. **AC4** `test_correction_volume_ready_boundary` — seed exactly `CORRECTION_VOLUME_TARGET - 1`

  8. **AC5** `test_churn_labels_ready_boundary` — same boundary shape for `CHURN_LABEL_TARGET` (use a loop

  9. **AC6** `test_recovered_churn_event_still_counts` — one churn event with `recovered_at` set: appears in

## 4. Phase 2 — Backend: GREEN (schema, route, registration)

- **Goal:** make every Phase 1 test pass without weakening any assertion.
- **Prereqs:** Phase 1 (tests exist and are RED).
- **Files:** create `src/schemas/ai_readiness.py`, create `src/api/routes/ai_readiness.py`; modify
  `src/api/main.py`.
- **Steps:**
  1. `src/schemas/ai_readiness.py` — `AIReadinessResponse` exactly as specified in `spec.md` (flat,
     `Dict[str, int]` for both breakdown fields, `model_config`/`Config` with `from_attributes = True`
     even though we build it from a plain dict, for consistency with sibling schemas).
  2. `src/api/routes/ai_readiness.py`:
     ```python
     analytics_router = APIRouter(prefix="/api/v1/analytics", tags=["ai-readiness"])
     router = analytics_router  # match churn_accuracy.py's `router = analytics_router` export convention
     ```
     Three private helpers, each a single filtered query:
     - `_feedback_volume(org_id, db) -> int` — `db.query(func.count(FeedbackItem.id)).filter(FeedbackItem.organization_id == org_id).scalar() or 0`.
     - `_correction_counts(org_id, db) -> tuple[int, dict[str, int]]` — total via `func.count`, breakdown via
       `db.query(AICorrection.correction_type, func.count(AICorrection.id)).filter(...).group_by(...)`
       (identical shape to `ai_corrections.py:165-172`).
     - `_churn_label_counts(org_id, db) -> dict` — total, recovered count
       (`filter(CustomerChurnEvent.recovered_at.isnot(None))`), by-reason and by-source group-bys — all
       filtered by `organization_id == org_id`.
     Endpoint:
     ```python
     @analytics_router.get("/ai-readiness", response_model=AIReadinessResponse)
     def get_ai_readiness(
         current_org: Organization = Depends(get_current_org),
         db: Session = Depends(get_db),
     ) -> AIReadinessResponse:
         """Per-org AI training-readiness snapshot (M5.0, no ML).

         Thresholds are v1 planning targets, not validated ML requirements — see
         docs/planning/local-analyzer-sentiment-model/m5.0-readiness-report/spec.md.
         """
         org_id = current_org.id
         feedback_volume = _feedback_volume(org_id, db)
         corrections_total, corrections_by_type = _correction_counts(org_id, db)
         churn = _churn_label_counts(org_id, db)
         return AIReadinessResponse(
             organization_id=org_id,
             generated_at=datetime.utcnow(),
             feedback_volume=feedback_volume,
             corrections_total=corrections_total,
             corrections_by_type=corrections_by_type,
             churn_labels_total=churn["total"],
             churn_labels_recovered=churn["recovered"],
             churn_labels_by_reason=churn["by_reason"],
             churn_labels_by_source=churn["by_source"],
             correction_volume_target=CORRECTION_VOLUME_TARGET,
             churn_label_target=CHURN_LABEL_TARGET,
             correction_volume_ready=corrections_total >= CORRECTION_VOLUME_TARGET,
             churn_labels_ready=churn["total"] >= CHURN_LABEL_TARGET,
         )
     ```
  3. `src/api/main.py` — add `from src.api.routes import ai_readiness as ai_readiness_router  # noqa: E402 — M5.0 readiness report`
     next to the `churn_accuracy_router` import; add
     `app.include_router(ai_readiness_router.router)  # AI training readiness (M5.0)` right after the
     existing `app.include_router(churn_accuracy_router.system_router)` line.
  4. Run tests, fix until green — iterate on the group-by dict-comprehension shape
     (`{ct: cnt for ct, cnt in rows}`) if empty-org group-bys return an empty list vs `None` mismatch.
- **Validation:** `cd services/backend-api && pytest tests/test_ai_readiness.py -v` — all green. Then run
  neighbors sharing the `main.py` edit to confirm no regression:
  `pytest tests/test_churn_accuracy_api.py tests/test_ai_corrections.py -q`.
- **Commit:** `feat(ai-readiness): GET /api/v1/analytics/ai-readiness — feedback volume, correction counts by type, churn-label counts + distribution, threshold ready flags (M5.0)`

## 7. Testing strategy, edge cases & agent execution notes

**Testing strategy**
- Backend: `cd services/backend-api && pytest tests/test_ai_readiness.py -v` is the primary command. Before
  considering Phase 2 done, also run `pytest tests/test_churn_accuracy_api.py tests/test_ai_corrections.py -q`
  (both share the `main.py` edit / route-registration surface and the `ai_corrections`/`churn` tables this
  aspect reads). **Do not run the bare full suite** — memory `rereflect-repo-test-gotchas` notes a
  pre-existing `test_report_ws.py` segfault that makes full-suite runs unreliable; scope test runs to the
  files touched.
- Frontend: `cd services/frontend-web && npm run test -- AIReadinessCard` (vitest) and `npm run lint`. No
  `npm run build` required for this aspect alone, but harmless to run if touching shared page files makes
  you want extra confidence.
- No migration to test (`alembic upgrade head` unaffected — confirm by running it once locally to be sure
  the repo's existing multi-heads state, if unresolved, isn't accidentally attributed to this work; this
  aspect adds zero migrations so it cannot be the cause).

**Edge cases & error handling**
- Empty org (zero rows in all three tables) → 200 with zeros, not 404/500 — an org with no data yet is the
  most common real-world case for a brand-new self-host install and must render cleanly, not error.
- `corrections_by_type`/`churn_labels_by_reason`/`churn_labels_by_source` must be **plain dicts with only
  observed keys** — never pre-populate with a fixed key set (e.g. don't default-zero `{"sentiment": 0,
  "category": 0, ...}` for keys that don't appear in the data) — this is the forward-compatible design
  decision from spec.md; a test should assert an org with only `sentiment` corrections does **not** get a
  synthetic `category: 0` key injected.
- Recovered churn events (`recovered_at` set) still count toward `churn_labels_total` — a customer who
  churned and came back is still a valid label for "did this customer churn at some point," which is the
  training signal M5.3 cares about; only `churn_labels_recovered` distinguishes them.
- No plan/feature gate, no admin-only gate — any authenticated user in the org can view this (matches
  "View dashboard & analytics" = all roles in the RBAC matrix). Do not accidentally wrap this in
  `require_admin_or_owner` or `require_feature(...)` — Phase 1's AC7 `test_member_role_can_view` exists
  specifically to catch that regression.
- Threshold boundary is **inclusive** (`>=`), matching the "at 500 you're ready" reading of "gate ... M5.3
  (~500 churn labels)" rather than "over 500."

**Agent execution notes**
- Checkpoint after Phase 2 (backend GREEN) before starting Phase 3 — the frontend types in
  `lib/api/ai-readiness.ts` are copied by hand from `AIReadinessResponse`; keep them byte-identical field
  names (snake_case, not camelCase — matches every other API client in this repo) to avoid a silent shape
  mismatch.
- If the sibling `eval-harness-and-card` aspect lands first and has already added an `'accuracy'`-tab widget
  or otherwise touched `VALID_TABS`, rebase this aspect's `VALID_TABS`/`TabsList`/`TabsContent` diff onto
  theirs — both diffs are additive (new array entries / new tab blocks), so a conflict here should be a
  trivial rebase, not a redesign.
- Do not use this aspect's endpoint or thresholds as a gate for anything at runtime — it is purely
  informational (an admin report), never a feature flag or a blocker for any other endpoint in this slice.
