## 4. Implementation phases (strict TDD)

### Phase 1 ‚Äî Multiclass metrics core (pure, no CSV, no providers)
- **Goal:** A tested, reusable multiclass metrics function built directly on the existing binary
  `tp/fp/fn/tn ‚Üí precision/recall/f1/accuracy` pattern, provably correct against hand-built
  confusion matrices before anything else depends on it.
- **Prereqs:** none ‚Äî pure function, no dependency on `sentiment-provider-core` or
  `model-packaging`.
- **Files:** `scripts/eval_sentiment.py` (new ‚Äî metrics section only this phase),
  `tests/test_eval_sentiment_script.py` (new ‚Äî `TestMulticlassMetrics` class only this phase).
- **Steps:**
  1. **RED:** `tests/test_eval_sentiment_script.py::TestMulticlassMetrics` ‚Äî write tests against
     a hand-built 3x3 confusion matrix (rows = true label, cols = predicted label) with known
     counts, e.g.:
     ```
     confusion = {
       "positive": {"positive": 8, "neutral": 1, "negative": 1},
       "neutral":  {"positive": 2, "neutral": 6, "negative": 2},
       "negative": {"positive": 0, "neutral": 1, "negative": 9},
     }
     ```
     Assert exact expected precision/recall/F1 for `"positive"` (derive tp/fp/fn/tn by hand in
     the test docstring ‚Äî mirrors `TestMetricsComputation`'s known-input-known-output style in
     `test_backtest_backfill_scripts.py`), assert macro-F1 = mean of the three per-class F1s,
     assert `accuracy` = trace / total. Also test the degenerate all-zero-confusion case returns
     zeros (not a `ZeroDivisionError`), and a single-class-only confusion matrix.
  2. **GREEN:** implement in `eval_sentiment.py`:
     - `_safe_precision_recall_f1_accuracy(tp, fp, fn, tn)` ‚Äî **copied** from
       `admin_backtest.py:56` (same math, same signature; not imported cross-module, matching
       the existing precedent that `backtest_churn.py` and `admin_backtest.py` each define their
       own copy rather than sharing one).
     - `confusion_to_binary_counts(confusion: dict, label: str, labels: list[str]) -> tuple[int,int,int,int]`
       ‚Äî one-vs-rest slice: `tp = confusion[label][label]`; `fp = sum(confusion[other][label] for
       other in labels if other != label)`; `fn = sum(confusion[label][other] for other in labels
       if other != label)`; `tn = total - tp - fp - fn`.
     - `compute_multiclass_metrics(confusion: dict, labels: list[str]) -> dict` ‚Äî for each label,
       call `confusion_to_binary_counts` then `_safe_precision_recall_f1_accuracy`; assemble
       `per_class: {label: {precision, recall, f1, support}}` (`support` = row sum = true count
       for that label); `macro_precision/macro_recall/macro_f1` = unweighted mean across labels;
       top-level `accuracy` = `sum(confusion[l][l] for l in labels) / total`.
  3. **REFACTOR:** extract the 3-label constant `SENTIMENT_LABELS = ["positive", "neutral",
     "negative"]` as a module constant used everywhere below.
- **Validation:** `cd services/backend-api && pytest tests/test_eval_sentiment_script.py -v -k MulticlassMetrics`.
- **Commit:** `test(sentiment-eval): multiclass precision/recall/F1/macro metrics core`

### Phase 2 ‚Äî CSV loading + fixtures (tiny fixture only)
- **Goal:** Load and validate the `text,label` CSV schema; prove it against a tiny, fully-known

- **Prereqs:** Phase 1 (uses `SENTIMENT_LABELS`).
- **Files:** `tests/fixtures/sentiment_eval/tiny_fixture.csv` (new, ~8-10 rows: 3
  positive/3 neutral/2-3 negative, unambiguous text like "This is fantastic, love it!" /

  (`load_eval_csv`), `tests/test_eval_sentiment_script.py` (`TestLoadEvalCsv` class).
- **Steps:**
  1. **RED:** `TestLoadEvalCsv` ‚Äî `load_eval_csv(tiny_fixture_path)` returns a list of
     `(text, label)` tuples with the exact known length and label distribution of the fixture;
     a malformed CSV (unknown label value, e.g. `"positve"` typo) raises a clear `ValueError`
     naming the bad row; a CSV missing the `text` or `label` header raises a clear error; an
     empty-text row raises.
  2. **GREEN:** `load_eval_csv(path: str) -> list[tuple[str, str]]` using `csv.DictReader`;
     validate `label in SENTIMENT_LABELS` and non-empty `text` per row, collecting all errors

  3. **REFACTOR:** none expected.
- **Validation:** `pytest tests/test_eval_sentiment_script.py -v -k LoadEvalCsv`.
- **Commit:** `feat(sentiment-eval): CSV loader + tiny test fixture`

### Phase 3 ‚Äî `run_provider` / `run_eval_set` (script core, provider-injectable)
- **Goal:** Wire the metrics core (Phase 1) and CSV loader (Phase 2) into the full per-provider,
  per-set eval flow ‚Äî with the provider **injected as a callable/object**, so this phase's tests
  never need a real transformer (or even real VADER) to prove the wiring is correct.
- **Prereqs:** Phase 1, Phase 2. **Does not require `sentiment-provider-core` to be merged** ‚Äî
  tests use a fake provider object; real-VADER wiring is proven in Phase 3b once
  `sentiment-provider-core` lands (or against the current `SentimentAnalyzer.analyze()` output
  shape if it lands first, since VADER output is byte-stable per that aspect's must-have #2).
- **Files:** `scripts/eval_sentiment.py` (`run_provider`, `run_eval_set`, `EvalSetResult`/
  `ProviderEvalResult` as plain dataclasses ‚Äî no Pydantic dependency in the script itself, keep
  it a standalone CLI tool), `tests/test_eval_sentiment_script.py` (`TestRunProvider`,
  `TestRunEvalSet` classes).
- **Steps:**
  1. **RED (`TestRunProvider`):** construct a fake provider with `.predict(text) -> str` (or
     `.analyze(text) -> {"label": ...}` ‚Äî confirm exact shape against
     `sentiment-provider-core`'s spec once available; default assumption documented in step 2)
     that returns a fixed/scripted label per input; call `run_provider(fake_provider, rows)`
     where `rows` is the tiny fixture loaded via Phase 2; assert the returned confusion matrix
     exactly matches hand-counted true/predicted pairs.
  2. **GREEN:** `run_provider(provider, rows: list[tuple[str,str]]) -> ProviderEvalResult`:
     - Adapter note (write explicitly in code comment): calls
       `provider.analyze(text)["label"]` to match `SentimentAnalyzer.analyze()`'s existing 7-key
       contract (`sentiment.py:29`) ‚Äî assumes `sentiment-provider-core` exposes each concrete
       provider with the **same** `.analyze(text) -> dict` shape (or the script wraps
       `SentimentAnalyzer(provider=...)`. If the landed interface differs (e.g. bare
       `.predict(text) -> str`), this is the **one function** that needs a signature tweak (R1 in
       spec.md) ‚Äî isolate the call behind a private `_predict_label(provider, text)` helper so
       the blast radius of an interface mismatch is one function, not the whole script.
     - Builds the confusion matrix as `{true_label: {pred_label: count}}` initialized to 0 for
       all label pairs (so `compute_multiclass_metrics` never KeyErrors on an unseen pair).
     - Calls `compute_multiclass_metrics(confusion, SENTIMENT_LABELS)`, wraps into
       `ProviderEvalResult(provider=..., n=len(rows), **metrics, confusion_matrix=confusion)`.
  3. **RED (`TestRunEvalSet`):** `run_eval_set("in_domain", tiny_fixture_path, vader_provider=fake_a,
     transformer_provider=fake_b)` returns an `EvalSetResult` with both `.vader` and
     `.transformer` populated, `macro_f1_delta = transformer.macro_f1 - vader.macro_f1`
     (assert exact value for scripted fakes), and `meets_target` computed **only** when
     `set_name == "in_domain"` (`delta >= 0.05`), else `None` for the public set.
  4. **RED (graceful-skip):** `run_eval_set(..., transformer_provider=None)` returns
     `EvalSetResult(transformer=None, macro_f1_delta=None, meets_target=None)` without raising ‚Äî
     this is the soft-dependency degrade path (spec.md's "graceful skip" requirement) exercised
     **now**, independent of `model-packaging` landing.
  5. **GREEN:** implement `run_eval_set` per the above; `transformer_provider=None` short-circuits
     before calling `run_provider` for the transformer side.
  6. **REFACTOR:** confirm dataclass field names match what Phase 6's Pydantic schema will need
     1:1 (avoids a translation layer later).
- **Validation:** `pytest tests/test_eval_sentiment_script.py -v`.
- **Commit:** `feat(sentiment-eval): run_provider/run_eval_set core with graceful transformer-skip`

## 6. Edge cases & error handling

- **Artifact missing:** endpoint returns `has_results: false`, 200 ‚Äî never 404/500. Card shows an
  honest "not run yet" state, not a spinner-forever or an error.
- **Artifact present but malformed/truncated JSON:** same degrade-to-`has_results:false` path
  (`json.JSONDecodeError` caught alongside `FileNotFoundError`).
- **Transformer half absent from a given run** (`model-packaging` not landed yet, or the model
  failed to load during the eval run): `transformer: null` for that set; `macro_f1_delta`/
  `meets_target` are `null`; card renders VADER-only with a "transformer not evaluated" note ‚Äî
  this must render cleanly for **both** `public` and `in_domain` independently (they can be in
  different states if, say, only one set was re-run).
- **`meets_target: false` (the model loses in-domain):** must render **identically visibly** to
  the `true` case except for the badge text/color ‚Äî this is the core disclosure requirement; the
  frontend test (Phase 7 step 3) explicitly locks this so a future "let's just hide the bad
  number" edit would fail CI.
- **Script run with a CSV that fails `load_eval_csv` validation:** `main()` propagates the
  `ValueError` and exits non-zero with a clear message (this is a developer-facing CLI failure,
  not a runtime API path ‚Äî acceptable to be loud here, unlike the endpoint).
- **Empty confusion-matrix cell combinations** (a class never predicted, or never present in
  gold): `compute_multiclass_metrics` must not `KeyError` ‚Äî confusion matrix is always
  initialized with all `SENTIMENT_LABELS √ó SENTIMENT_LABELS` pairs at 0 (Phase 3 step 2).
- **Multi-tenancy:** not applicable ‚Äî this is a single, org-independent, global disclosure
  artifact (no `organization_id` scoping anywhere in this aspect, consistent with the eval
  harness being explicitly offline/reproducible per must-have #6, not a per-org metric).
