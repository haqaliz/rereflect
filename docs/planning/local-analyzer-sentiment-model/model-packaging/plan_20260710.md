# Implementation Plan — model-packaging

**Date:** 2026-07-10 · **Slug:** `local-analyzer-sentiment-model` · **Aspect:** `model-packaging`
**Spec:** `./spec.md` · **PRD:** `../prd.md`
**Branch:** `feat/local-analyzer-sentiment-model` · **Validation style:** build-oriented (this
aspect ships no Python application logic — `docker build` + smoke-test scripts are the tests).

> Infrastructure aspect. Soft-depends on `sentiment-provider-core` for the pinned model revision
> SHA only (placeholder used until then); otherwise fully independent and safe to build in
> parallel. No provider code, no resolver code, no migration.

## 1. Scope & impact

- **Services touched:** `services/worker-service` (requirements.txt, Dockerfile),
  `services/backend-api` (requirements.txt, Dockerfile), root `docker-compose.yml` +
  `docker-compose.prod.yml` (backend build context + new build arg), `docs/SELF_HOSTING.md`.
- **Not touched:** `services/analysis-engine` (no Dockerfile/compose service — nothing to bake),
  any Python source under `src/analyzer/`, `src/tasks/analysis.py`, `src/api/routes/feedback.py`,
  `OrgAIConfig`, any migration, any frontend file, `AI-TRACKING.md`, `README`.
- **Net effect for an operator who changes nothing:** identical behavior. Default builds make no
  network call to Hugging Face and bake no weights; the only observable change is a modest,
  unconditional image-size increase from the new `torch`/`transformers` **package** deps (not
  weights) landing in the worker and backend images, and the backend gaining the ability to
  actually import `analyzer` at all (previously broken — see spec R2).

## 3. Design decision — bake vs pull-on-first-enable

**Decision: don't bake by default; gate the pre-bake behind a build ARG (`BAKE_SENTIMENT_MODEL`,
default `false`), document the pre-bake path for air-gap.**

Rationale:
- The PRD explicitly leans this way (Risks: "Lean: don't bake by default to keep the base image
  lean; document both") and must-have #9 already requires a **runtime pull-and-cache-on-first-use**
  fallback via `from_pretrained` (no `local_files_only` forced) for the common case — a self-hoster
  with internet access who flips the setting on gets the model downloaded and cached in the running
  container's `HF_HOME` on first request, no rebuild needed. That path needs no packaging work
  beyond `HF_HOME` being a writable, persistent-enough path (documented caveat: an ephemeral
  container filesystem re-downloads on every restart unless `HF_HOME` is a mounted volume — noted
  in `SELF_HOSTING.md`, not solved here since compose volume wiring for a stateless cache is a
  judgment call left to the operator, same posture as `postgres_data`/`redis_data`).
- Air-gapped operators (no runtime network at all) are a **minority but real** persona per the PRD
  ("Air-gapped operator" scenario) and cannot rely on first-use pull. For them, the ARG gate is the
  only way to get weights into the image without forcing the cost on everyone else.
- A build ARG (not a separate Dockerfile/image) keeps one image definition, one CI matrix cell to
  reason about, and reuses the exact precedent already in the file (`nltk.download`) — lowest-risk
  option that a reviewer can diff against something that already exists and works.
- Rejected alternative: bake by default. Rejected because must-have #5's own image-size framing and
  the PRD's non-goal ("no absolute accuracy number as marketing") both push toward *not* taxing the
  100% of self-hosters who never touch AI settings with a ~450MB+ unconditional download during
  every `docker build`.
- Rejected alternative: separate `Dockerfile.sentiment` / second image. Rejected — doubles the
  images to build/publish/document for a single conditional `RUN` step; not proportionate.

## 4. Implementation phases

### Phase 1 — Pin deps in worker + backend, CPU-only wheel
- **Goal:** `torch`/`transformers` importable in both runtime images, without pulling the
  multi-GB CUDA wheel.
- **Prereqs:** none.
- **Files:** `services/worker-service/requirements.txt`, `services/backend-api/requirements.txt`,
  `services/worker-service/Dockerfile`, `services/backend-api/Dockerfile`.
- **Steps:**
  1. Append to `worker-service/requirements.txt` (own section, matching the file's existing
     `# Analysis engine dependencies (shared)` comment style):
     ```
     # Transformer sentiment provider (opt-in; mandatory dep once any org enables it — M5.1)
     transformers==4.46.0
     torch==2.5.0
     ```
  2. Append the same two lines (own section) plus `vaderSentiment==3.3.2` to
     `backend-api/requirements.txt` — VADER is currently **absent** there too (spec R2); without it
     the sync path can't run any sentiment provider, default or transformer.
  3. In both Dockerfiles, change the `pip install` line for the heavy deps to use the CPU-only
     PyTorch index, keeping the existing single `pip install -r requirements.txt` call to
     preserve Docker layer caching, e.g.:
     ```dockerfile
     RUN pip install --no-cache-dir --extra-index-url https://download.pytorch.org/whl/cpu \
         -r requirements.txt
     ```
     (`--extra-index-url` still resolves non-torch packages from PyPI normally; pip prefers the
     more specific CPU wheel when both indexes offer `torch==2.5.0`.)
  4. Do **not** touch `analysis-engine/requirements.txt` — it is already correct and unaffected.
- **Validation:**
  - `docker build -f services/worker-service/Dockerfile -t rrf-worker-test ./services` succeeds.
  - `docker build -f services/backend-api/Dockerfile -t rrf-backend-test ./services/backend-api`
    succeeds (still on the *old* build context for this phase — context move is Phase 2).
  - `docker run --rm rrf-worker-test python -c "import torch, transformers; print(torch.__version__, transformers.__version__)"`
    prints `2.5.0 4.46.0` with no CUDA-related download messages in the build log.
  - `docker images rrf-worker-test --format '{{.Size}}'` recorded as the "deps-only" baseline size
    for later comparison against the baked build (Phase 3 AC4).
- **Commit:** `build(worker,backend): add pinned CPU-only torch+transformers deps for opt-in sentiment transformer`

### Phase 2 — Fix backend image so `analyzer` is importable at all
- **Goal:** The backend's existing (already-broken) relative-path import of `analysis-engine`
  resolves inside the built image, for VADER today and the transformer once
  `per-org-resolution` wires it in.
- **Prereqs:** Phase 1 (requirements.txt already has the deps that will need `analyzer` to import
  cleanly).
- **Files:** `services/backend-api/Dockerfile`, `docker-compose.yml`, `docker-compose.prod.yml`.
- **Steps:**
  1. Rewrite `backend-api/Dockerfile` to mirror the worker's context-relative COPY shape:
     ```dockerfile
     WORKDIR /app
     ...
     COPY backend-api/requirements.txt .
     RUN pip install --no-cache-dir --extra-index-url https://download.pytorch.org/whl/cpu \
         -r requirements.txt
     COPY backend-api/ .
     # Copy analysis engine so backend-api/src/api/routes/feedback.py's sys.path lookup
     # (dirname(__file__) + "../../../analysis-engine") resolves to a real 'analyzer' package.
     COPY analysis-engine/src/analyzer ./analysis-engine/analyzer
     ```
     Note the `COPY backend-api/ .` (not `COPY . .`) — build context is moving to `./services` in
     step 2, so every existing `COPY` line in this Dockerfile needs the `backend-api/` prefix, not
     just the new one.
  2. In `docker-compose.yml` and `docker-compose.prod.yml`, change the `backend` service's build
     block from
     ```yaml
     build:
       context: ./services/backend-api
       dockerfile: Dockerfile
     ```
     to
     ```yaml
     build:
       context: ./services
       dockerfile: backend-api/Dockerfile
     ```
     (exactly the worker service's existing shape, two lines above it in both files).
  3. Double check no other relative path in `backend-api/Dockerfile` assumed the old context (the
     `CMD sh -c "python -m alembic upgrade head && ..."` line is unaffected — it runs from
     `WORKDIR /app`, which still contains `alembic.ini` etc. after the `COPY backend-api/ .` step).
- **Validation:**
  - `docker build -f services/backend-api/Dockerfile -t rrf-backend-test ./services` succeeds
    (context is now `./services`).
  - `docker run --rm rrf-backend-test python -c "import sys; sys.path.insert(0, 'analysis-engine'); from analyzer.sentiment import SentimentAnalyzer; print(SentimentAnalyzer().analyze('great job'))"`
    returns the 7-key dict with no `ModuleNotFoundError` (this is AC3 from the spec — proves the
    pre-existing bug is fixed).
  - `docker compose -f docker-compose.yml build backend worker` and `docker compose -f
    docker-compose.prod.yml build backend worker` both succeed unchanged otherwise (AC5).
  - Existing backend tests unaffected: `cd services/backend-api && pytest tests/ -v` still green
    locally (this phase changes packaging/Docker only, not application code or `requirements.txt`
    semantics beyond additive lines).
- **Commit:** `fix(backend-api): copy analysis-engine into image; fix broken build context so sentiment analyzer import resolves`

### Phase 3 — `HF_HOME` + gated pre-bake ARG
- **Goal:** A cache directory always exists; weights land in it only when explicitly requested at
  build time.
- **Prereqs:** Phase 1 (deps present to run the pre-bake command against).
- **Files:** `services/worker-service/Dockerfile`, `services/backend-api/Dockerfile`.
- **Steps:**
  1. Add near the top of both Dockerfiles (after `WORKDIR /app`, before the heavy `pip install`):
     ```dockerfile
     ENV HF_HOME=/app/models
     ARG BAKE_SENTIMENT_MODEL=false
     ```
  2. Immediately after the existing NLTK download line in `worker-service/Dockerfile` (and at the
     equivalent point — after `pip install`, before `COPY` of app source — in
     `backend-api/Dockerfile`), add:
     ```dockerfile
     # Optional build-time pre-bake of the transformer sentiment model for air-gapped installs.
     # Off by default (BAKE_SENTIMENT_MODEL=false) to keep the base image lean; the transformer
     # provider is opt-in per-org and, when enabled without a pre-baked cache, downloads and
     # caches the model into HF_HOME on first use instead (requires runtime network access once).
     RUN if [ "$BAKE_SENTIMENT_MODEL" = "true" ]; then \
           python -c "from transformers import AutoModelForSequenceClassification, AutoTokenizer; \
     MODEL_ID='cardiffnlp/twitter-roberta-base-sentiment-latest'; REVISION='main'; \
     AutoModelForSequenceClassification.from_pretrained(MODEL_ID, revision=REVISION); \
     AutoTokenizer.from_pretrained(MODEL_ID, revision=REVISION)" ; \
         fi
     ```
     `REVISION='main'` is a placeholder — see spec OQ1. Tracked with a `# TODO(model-packaging):
     replace with sentiment-provider-core's pinned revision SHA before merge` comment directly
     above the `RUN` line so it isn't missed.
  3. Ensure `/app/models` is writable by the non-root `appuser` created later in the Dockerfile —
     the `RUN chown -R appuser:appuser /app` line already present in both Dockerfiles covers this
     as long as the pre-bake `RUN` (which executes as root, pre-`USER appuser`) happens before that
     chown — verify ordering, don't reorder existing steps unnecessarily.
- **Validation:**
  - Default build: `docker build -f services/worker-service/Dockerfile -t rrf-worker-default ./services`
    → build log shows no `huggingface.co` network activity; `docker run --rm rrf-worker-default
    ls /app/models` is empty or missing (AC1).
  - Baked build: `docker build --build-arg BAKE_SENTIMENT_MODEL=true -f
    services/worker-service/Dockerfile -t rrf-worker-baked ./services` succeeds; `docker run --rm
    -e HF_HUB_OFFLINE=1 -e TRANSFORMERS_OFFLINE=1 rrf-worker-baked python -c "from transformers import
    AutoModelForSequenceClassification, AutoTokenizer; AutoModelForSequenceClassification.from_pretrained('cardiffnlp/twitter-roberta-base-sentiment-latest', revision='main', local_files_only=True); AutoTokenizer.from_pretrained('cardiffnlp/twitter-roberta-base-sentiment-latest', revision='main', local_files_only=True)"`
    succeeds with **no** network access — this is AC2, the core offline-capability proof.
  - Repeat both builds for `backend-api/Dockerfile` (context `./services`) — AC3's baked half.
  - `docker images --format '{{.Repository}}: {{.Size}}'` for `rrf-worker-default` vs
    `rrf-worker-baked` — record the delta; expect roughly 440–500 MB attributable to weights alone
    (AC4). Note the exact figure in the PR description for reviewer sanity-check, don't hardcode an
    assertion into a script (weight sizes can drift slightly across HF revisions).
- **Commit:** `build(worker,backend): gate transformer model pre-bake behind BAKE_SENTIMENT_MODEL build arg`

### Phase 4 — Compose wiring for the build arg + offline flags
- **Goal:** An operator can flip on the pre-bake and the offline flags without editing
  Dockerfiles directly.
- **Prereqs:** Phase 3.
- **Files:** `docker-compose.yml`, `docker-compose.prod.yml`.
- **Steps:**
  1. On the `worker` and `backend` build blocks in **both** compose files, add:
     ```yaml
     build:
       context: ./services
       dockerfile: worker-service/Dockerfile   # or backend-api/Dockerfile
       args:
         BAKE_SENTIMENT_MODEL: ${BAKE_SENTIMENT_MODEL:-false}
     ```
  2. Do **not** add `HF_HUB_OFFLINE`/`TRANSFORMERS_OFFLINE` to the default `environment:` blocks —
     these are documented, opt-in, operator-set values (an operator who bakes the model but forgets
     to unset offline mode when they *don't* have local weights would break the fallback-to-VADER
     path unnecessarily; leaving them undocumented-by-default in compose and explicit-in-docs is
     safer). `docs/SELF_HOSTING.md` (Phase 5) shows the exact `.env` lines to add.
  3. `docker-compose.prod.yml`'s header comment block (lines 1-22) gets one new bullet noting the
     optional pre-bake, consistent with its existing "Notes:" bullet style.
- **Validation:**
  - `BAKE_SENTIMENT_MODEL=true docker compose -f docker-compose.prod.yml build backend worker`
    produces baked images; unset (default) produces lean images — both succeed.
  - `docker compose -f docker-compose.yml config` and `-f docker-compose.prod.yml config` both
    parse cleanly (no YAML/interpolation errors) after the edits.
- **Commit:** `build(compose): expose BAKE_SENTIMENT_MODEL build arg on worker + backend services`

### Phase 5 — `docs/SELF_HOSTING.md`
- **Goal:** An air-gapped operator can follow the doc end-to-end with no other context; a normal
  operator understands the feature is off-by-default and costs them nothing extra to ignore.
- **Prereqs:** Phases 1–4 (the doc must describe what actually exists).
- **Files:** `docs/SELF_HOSTING.md`.
- **Steps:**
  1. Add a new TOC entry + section, placed directly after "Fully-local LLM..." and before "Adding
     your own LLM key (BYOK)" (same neighborhood as the other local/offline AI content, same
     honest tone as the rest of the doc — no accuracy claims, that's the eval card's job per PRD).
     Section outline:
     - **What it is / default state:** VADER remains the default; the transformer is a per-org
       opt-in toggle (Settings → AI) that ships in `sentiment-provider-core` /
       `per-org-resolution` — this doc only covers *getting the weights onto the box*.
     - **Fastest path (online, no pre-bake):** enable the setting; the worker/backend download and
       cache the model into `HF_HOME=/app/models` on first use. Requires outbound network from the
       container the first time; cached afterward for the life of the container (note the
       ephemeral-filesystem caveat from §3 above — mount a volume at `/app/models` if you want the
       cache to survive container recreation, same idea as `postgres_data`).
     - **Air-gapped path (pre-bake at build time):**
       ```bash
       BAKE_SENTIMENT_MODEL=true docker compose -f docker-compose.prod.yml build backend worker
       ```
       then set `HF_HUB_OFFLINE=1` and `TRANSFORMERS_OFFLINE=1` in `.env` so the containers never
       attempt an outbound call even if one becomes reachable later (belt-and-suspenders for a
       genuinely air-gapped box).
     - **Minimum RAM / first-request latency** — a short table, explicitly labeled:
       > *Approximate, CPU-only, `roberta-base`-class model; confirm on your hardware. Numbers
       > below are placeholders pending measurement during implementation — replace before this
       > doc ships.*

       | | Estimate |
       |---|---|
       | Additional RAM per worker process (model loaded) | ~500 MB–1 GB |
       | Cold model load (first request after container start) | low single-digit seconds |
       | Steady-state inference per feedback item (CPU) | tens of ms |
     - **Image size note:** the `torch`/`transformers` packages themselves add roughly
       250–350 MB to the worker/backend images unconditionally once this ships (present whether or
       not any org enables the setting); the pre-bake ARG additionally adds ~450–500 MB **only**
       when explicitly requested.
  2. Cross-link from the existing "Running with no API key ($0, fully local)" section with one
     sentence noting the transformer is a further, still-local, still-$0, opt-in upgrade path —
     keeps the doc's existing narrative arc intact instead of orphaning the new section.
- **Validation:** manual read-through against Phases 1–4's actual commands (every command in the
  doc must be copy-pasteable and match exactly what was implemented); markdown TOC anchor added and
  verified to match the new heading's auto-generated slug.
- **Commit:** `docs(self-hosting): document local transformer sentiment model pre-bake + offline flags`

## 5. Environment & secrets

| Variable | Where | Default | Purpose |
|---|---|---|---|
| `HF_HOME` | Dockerfile `ENV` (worker + backend) | `/app/models` | Cache dir for any HF download/pre-bake; always set, costs nothing unset. |
| `BAKE_SENTIMENT_MODEL` | Docker build `ARG` / compose `build.args` | `false` | Gates the pre-bake `RUN` step. Build-time only, not a runtime env var. |
| `HF_HUB_OFFLINE`, `TRANSFORMERS_OFFLINE` | Runtime env (operator-set in `.env`, documented, **not** defaulted in compose) | unset | Forces zero network calls from `transformers`/`huggingface_hub`; only safe to set once weights are actually present (pre-baked or already pulled). |

No new secrets. No changes to `LLM_ENCRYPTION_KEY`, `JWT_SECRET`, or any BYOK plumbing.

## 6. Edge cases & error handling

- **Operator sets `HF_HUB_OFFLINE=1` without pre-baking and without a prior successful pull:** the
  `from_pretrained` call inside the (out-of-scope) provider code will raise; must-have #9's
  fallback-to-VADER-and-log is the provider aspect's responsibility, not this one's — but this plan
  documents the failure mode in `SELF_HOSTING.md` so it isn't a support-ticket surprise.
- **`BAKE_SENTIMENT_MODEL=true` build with no network access at build time (contradiction case):**
  the pre-bake `RUN` step fails the `docker build` outright (can't bake without the initial
  download) — this is correct, fail-fast behavior; documented as "pre-bake requires network access
  *at build time only*; the resulting image needs none at runtime."
- **Placeholder revision (`main`) drifts before `sentiment-provider-core` lands:** if `main` on the
  HF repo changes between this aspect merging and the provider aspect pinning a real SHA, a
  pre-baked image built in that window could cache different weights than what the provider
  eventually requests by SHA, causing a cache-miss network fetch at runtime (breaks air-gap
  silently until noticed). Mitigated by treating the `TODO` comment as a merge-blocking checklist
  item for whichever aspect merges second, not by inventing a fake SHA now.
- **`pip install --extra-index-url` resolution ambiguity:** confirm in Phase 1 validation that pip
  actually selects the CPU wheel and not a mixed CUDA/CPU install (`torch.__version__` won't show
  this — check `python -c "import torch; print(torch.version.cuda)"` prints `None` in the built
  image as a stronger signal).
- **Non-root user permissions on `/app/models`:** verified in Phase 3 step 3; if the pre-bake `RUN`
  ever gets reordered after `USER appuser`, the build fails with a permissions error immediately
  (loud, not silent) — acceptable to catch this at CI time rather than needing a special test.

## 7. Definition of done

- `docker build` of `worker-service/Dockerfile` and `backend-api/Dockerfile` succeeds for both
  `BAKE_SENTIMENT_MODEL=false` (default) and `=true`, from the correct (new, for backend) build
  contexts.
- Default builds: no Hugging Face network calls, no baked weights, image size only up by the
  CPU-wheel dep delta.
- Baked builds: `from_pretrained(..., local_files_only=True)` succeeds offline for both model and
  tokenizer, in both images.
- Backend image can `import analyzer` end-to-end (pre-existing bug fixed) — proven by AC3's smoke
  test.
- `docker compose build` (both compose files, both services) succeeds unchanged for an operator who
  sets nothing new.
- `docs/SELF_HOSTING.md` documents the pre-bake command, both offline flags, and RAM/latency
  estimates with the "confirm empirically" caveat.
- `sentiment-provider-core`'s eventual revision SHA is a one-line follow-up swap in the `RUN`
  command (the `TODO` comment), not a re-plumb of this aspect's Docker/compose work.
