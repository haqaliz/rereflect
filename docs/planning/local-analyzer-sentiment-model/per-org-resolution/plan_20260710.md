## 1. Context & Goals

`sentiment-provider-core` (FOUNDATION aspect, must land first) makes
`analyzer.sentiment.SentimentAnalyzer` accept an optional `provider: str | SentimentProvider =
"vader"` constructor arg with zero behavior change when omitted. It does **not** wire anything
per-org — both runtime call sites still construct `SentimentAnalyzer()` with zero args today.

This aspect is the DB-aware "last mile": a new `OrgAIConfig.sentiment_provider` column, a
never-raising backend/worker resolver pair that reads it, AI-settings PATCH/GET wiring +
validation, a `GET /sentiment/status` endpoint, and the two call-site edits that resolve-then-
inject a provider name into `SentimentAnalyzer(provider=...)`, falling back to VADER on any
failure at every layer.

**Headline acceptance criterion:** an org that never touches `sentiment_provider` produces
byte-identical analysis output to today (characterization-tested at both call sites). A resolver
or provider-construction failure never surfaces as a task error — it silently degrades to VADER
and logs.

**What this aspect does NOT do:** implement `SentimentProvider`/`VaderSentimentProvider`/
`TransformerSentimentProvider`/`SentimentProviderFactory` (`sentiment-provider-core` — I only call
the resulting `SentimentAnalyzer(provider=...)` constructor), touch `requirements.txt`/Dockerfiles
(`model-packaging`), build the eval harness/accuracy card (`eval-harness-and-card`), or wire a
frontend toggle (out of this aspect's file boundary — flagged as an open question).

## 4. Service & Route Design

### 4.1 Backend resolver — `src/services/sentiment_resolver.py` (new file)

```python
"""
Org-scoped sentiment provider resolver.

Single entry point for both sentiment call sites:

    resolved = resolve_sentiment_provider(org_id, db)
    provider_name = resolved.provider if resolved else "vader"
    analyzer = get_sentiment_analyzer(provider_name)

Design:
  - Reads OrgAIConfig.sentiment_provider for the org.
  - Returns None whenever the org isn't explicitly, validly configured (no
    OrgAIConfig row, NULL column, or an unrecognized value) — the caller's own
    "vader" fallback then applies. This differs from resolve_embedding_provider's
    None-means-"unconfigured, degrade" semantics only in that here VADER is
    always a fully-working default, not a degraded state.
  - Never raises to the caller.
  - Multi-tenant: always scoped by org_id.

References:
  - services/backend-api/src/services/embeddings/resolver.py (shape mirrored)
  - services/worker-service/src/services/sentiment_resolver.py (independent
    mirror — no cross-service import; reads the worker's own OrgAIConfig)
"""
from __future__ import annotations

import logging
from dataclasses import dataclass
from typing import Optional

logger = logging.getLogger(__name__)

VALID_SENTIMENT_PROVIDERS: frozenset[str] = frozenset({"vader", "transformer"})


@dataclass
class ResolvedSentiment:
    """Result of a successful sentiment provider resolution.

    Attributes:
        provider: Validated provider name, one of VALID_SENTIMENT_PROVIDERS.
    """
    provider: str


def resolve_sentiment_provider(org_id: int, db) -> Optional[ResolvedSentiment]:
    """
    Resolve the org's configured sentiment provider.

    Returns:
        ResolvedSentiment(provider=...) if the org has an explicit, valid
        sentiment_provider set ('vader' or 'transformer').
        None if there is no OrgAIConfig row, the column is NULL/unset, the
        value is unrecognized, or any error occurs reading it. Callers treat
        None as "use VADER" — this function never raises.
    """
    try:
        from src.models.org_ai_config import OrgAIConfig

        config = db.query(OrgAIConfig).filter_by(organization_id=org_id).first()
        if config is None:
            logger.debug(
                "resolve_sentiment_provider: no OrgAIConfig for org=%s", org_id
            )
            return None

        # getattr with default so this resolver never breaks against a DB that
        # hasn't run this aspect's migration yet (mirrors model_embeddings
        # precedent in resolve_embedding_provider).
        provider: Optional[str] = getattr(config, "sentiment_provider", None)

        if not provider:
            return None

        if provider not in VALID_SENTIMENT_PROVIDERS:
            logger.warning(
                "resolve_sentiment_provider: unrecognized sentiment_provider=%r "
                "for org=%s — degrading to vader",
                provider, org_id,
            )
            return None

        return ResolvedSentiment(provider=provider)

    except Exception as exc:
        logger.warning(
            "resolve_sentiment_provider: failed for org=%s: %s",
            org_id, exc, exc_info=True,
        )
        return None
```

### 4.4 `GET /api/v1/settings/ai/sentiment/status` (new route, after line 866)

```python
class SentimentStatusResponse(BaseModel):
    provider: str
    available: bool
    model: Optional[str] = None


@router.get("/sentiment/status", response_model=SentimentStatusResponse)
def get_sentiment_status(
    current_org: Organization = Depends(get_current_org),
    db: Session = Depends(get_db),
):
    """Return the org's sentiment-engine configuration status.

    Never raises to the caller — mirrors GET /embeddings/status. 'available'
    reflects dependency importability for 'transformer' (not a full model-load
    check, which is deliberately out of this endpoint's scope).
    """
    config = db.query(OrgAIConfig).filter_by(organization_id=current_org.id).first()
    provider = (
        config.sentiment_provider
        if config and getattr(config, "sentiment_provider", None)
        else "vader"
    )
    if provider not in VALID_SENTIMENT_PROVIDERS:
        provider = "vader"

    if provider == "vader":
        available = True
        model = None
    else:
        available = _sentiment_transformer_deps_available()
        model = _SENTIMENT_TRANSFORMER_MODEL_ID

    return SentimentStatusResponse(provider=provider, available=available, model=model)
```

**`get_sentiment_analyzer`** (lines 25-31) — replace with:

```python
# Process-level cache: one SentimentAnalyzer instance per provider name per
# process (must-have #9's call-site half — the model itself is loaded lazily
# and singleton-cached inside the transformer provider by sentiment-provider-core;
# this cache additionally avoids re-running SentimentAnalyzer(provider=...)
# construction, including its try/except fallback path, on every request).
_sentiment_analyzer_cache: dict = {}


def get_sentiment_analyzer(provider_name: str = "vader"):
    """Get SentimentAnalyzer with lazy import, cached per provider per process.

    provider_name is resolved by resolve_sentiment_provider (per-org-resolution
    aspect) and injected here via SentimentAnalyzer's `provider` constructor arg
    (sentiment-provider-core aspect). If construction fails for any reason
    (unknown name, missing deps, model load error), falls back to the VADER
    default and logs — never raises.
    """
    if provider_name in _sentiment_analyzer_cache:
        return _sentiment_analyzer_cache[provider_name]

    analysis_engine_path = os.path.abspath(os.path.join(os.path.dirname(__file__), "../../../analysis-engine"))
    if analysis_engine_path not in sys.path:
        sys.path.insert(0, analysis_engine_path)
    from analyzer.sentiment import SentimentAnalyzer

    try:
        analyzer = SentimentAnalyzer(provider=provider_name)
    except Exception as exc:
        import logging
        logging.getLogger(__name__).warning(
            "get_sentiment_analyzer: failed to construct provider=%r, falling "
            "back to VADER: %s", provider_name, exc, exc_info=True,
        )
        analyzer = SentimentAnalyzer()

    _sentiment_analyzer_cache[provider_name] = analyzer
    return analyzer
```

### 6.4 Worker/backend consistency

Because both resolvers read the **same physical column** with the **same validation set**
(`VALID_SENTIMENT_PROVIDERS = {"vader", "transformer"}`, duplicated by necessity — no shared
package between the two services), a given org always resolves to the same provider name whether
processed by the worker (async pipeline) or the backend (sync create/re-analyze path). Any future
change to the valid-provider set (e.g. adding `"per_org"` in M5.2) must update **both** copies of
the constant in the same PR — call this out in the PR description; no automated check exists for
drift between them (same limitation the crm-health-component plan flagged for `OrgAIConfig` in
general, R3 in that plan).

Both new resolver files and the AI-settings/status route changes have **zero import dependency**

## 7. Open Questions & Risks

### R-1: Exact `down_revision` for the migration

**Status:** Unresolved at plan time in the sense that it must be re-checked, not assumed — see
§6.1. Recorded single-head value (`a6b703d7a303`) is current as of spec/plan authoring
(2026-07-10) but this repo has a documented history of multi-heads from parallel branches.

### R-2: `SentimentAnalyzer(provider=...)` construction-failure semantics unconfirmed

**Status:** Open (see spec.md). `sentiment-provider-core`'s spec documents the constructor's
happy-path shape but not what it raises (or whether it raises at all vs. deferring to `.score()`)
for a bad/unavailable provider. **Mitigation:** the call-site `try/except Exception` (§4.5, §4.6)
is deliberately broad so it is correct under any of the plausible behaviors. Narrow it once the
real module's contract is confirmed, if a reviewer wants tighter exception handling.

### R-3: Frontend toggle has no owner in this aspect

**Status:** Flagged, not resolved. The PATCH/status endpoints this aspect builds exist to serve a
Settings → AI toggle, but `services/frontend-web` is outside this aspect's file boundary. Confirm
with the PRD/tech-plan owner which aspect (or a follow-up ticket) wires
`settings/ai/page.tsx`. Not a blocker for this aspect's own acceptance criteria (backend/worker
correctness stands alone and is testable via `curl`/pytest without a UI).

### R-4: Dependency-availability check depth (`find_spec` only)

**Decision:** deliberately shallow — checks importability, not "the model actually loads." A
deeper check (attempting a real `from_pretrained` call) would make `PATCH`/`GET /sentiment/status`
slow and network-dependent in exactly the air-gapped scenarios the PRD cares about protecting.
Documented as an explicit scope line in spec.md; revisit only if `model-packaging`/
`eval-harness-and-card` surface a real gap this leaves uncovered (e.g. deps present but wrong
version).

### R-5: No plan-tier gating on `sentiment_provider`

**Decision:** matches existing precedent — `default_provider`, `base_url`, and every other
AI-settings field in `ai_settings.py` are ungated by plan tier (BYOK/local-model settings are
available to all plans in the OSS/self-hosted framing). Confirmed consistent, not overlooked;
flagged in spec.md for explicit sign-off since it's a monetization-adjacent decision even though
the precedent is unambiguous.

### R-6: Existing test isolation

`test_ai_settings.py`, `test_ai_settings_local_llm.py`, `test_embeddings_status.py`, and
`test_keyword_analysis.py` do not construct `AISettingsResponse`/`AISettingsUpdate`/
`get_sentiment_analyzer` with positional args anywhere (all keyword or via the mocked-return
pattern) — confirmed by reading each file before this plan was written. Adding
`sentiment_provider: str = "vader"` / `Optional[str] = None` fields and an optional
`provider_name: str = "vader"` parameter is purely additive; no existing call site breaks.
