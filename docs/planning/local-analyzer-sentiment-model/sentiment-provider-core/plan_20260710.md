# Implementation Plan — sentiment-provider-core

**Date:** 2026-07-10 · **Slug:** `local-analyzer-sentiment-model` · **Aspect:** `sentiment-provider-core`
**Spec:** `./spec.md` · **PRD:** `../prd.md`
**Branch:** `feat/local-analyzer-sentiment-model` · **TDD:** strict RED → GREEN → REFACTOR per phase.

> Foundation aspect — **lands first**. Blocks `per-org-resolution`, `model-packaging`, and
> `eval-harness-and-card`. The single most load-bearing guarantee in this whole plan is
> **AC1/AC2 (byte-stability)**: a characterization test locks today's `analyze()` output
> **before** any refactor touches `sentiment.py`, and that same test must still pass, unmodified,
> after the provider seam is extracted. If it doesn't, the refactor is wrong — full stop.

## Current contract (quoted, exact — this is what must not change for the default path)

`services/analysis-engine/src/analyzer/sentiment.py:29-64` today:

```python
def analyze(self, text: str) -> Dict[str, float]:
    scores = self.vader.polarity_scores(text)
    compound = scores['compound']
    if compound >= 0.05:
        sentiment_label = 'positive'
    elif compound <= -0.05:
        sentiment_label = 'negative'
    else:
        sentiment_label = 'neutral'
    is_extreme = self._is_extreme_negative(text)
    has_churn_risk = self._has_churn_indicators(text)
    return {
        'compound': compound,
        'pos': scores['pos'],
        'neu': scores['neu'],
        'neg': scores['neg'],
        'label': sentiment_label,
        'is_extreme': is_extreme,
        'churn_risk': has_churn_risk
    }
```

7 keys, in this exact order: `compound, pos, neu, neg, label, is_extreme, churn_risk`. Floats are
**not rounded** at this layer. `SentimentAnalyzer()` takes zero constructor args today.

- **Critical import-path gotcha (from the code dig, `understanding.md` §4 and spec.md Risk):**
  this package is imported two different ways at runtime:
  - `services/analysis-engine` tests/API: `from src.analyzer.sentiment import SentimentAnalyzer`
  - `worker-service` / `backend-api` (copy the `analyzer/` dir verbatim, `PYTHONPATH`-rooted at
    its parent): `from analyzer.sentiment import SentimentAnalyzer`
  All new intra-package imports **must be relative** (`from .sentiment_providers.base import ...`),
  never `from src.analyzer...` — an absolute import would silently break the worker/backend copy
  while still passing analysis-engine's own test suite. This is the single easiest mistake to make
  in this aspect; call it out at every phase below.

### Phase 5 — `TransformerSentimentProvider` (mocked model — no real weights in unit tests)

- **Prerequisites:** Phase 4.
- **Files:** `sentiment_providers/providers/transformer.py`,

  2. **GREEN:** `sentiment_providers/providers/transformer.py`:
     ```python
     from __future__ import annotations
     import threading
     from ..base import SentimentProvider, SentimentScore

     _MODEL_NAME = "cardiffnlp/twitter-roberta-base-sentiment-latest"
     # TODO(model-packaging / eval-harness-and-card): pin to a specific commit SHA once the real
     # weights are downloaded and license-checked (see PRD Risks — model licensing/provenance;
     # spec.md OQ2). "main" is a placeholder, not a final pin.
     _MODEL_REVISION = "main"
     # cardiffnlp/twitter-roberta-base-sentiment-latest label order (model card): 0=negative,
     # 1=neutral, 2=positive — matches PRD must-have #3's stated softmax order.

     _singleton_lock = threading.Lock()
     _singleton_model = None
     _singleton_tokenizer = None


     def _get_model_and_tokenizer():
         """Lazily load + cache the model/tokenizer once per process (double-checked locking).
         Imports torch/transformers here, NOT at module level — this function is the only place
         in the whole sentiment_providers package that touches those heavy deps."""
         global _singleton_model, _singleton_tokenizer
         if _singleton_model is None:
             with _singleton_lock:
                 if _singleton_model is None:
                     from transformers import (
                         AutoModelForSequenceClassification,
                         AutoTokenizer,
                     )
                     tokenizer = AutoTokenizer.from_pretrained(_MODEL_NAME, revision=_MODEL_REVISION)
                     model = AutoModelForSequenceClassification.from_pretrained(
                         _MODEL_NAME, revision=_MODEL_REVISION
                     )
                     model.eval()
                     _singleton_tokenizer = tokenizer
                     _singleton_model = model
         return _singleton_model, _singleton_tokenizer


     class TransformerSentimentProvider(SentimentProvider):
         """CPU transformer provider — cardiffnlp/twitter-roberta-base-sentiment-latest.

         Model loads once per process on first score() call (module-level singleton, shared
         across every provider instance — PRD #9). Deterministic: eval() mode, no sampling.
         """

         MODEL_NAME = _MODEL_NAME

         def score(self, text: str) -> SentimentScore:
             import torch  # deferred — see _get_model_and_tokenizer docstring

             model, tokenizer = _get_model_and_tokenizer()
             inputs = tokenizer(text or "", return_tensors="pt", truncation=True, max_length=512)
             with torch.no_grad():
                 logits = model(**inputs).logits
             p_neg, p_neu, p_pos = torch.softmax(logits, dim=-1)[0].tolist()
             return {
                 "compound": p_pos - p_neg,
                 "pos": p_pos,
                 "neu": p_neu,
                 "neg": p_neg,
             }
     ```

### Phase 7 — Lazy-import guard (AC8 — proves torch is truly optional at import time)

- **Goal:** Prove `import`ing the sentiment/provider stack, and constructing/using the **vader**
  path through it, never requires `torch`/`transformers` to be importable.
- **Prerequisites:** Phase 6 (full stack exists).
- **Files:** `tests/sentiment_providers/test_lazy_import.py`.
- **Steps:**
  1. **RED:** write a subprocess-based test (a true fresh interpreter is the only reliable way to
     test "was X ever imported" — in-process `sys.modules` manipulation is fragile if another test
     already imported torch earlier in the same session):
     ```python
     import subprocess
     import sys
     from pathlib import Path

     ANALYSIS_ENGINE_ROOT = Path(__file__).resolve().parents[2]  # services/analysis-engine

     def test_vader_path_does_not_require_torch():
         code = (
             "import sys\n"
             "sys.modules['torch'] = None\n"          # any `import torch` now raises ImportError
             "sys.modules['transformers'] = None\n"    # same for transformers
             "from src.analyzer.sentiment import SentimentAnalyzer\n"
             "from src.analyzer.sentiment_providers.factory import SentimentProviderFactory\n"
             "a = SentimentAnalyzer()\n"                       # default = vader
             "r = a.analyze('This is fine, nothing bad here.')\n"
             "assert set(r.keys()) == {'compound','pos','neu','neg','label','is_extreme','churn_risk'}\n"
             "SentimentProviderFactory.create('vader')\n"      # explicit vader also fine\n"
             "print('OK')\n"
         )
         result = subprocess.run(
             [sys.executable, "-c", code],
             cwd=str(ANALYSIS_ENGINE_ROOT),
             capture_output=True, text=True, timeout=30,
         )
         assert result.returncode == 0, result.stderr
         assert "OK" in result.stdout

     def test_transformer_path_fails_cleanly_without_torch():
         """Documents where the import boundary actually is (OQ1): requesting the transformer
         provider is fine; SCORING with it is where the missing dep surfaces, as an ImportError
         raised from score(), which SentimentAnalyzer.analyze() catches per Phase 6's fallback."""
         code = (
             "import sys\n"
             "sys.modules['torch'] = None\n"
             "sys.modules['transformers'] = None\n"
             "from src.analyzer.sentiment_providers.factory import SentimentProviderFactory\n"
             "p = SentimentProviderFactory.create('transformer')\n"  # construction: no import yet
             "try:\n"
             "    p.score('x')\n"
             "    raise SystemExit('expected ImportError')\n"
             "except ImportError:\n"
             "    print('OK')\n"
         )
         result = subprocess.run(
             [sys.executable, "-c", code],
             cwd=str(ANALYSIS_ENGINE_ROOT),
             capture_output=True, text=True, timeout=30,
         )
         assert result.returncode == 0, result.stderr
         assert "OK" in result.stdout
     ```
  2. **GREEN:** if either test fails, it means some import in the chain (`sentiment.py` →
     `sentiment_providers/__init__.py` → `base.py`/`factory.py` → `providers/transformer.py`) is
     eagerly importing torch/transformers at module level instead of inside
     `_get_model_and_tokenizer()`. Fix by pushing the import further down (this is OQ1's resolution
     landing as executable proof, not just a design note).
  3. **REFACTOR:** none.
- **Validation:** `pytest tests/sentiment_providers/test_lazy_import.py -v`, then the **full**
  `pytest tests/ -v` from `services/analysis-engine` — this is the final gate for the aspect.
- **Commit:** `test(analysis-engine): prove sentiment_providers import lazily, vader path needs no torch`

## 6. Testing strategy

- **Scope:** unit tests only, entirely within `services/analysis-engine/tests/`. No integration
  test against a real HF model download — that's exploratory/manual (see §5 below) or the
  eval-harness aspect's job.
- **Mocking:** the transformer provider's tests (Phase 5) mock
  `transformers.AutoTokenizer.from_pretrained` and
  `transformers.AutoModelForSequenceClassification.from_pretrained` — no network call, no real
  weights, fast and deterministic in CI. This is intentional: this aspect proves the **contract
  and wiring** are correct; the eval-harness aspect proves the **real model's accuracy**.
- **Regression net:** the existing `tests/test_sentiment.py` (7 tests, label/sign-level
  assertions) and the new `tests/test_sentiment_characterization.py` (exact-value assertions) both
  run on every phase from Phase 3 onward — treat any red in either as a stop-the-line signal.
- **Order matters:** Phase 1 must run and pass **before** any file under `src/` changes. Every
  subsequent phase re-runs the full suite (`pytest tests/ -v`), not just its own new file, so a
  regression in an earlier phase is caught immediately rather than at the end.
- **Validation commands** (run from `services/analysis-engine`):
  ```bash
  cd services/analysis-engine
  pytest tests/ -v                                   # full suite, run after every phase
  pytest tests/test_sentiment_characterization.py -v  # the one test that must never move
  pytest tests/sentiment_providers/ -v                # new package's own tests
  ```
- **Environment note:** `python3 --version` in this worktree is 3.9.6 (per repo CLAUDE.md,
  "Python 3.9+" is the floor) — avoid syntax that needs 3.10+ (e.g. `match` statements,
  `X | Y` unions in runtime-evaluated positions without `from __future__ import annotations`,
  which is why every new file opens with that import, matching the embeddings package's style).

## 8. Edge cases & error handling

- **Empty string / whitespace-only text:** VADER already handles empty strings gracefully
  (returns near-zero scores); `TransformerSentimentProvider.score` passes `text or ""` into the
  tokenizer to avoid a `None` crash if a caller somehow passes `None` (defensive, matches the
  embeddings provider's "handle gracefully" doc comment in `base.py`).
  `_is_extreme_negative`/`_has_churn_indicators` already handle empty text (no regex match, `!= 3`
  count, no crash) — unchanged.
- **Very long text:** transformer tokenizer truncates to `max_length=512` tokens
  (`truncation=True`) — long feedback doesn't crash or silently OOM; VADER has no such limit and
  is unaffected either way.
- **Non-UTF8 / control characters / emoji-only text:** both providers pass text straight to their
  underlying library (`vaderSentiment` / HF tokenizer), which already handle arbitrary Unicode;
  no new handling needed, but worth one characterization-style test input with emoji if easy.
- **Provider `score()` raises (model load failure, tokenizer error, OOM, generic `Exception`):**
  caught by `SentimentAnalyzer.analyze()`'s try/except (Phase 6), falls back to VADER for that one
  item, logs a warning with `exc_info=True`, never propagates — **the analysis task must never
  error because of this aspect** (PRD #9, analyzer-side half).
- **Unknown/empty provider name at construction (`SentimentAnalyzer("nonsense")`):** propagates
  `SentimentProviderFactory`'s `ValueError` — this is a **construction-time** config error (the
  caller passed a bad provider name), deliberately **not** swallowed like a runtime `score()`
  failure would be; the per-org-resolution aspect is responsible for never passing an invalid name
  down (its resolver defaults to `"vader"` on any doubt, per the PRD's "unset/unknown → VADER"
  requirement — but that default lives in the resolver, not here).
- **Concurrent access to the transformer singleton (Celery prefork workers, potential future
  thread pools):** `_get_model_and_tokenizer()` uses double-checked locking with a module-level
  `threading.Lock()` — safe for multi-threaded access within one process. Celery's default prefork
  pool is multi-*process*, so each worker process gets its own singleton (which is exactly the
  "per-process lazy singleton" the PRD asks for, not a cross-process shared model) — no shared-memory
  concerns, no need for a multiprocessing-safe lock.
- **Model output contains `NaN`/`inf` (corrupted weights, edge-case input):** not explicitly
  guarded in this aspect — `torch.softmax` on `NaN` logits would propagate `NaN` into
  `compound`/`pos`/`neu`/`neg`, which then flows into `_classify_label` (`NaN >= 0.05` is `False`
  and `NaN <= -0.05` is `False`, so it silently resolves to `'neutral'` — not a crash, but worth
  flagging as a known soft edge case for the eval-harness aspect to notice if it shows up in
  results, rather than something this aspect adds defensive code for).
- **Double `VaderSentimentProvider` construction avoided:** `SentimentAnalyzer.__init__` reuses
  `self._provider` as `self._fallback_provider` when the primary provider already *is* VADER —
  avoids constructing `SentimentIntensityAnalyzer()` twice per `SentimentAnalyzer` instance.

## 9. Agent execution notes

- **Single agent, sequential phases.** This is one cohesive, small module — the phases are
  ordered and interdependent (especially Phase 3's rewire touching the one file every other phase
  builds around); subagent fan-out would add coordination overhead for no parallelism benefit.
  Dispatch one implementation agent to run Phases 1–7 in order, reporting after each phase's
  validation command.
- **Hard checkpoint after Phase 3.** This is the phase that modifies `sentiment.py` and is the
  entire byte-stability guarantee the rest of M5.1 depends on. Require the agent to paste the full
  diff of `sentiment.py` plus the `pytest tests/test_sentiment_characterization.py -v` and
  `pytest tests/test_sentiment.py -v` output (both green) before proceeding to Phase 4. Do not let
  the agent "fix" the characterization test to match new output — if it goes red, the refactor is
  wrong, not the test.
- **Ask before proceeding if:**
  - Phase 5 discovers the pinned `transformers==4.46.0` / `torch==2.5.0` versions in
    `requirements.txt` are incompatible with `cardiffnlp/twitter-roberta-base-sentiment-latest`
    (e.g. tokenizer API mismatch) — flag rather than silently bumping versions (that's arguably
    model-packaging's call, since it owns the Dockerfile pins too).
  - The real HF model's `id2label` mapping (verifiable only with network access, out of this
    aspect's mocked-test scope) turns out to differ from the assumed `0=negative, 1=neutral,
    2=positive` order — flag immediately, this would silently invert `compound`'s sign.
- **Do not:**
  - Modify `src/analyzer/core.py`, `src/analyzer/models.py`, `src/analyzer/__init__.py`, or
    `src/api/main.py` — out of scope, `FeedbackAnalyzer` needs no changes for this aspect.
  - Touch anything under `services/worker-service/`, `services/backend-api/`, or
    `services/frontend-web/` — those are `per-org-resolution`, `model-packaging`, and
    `eval-harness-and-card`'s aspects.
  - Add `OrgAIConfig`, migrations, or any DB/SQLAlchemy import anywhere in
    `sentiment_providers/` — this package must remain pure and DB-free (the PRD's central
    "DB-access split" architecture constraint).
  - Use absolute intra-package imports (`from src.analyzer...`) anywhere inside
    `src/analyzer/sentiment.py` or `src/analyzer/sentiment_providers/` — always relative
    (`from .sentiment_providers...`, `from ..base import ...`) so the worker/backend's bare
    `analyzer.*` import root keeps working (see §1's import-path gotcha).
