## 2. Implementation Phases

### Phase 1 — Backend gap fills (small)

- **Goal:** pin the two invariants D6/M8 depend on that aren't yet asserted.
- **Prerequisites:** none.
- **Files:**
  - `services/backend-api/tests/test_google_auth.py` **(new)** — service-level unit tests for
    `verify_google_access_token` / `verify_google_token`.
  - `services/backend-api/tests/test_auth.py` **(modify)** — one explicit JWT-claim-shape assertion if
    absent after reading.
- **Steps (TDD — write the assertion, watch it pass on HEAD = characterization):**
  1. Read `test_auth.py` **fully** and mark each backend spec item covered/absent (spec AC5). Expected
     result from the pre-read: password-null branch **covered** (`:420`), linking **covered** (`:322`),
     Google signup/login **covered**. Do not duplicate them.
  2. In `test_google_auth.py`, mock `httpx`/`google.oauth2.id_token` so the userinfo/id_token carries
     `email_verified: False`; assert the verifier returns `None` (`google_auth.py:92-94`, `:52`). Add
     the mirror `True` case for symmetry. This is the D6 invariant — the only genuinely missing backend
     test.
  3. Assert the JWT minted by `/login` decodes to exactly `{user_id, organization_id, role}` (the shape
     OIDC will reuse, M8). If `test_login_success` already decodes it, extend that assertion instead of
     adding a test.
- **Validation:** `cd services/backend-api && source venv/bin/activate && pytest tests/test_auth.py tests/test_google_auth.py -v` → green. **Scope to these files** — do not run the full suite (pre-existing `test_report_ws.py` segfault, PRD R7).
- **Commit:** `test(oidc-sso): characterize email_verified enforcement + JWT claim shape (M1 backend)`

### Phase 2 — Frontend: AuthContext characterization

- **Goal:** pin `AuthContext`'s token lifecycle and redirect behaviour.
- **Prerequisites:** Phase 1 committed (keeps history clean; not a code dependency).
- **Files:** `services/frontend-web/contexts/__tests__/AuthContext.test.tsx` **(new)**.
- **Steps:**
  1. Copy the render/provider pattern from `ThemeContext.test.tsx`. Mock `next/navigation`
     (`useRouter().push`) and seed/clear `localStorage`.
  2. Assert: token present on mount → `getMe()` called, user populated (mock the API module); no token →
     `router.push('/login')` (`AuthContext.tsx:60-62`); `login(token)` writes `localStorage['access_token']`
     and flips state (`:100`); `logout()` clears it (`:106`); `getMe()` throwing → clears token + redirects
     (`:82-90`).
  3. Mock `lib/api/auth` (`authAPI.getMe`) at the module boundary — do not hit the network.
- **Validation:** `cd services/frontend-web && npm run test -- AuthContext` → green.
- **Commit:** `test(oidc-sso): characterize AuthContext token lifecycle (M1 frontend)`

### Phase 3 — Frontend: api-client interceptors

- **Goal:** pin the Bearer-attach and 401-redirect behaviour a new auth path must not break.
- **Files:** `services/frontend-web/lib/__tests__/api-client.test.ts` **(new)**.
- **Steps:**
  1. Assert the request interceptor attaches `Authorization: Bearer <token>` from `localStorage`
     (`api-client.ts:19-21`) and omits it when absent.
  2. Assert the 401 response interceptor's **skip-list** logic (`:41-44`) — that `/login`, `/signup`,
     `/invite*`, `/shared*` are exempt from the redirect. Per spec R2, if asserting the actual
     `window.location.href` assignment (`:46`) is brittle in jsdom, assert the **decision** (skip-list
     membership / interceptor branch taken) and note the limitation in a test comment.
- **Validation:** `npm run test -- api-client` → green.
- **Commit:** `test(oidc-sso): characterize api-client Bearer + 401 interceptors (M1 frontend)`

### Phase 4 — Frontend: GoogleSignInButton + login/signup smoke

- **Goal:** pin the conditional-mount idiom (the exact pattern OIDC's button will copy) and the
  password/Google submit paths — a regression net, not exhaustive.
- **Files:**
  - `services/frontend-web/components/__tests__/GoogleSignInButton.test.tsx` **(new)**.
  - `services/frontend-web/app/login/__tests__/page.test.tsx` **(new, smoke only)**.
- **Steps:**
  1. `GoogleSignInButton`: renders nothing when `GOOGLE_CLIENT_ID` is unset (`:14,96-100`); on
     `useGoogleLogin` success, calls its `onSuccess` prop with the `access_token` (`:29`). Mock
     `@react-oauth/google`'s `useGoogleLogin`.
  2. `login/page`: password submit calls `authAPI.login` and stores the returned token; the Google
     branch calls `authAPI.googleLogin` (`login/page.tsx:85,101`). Mock the API module; assert calls,
     not navigation. Keep it a smoke test.
- **Validation:** `npm run test -- GoogleSignInButton login` → green; then `npm run test` (all frontend)
  → green; then `npm run lint` → clean.
- **Commit:** `test(oidc-sso): characterize Google button + login submit paths (M1 frontend)`

## 4. Testing Strategy

- This aspect **is** the tests. "Passing" = the new tests are green **on current HEAD**, proving they
  characterize existing behaviour (spec AC4) — none may depend on unbuilt OIDC.
- Backend: `pytest tests/test_auth.py tests/test_google_auth.py -v` (scoped; never bare full-suite —
  R7 segfault).
- Frontend: `npm run test` + `npm run lint`.
- Manual: none required — no runtime behaviour changes.

## 5. Environment & Secrets

- None new. Backend tests use the in-memory SQLite harness in `conftest.py`; `_disable_emails` is
  autouse. Frontend tests run under jsdom. No Redis, no live IdP, no `LLM_ENCRYPTION_KEY` needed (no
  encryption exercised in this aspect).

---

## 6. Edge Cases & Error Handling

- `email_verified` **absent** (not just `False`) must also yield `None` — assert both (`google_auth.py`
  uses `.get('email_verified', False)`).
- jsdom cannot navigate; `window.location.href` assignment is a documented soft spot (spec R2) —
  assert the decision, not the navigation, if brittle.
- Do not assert on exact error-copy strings beyond what the code guarantees; match the existing tests'
  level (they check substrings like `"Invalid email or password"`).
