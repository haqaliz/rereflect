**Aspect:** `worker-trainer-and-schedule` (the Celery driver)

## 1. Outcome

A Celery task module `services/worker-service/src/tasks/classifier_training.py` that runs the pure
training/eval core across all orgs weekly, persists versioned per-org sentiment classifier artifacts
with an **atomic active-model swap**, records an `org_classifier_eval_runs` row on **every** run
(`promoted`/`retained`/`skipped`), guards against overlapping per-org refits, purges inactive models
> 90 days, and is wired into `celery_app.py` on an uncrowded Monday slot. Mirrors
`tasks/churn_calibration.py` + `services/calibration_refit.py` conventions verbatim.

This aspect is the **only writer** of `org_classifier_models`. It does not touch predict-at-ingest,
API, or UI.

## 2. Grounded facts (verified against real code)

- **Beat slot `crontab(day_of_week=1, hour=6, minute=30)` (Mon 06:30 UTC) is FREE.** Monday slots
  currently used in `celery_app.py` `beat_schedule`: 07:00 / 07:15 / 07:30 (insights), 07:45 (churn
  refit), 08:30 (weekly insights). 06:30 is unoccupied. Nearest other Mon-agnostic daily jobs are
  02:00–04:15 and hourly `:00/:05/:30`; none collide with 06:30 Mon.
- **All-orgs helper:** churn uses module-private **`_all_org_ids(db)`** in
  `src/tasks/churn_calibration.py` (`db.query(Organization.id).all()` → list of ids); `refit_all_orgs`
  iterates it. This aspect defines its own local `_all_org_ids(db)` mirroring that (keeps the task
  module self-contained; do not import churn's private helper).
- **Churn has NO advisory lock.** The spec's "match existing churn approach if any, else a simple
  Redis key" therefore resolves to a **Redis lock**. The established worker pattern is in
  `src/tasks/analysis.py`: a module `_get_redis()` singleton (`redis.from_url(get_redis_url(0))`) plus
  `r.lock("lock:...", timeout=..., blocking=False)` then `lock.acquire(blocking=False)`. We mirror it
  with a **per-org key** `lock:classifier_refit:{org_id}`.
- **Session lifecycle:** `src/database.py::get_db_session()` is a `@contextmanager` that commits on
  success / rolls back on exception. `retrain_all_orgs` opens one session and passes it into
  `retrain_org(org_id, db)`; the callee owns its own `db.commit()` for the atomic swap (mirrors how
  `refit_org` commits internally).
- **Versioning template** (from `calibration_refit.refit_org`): load prev active → set
  `prev.is_active=False; db.add(prev)` → construct new row `is_active=True` with **denormalized
  metrics** → `db.add(new)` → `db.flush()` (populate `new.id` before the eval-run FK) → insert
  eval-run → `db.commit()`. This ordering is exactly what keeps the partial-unique satisfied.
- **Purge template** (`purge_old_calibration_models`): `filter(is_active == False, fit_at < now-90d)`
  → delete rows → commit → return `{"deleted": n}`.
- **Lazy heavy imports:** churn imports `numpy`/`sklearn` **inside** functions only. We keep the same
  rule; the classifier's sklearn lives entirely in the analysis-engine core — this task module must
  have **zero** module-level `sklearn`/`numpy` imports (and no import of the core at module top; import
  it lazily inside `retrain_org`).

## 3. Dependencies & assumed contracts (blocked-by)

This aspect is **blocked by** `data-layer` and `training-and-eval-core`. It consumes their surfaces.
If a signature below differs at integration time, adjust the thin adapter in `retrain_org` only.

**From `data-layer` (worker `src/models/__init__.py`), assumed importable:**
- `OrgClassifierModel` — cols: `organization_id` (nullable), `classifier_type`, `model_json` (JSON),
  `label_count`, `precision`, `recall`, `macro_f1`, `accuracy` (Numeric(5,4), nullable), `fit_at`,
  `is_active`; partial-unique `(organization_id, classifier_type) WHERE is_active`.
- `OrgClassifierEvalRun` — cols: `organization_id`, `classifier_model_id` (nullable FK), `classifier_type`,
  `incumbent_macro_f1`, `challenger_macro_f1`, `macro_f1_delta` (Numeric(5,4), nullable), `decision`
  (`promoted`|`retained`|`skipped`), `n` (int), `duration_ms` (int), `notes` (text), `created_at`.
- `AICorrection` mirrored into the worker (needed by the core's dataset builder).
- `Organization` (already present).

**From `training-and-eval-core` (`analysis-engine/src/analyzer/corrections_classifier/`), assumed:**
- `build_sentiment_dataset(org_id, db) -> Dataset` — reads `ai_corrections` (join `FeedbackItem.text`),
  fixed vocab `{positive,neutral,negative}`; exposes `.n` (usable label count).
- `train_and_eval(dataset, incumbent_predict, *, min_labels=20, margin=0.02) -> EvalResult` — the
  **pure** compute that owns the `MIN_LABELS` gate, holdout/min-holdout guard, shadow-A/B, and the
  promote decision. `EvalResult` fields consumed here: `.decision` (`promoted`|`retained`|`skipped`),
  `.incumbent_macro_f1`, `.challenger_macro_f1`, `.macro_f1_delta`, `.n`, `.notes`, and — when
  `decision=='promoted'` — `.model_json`, `.macro_f1`, `.precision`, `.recall`, `.accuracy`,
  `.label_count`, `.fit_at` (or the values retrain_org needs to denormalize).
- `incumbent_predict` is a callable `(text:str) -> label:str` that retrain_org builds from a **live**
  `SentimentAnalyzer` (lazy import from analysis-engine; default provider = incumbent VADER path;
  R3 = live-score for fairness).

> **Contract note / minor ambiguity (flagged):** the spec text lists the `MIN_LABELS` gate under
> `retrain_org` *and* the locked constraint says the gate/holdout guard is "handled in the core."
> This plan makes the **core authoritative** for all decision logic and treats `retrain_org` as a
> thin, decision-driven persister: it always writes an eval-run keyed off `result.decision`, and only
> writes a model row when `decision=='promoted'`. `retrain_org` does **not** re-implement the gate.
> This satisfies both statements without duplicating knobs. Confirm at integration.

## 5. Task module shape (target, for reference — built via TDD below)

```
src/tasks/classifier_training.py
  _MIN_LABELS = 20            # coarse mirror; authoritative gate is in the core
  _MARGIN = 0.02             # macro-F1 promotion margin
  _PURGE_AFTER_DAYS = 90
  _CLASSIFIER_TYPE = "sentiment"

  _get_redis()                        # singleton, mirror analysis.py
  _all_org_ids(db) -> list[int]       # mirror churn_calibration._all_org_ids
  _build_incumbent_predict()          # lazy SentimentAnalyzer -> callable(text)->label

  retrain_org(org_id, db) -> dict     # advisory lock; build->core->persist(decision); always eval-run
  retrain_all_orgs() -> dict          # {"trained","promoted","skipped"}; per-org try/except
  purge_old_classifier_models() -> dict  # {"deleted": n}
```

## 6. Phased TDD plan (RED → GREEN → REFACTOR)

Each phase is independently committable and ends green on `pytest tests/ -v`. Tests mirror
`tests/test_churn_calibration_tasks.py`: in-memory SQLite (`Base.metadata.create_all`), a `db`
fixture, `_make_org` / seed helpers, a `_get_tasks()` lazy import to avoid importing Celery at
collection, and an autouse fixture that stubs `sklearn`/`numpy` (and the analysis-engine core) so no
heavy wheels are needed. Because SQLite does **not** enforce the Postgres partial-unique, the
"exactly one active" invariant is asserted via explicit `count(*)` queries (this validates the code's
swap ordering, which is the real target).

## 8. Risks / open questions / ambiguities

1. **Gate ownership (flagged in §3).** Core is authoritative for `MIN_LABELS`/holdout; `retrain_org`
   is decision-driven. If integration reveals the core does *not* short-circuit below-gate (returns a
   trainable result instead of `decision='skipped'`), add a thin `if dataset.n < _MIN_LABELS: decision
   = 'skipped'` guard in `retrain_org` (constants already present). Zero test churn.
2. **Core/data-layer signatures are assumed** (§3). Only the thin adapter inside `retrain_org` binds to
   them; the autouse test stub decouples this aspect's tests from the core's real implementation.
3. **Purge placement.** Chosen: standalone `purge_old_classifier_models()` **folded** into
   `retrain_all_orgs` (no new beat slot), vs churn's separate purge beat entry. Folding keeps the beat
   uncrowded and still satisfies the AC. Trivially switchable to a standalone Sunday beat entry if
   preferred at review.
4. **Lock-miss records nothing** (Phase 6): a locked org writes no eval-run, because another in-flight
   run owns it. If product wants visibility into skipped-due-lock, add a `skipped` eval-run with a
   `notes='locked'` — deferred as it muddies the "one run, one record" invariant.
5. **SQLite can't enforce partial-unique** — invariants are asserted via `count(*)` on the code's swap
   ordering (the real thing under test). The DB constraint itself is data-layer's responsibility and
   is exercised in Postgres integration, not here.
