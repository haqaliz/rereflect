## 1. Outcome

Wire the trained per-org classifier artifact into the **live sentiment path** with `off`/`shadow`/`auto`
semantics + a 3-tier load fallback, exactly mirroring how M5.1's `resolve_sentiment_provider` +
provider injection already work at the two sentiment call-sites. Default `off` keeps analyzer output
**byte-identical**. This aspect owns only the *read/predict/inject* seam — never training, scheduling,
schema, or UI.

### Locked decisions (from spec open questions, resolved here)

1. **Shadow-log destination → dedicated structured application log, NOT a DB row per item.**
   Shadow predictions are logged via a dedicated logger `rereflect.classifier.shadow` at `INFO`
   with a structured `extra` payload (`org_id`, `feedback_id`, `classifier_type`, `incumbent_label`,
   `incumbent_score`, `challenger_label`, `challenger_score`, `model_id`, `fit_at`). Rationale:
   `org_classifier_eval_runs` is a **per-refit aggregate** table (one row per training run, holding
   challenger-vs-incumbent macro-F1 + decision) written by the worker-trainer aspect; writing one row
   per predicted feedback item would corrupt its semantics and add a DB write on the hot analysis path.
   A per-item log is cheap, needs **no migration** (this aspect owns no schema), is trivially testable
   via `caplog`, and matches M5.1's "disclosure-only, never mutate" posture. The accuracy card reads
   aggregate `eval_runs` (trainer aspect), not these per-item lines. Tradeoff stated: shadow lines are
   not SQL-queryable — acceptable for v1; a queryable per-item table is a documented later option.

2. **Ownership of `auto` override → WORKER is authoritative; backend inline path only shadow-logs.**
   (Locked by task.) The single shared helper is parametrized by one boolean `allow_override`:
   worker passes `allow_override=True` (in `auto`, mutates `feedback.sentiment_label/score`); backend
   create-time inline path passes `allow_override=False` (in `auto`, behaves like `shadow` — logs the
   challenger, leaves stored incumbent values). This keeps the mode-branching logic identical in both
   call-sites; the *only* divergence is that one flag. Prevents double-write races and a create-time /
   re-analysis flip-flop.

3. **`sentiment_score` mapping → `score = clamp(P(positive) − P(negative), -1, 1)`**, supplied by
   aspect B's `score_from_proba(proba, classes)`. Downstream `health_score_service` averages
   `FeedbackItem.sentiment_score` for trend math (confirmed: `health_score_service.py` L667–720
   `func.avg(FeedbackItem.sentiment_score)`), so the override MUST write a coherent `[-1,1]` float, not
   just a label. Characterization-tested against health input.

### Anti-drift strategy (spec R4)

The mode-branch + log-format + score-mapping live in **one helper**, `apply_classifier_override(...)`,
**byte-mirrored** in `services/backend-api/src/services/classifier_predict.py` and
`services/worker-service/src/services/classifier_predict.py` (same duplication convention as
`sentiment_resolver.py` — no cross-service import). Drift is guarded three ways:
- a **shared behavior-matrix constant** (`CLASSIFIER_SEAM_CASES`, the `(mode, promoted?, corrupt?,
  allow_override) → expected outcome` table) copied into both suites and asserted identical;
- a **mirror-equivalence test** in each suite that normalizes (strips the module docstring + import
  line that differs) and diffs the two `classifier_predict.py` / `classifier_resolver.py` bodies —
  same technique the repo already uses for hand-mirrored models;
- the worker/backend divergence expressed as the single `allow_override` argument, so there is no
  second code path to keep in sync.

## 2. Contracts consumed (dependencies)

- **Blocked by data-layer:** `OrgClassifierModel` ORM (both services' `models/`) with
  `organization_id` (nullable), `classifier_type`, `model_json` (JSON), `fit_at`, `is_active`,
  partial-unique one-active-per-`(org, type)`; `OrgAIConfig.classifier_mode` column
  (`off|shadow|auto`, default `'off'`). This aspect reads them getattr-defensively so it is testable
  **before** the migration lands (un-migrated-DB test).
- **Blocked by training-and-eval-core (aspect B):** pure `predict(model_json, text) -> (label, proba)`
  and `score_from_proba(proba, classes) -> float` in
  `services/analysis-engine/src/analyzer/corrections_classifier/` (dir does **not** exist yet).
  This aspect depends only on that signature; every test here injects a **fake predictor** through the
  loader seam, so the aspect is fully testable with aspect B unmerged. A single contract-adapter test
  (skipped/xfail until the module exists) pins the real signature.
- **Templates mirrored (do not reinvent):**
  `probability_updater._load_active_model` / `_deserialize_model` / `_identity_model` (3-tier +
  corrupt-artifact defensiveness), `sentiment_resolver.resolve_sentiment_provider` (getattr-defensive,
  never-raises resolver), `analysis.py::get_sentiment_analyzer` (lazy import + process cache),
  `feedback.py` inline cache+lock.

## 3. New modules (both services unless noted)

| File | Responsibility |
|---|---|
| `.../src/services/classifier_resolver.py` | `resolve_classifier(org_id, classifier_type, db) -> ResolvedClassifier \| None`. Reads `OrgAIConfig.classifier_mode` via `getattr(config,'classifier_mode','off')`. Returns `None` for `off`/missing row/missing column/invalid. **Never raises.** `VALID_CLASSIFIER_MODES = frozenset({"shadow","auto"})` (`off`→None). |
| `.../src/services/classifier_predict.py` | 3-tier loader + per-org cache + shared override helper. `load_active_classifier(org_id, type, db) -> LoadedClassifier \| None` (org active → global active `org_id IS NULL` → `None`=incumbent; corrupt `model_json` → warn + `None`). `LoadedClassifier.predict(text) -> (label, score)` wraps aspect-B `predict`+`score_from_proba` (lazy import). `apply_classifier_override(feedback, db, *, classifier_type, allow_override)` — resolves mode, loads, branches off/shadow/auto, logs shadow line, mutates only when `auto and allow_override and promoted-model-exists`. **Never raises.** |

Cache: module-level `dict` keyed `(org_id, classifier_type)` → `(model_id, fit_at, LoadedClassifier)`.
Lookup does a cheap **active-pointer query** (`id, fit_at` only, uses the partial-unique index) to
decide hit vs miss; on a matching `(id, fit_at)` it reuses the deserialized predictor (skips the
JSON→sklearn rebuild, the expensive part); a newer active `fit_at`/`id` invalidates and rebuilds.
Mirrors `_reset_sentiment_analyzer_cache` autouse fixture convention for test isolation.

## 6. Risks

- **R-A aspect B not merged** → all predict tests inject a fake `LoadedClassifier`; real signature
  pinned by one xfail contract test. **R-B data-layer not merged** → resolver/loader are
  getattr/try-defensive and tested against a schema-less mock (un-migrated-DB case) so this aspect is
  green independently. **R-C double-write** → resolved by `allow_override` ownership. **R-D hot-path
  cost** → shadow is a log line, not a DB write; loader cache skips sklearn rebuild.
