## 2. Implementation phases (strict TDD)

### Phase 1 — Write + validate methods on HubSpotClient
- **Goal:** add `update_contact_property` (PATCH) and `get_contact_property_def` (GET) with correct error classification.
- **Prerequisites:** none.
- **Files:**
  - modify `services/worker-service/src/clients/hubspot.py`
  - create `services/worker-service/tests/test_hubspot_write_client.py`
- **RED → GREEN steps:**
  1. Write failing unit tests (mock httpx via `respx` or monkeypatched transport — match whatever the existing HubSpot client tests use; check `tests/` for the current mocking style first):
     - `update_contact_property("123","rereflect_health_score",47)` issues `PATCH {BASE_URL}/crm/v3/objects/contacts/123` with JSON body `{"properties":{"rereflect_health_score":"47"}}` (HubSpot expects string values even for number props) and `Authorization: Bearer <token>`.
     - 429 and 500 → raises `HubSpotTransientError`.
     - 403 → raises a new permanent `HubSpotScopeError`.
     - 404 → raises a new permanent `HubSpotNotFoundError` (caller distinguishes contact-vs-property by the 404 source: PATCH-on-contact vs GET-on-property).
     - `get_contact_property_def("rereflect_health_score")` returns the parsed def dict for 200; returns `None` for 404; classifies `type`/`fieldType` and read-only flags so the caller can reject non-`number`/`calculated` props.
  2. Implement the two methods + the two new exception classes (subclass the existing permanent-error base if one exists; otherwise plain `Exception` subclasses next to `HubSpotTransientError`).
  3. Keep number formatting centralised: a small `_format_number(value)` helper so the task layer passes an int and the client stringifies.
- **Validation:** `cd services/worker-service && pytest tests/test_hubspot_write_client.py -v` green; full `pytest tests/ -v` still green.
- **Commit:** `feat(hubspot-client): PATCH contact property + property-def validation (writeback surface)`

## 6. Edge cases & error handling

- HubSpot returns 200 with a per-property error envelope in some cases — assert on status first, but if the existing client already inspects bodies, follow that pattern.
- Number property values must be sent as strings; a `None` score must never be pushed (caller guards, but `_format_number(None)` should raise to fail loud in tests).
- Property def `type` may be `number` with `fieldType` `number`/`calculation` — reject `calculation`/read-only.
