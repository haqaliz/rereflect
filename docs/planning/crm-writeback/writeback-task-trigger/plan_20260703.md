**Aspect spec:** `../writeback-task-trigger/spec.md` · **PRD:** `../prd.md`

**Depends on:** hubspot-write-client (PATCH surface), writeback-config-api (columns + flag).

## 0. Impact analysis — resolved risks

- **Enqueue-from-backend is CONFIRMED feasible.** `services/backend-api/src/background/celery_client.py` exposes `get_celery_app().send_task(...)`, already used by the HubSpot `/sync` route (`hubspot_integration.py:349,386`). → The **on-change trigger is a go** (no fallback to daily batch needed).
- Hook site: `services/backend-api/src/services/health_score_service.py:335`, immediately after `_check_health_drop_alert(...)`, in the existing-customer branch where `old_score`/`new_score` are in scope. `update_customer_health` does **not** commit (caller owns txn) → enqueue only (never call HubSpot inline).
- Worker sync task shape to mirror: `services/worker-service/src/tasks/hubspot_sync.py` (`_decrypt` via `LLM_ENCRYPTION_KEY`, `HubSpotClient` context manager, `@shared_task(bind=True, max_retries=3, default_retry_delay=30)`, `HubSpotTransientError`→`self.retry`). Salesforce `invalid_grant` disconnect (`salesforce_sync.py:372`) is the anti-pattern here — we **soft-pause instead**.
- `celery_app.include` list + `beat_schedule` in `services/worker-service/src/celery_app.py`.

- New worker task module `src/tasks/hubspot_writeback.py` added to `celery_app.include`. No migration (columns land in writeback-config-api). No env keys beyond existing `LLM_ENCRYPTION_KEY`/`REDIS_URL`.

## 2. Implementation phases (strict TDD)

### Phase 1 — The push task
- **Goal:** `push_health_to_hubspot(org_id, customer_email)` — idempotent, gated, soft-pausing.
- **Files:** create `services/worker-service/src/tasks/hubspot_writeback.py`; modify `services/worker-service/src/celery_app.py` (`include`); test `services/worker-service/tests/test_hubspot_writeback_task.py`.
- **RED → GREEN (write these failing tests first):**
  1. **no-op guards** (no PATCH): integration inactive; `writeback_enabled` False; `writeback_field_name` empty; no `crm_enrichment` row; enrichment row has no `hubspot_contact_id`; `CustomerHealth.health_score == last_written_health_score`; `abs(score - last_written) < 2` when `last_written` is set.
  2. **happy path:** active+enabled+field+matched contact+changed score → `HubSpotClient.update_contact_property(contact_id, field_name, score)` called once; on success set `last_written_health_score`, `last_health_written_at`, `last_writeback_at`, `last_writeback_status="ok"`, `last_writeback_error=None`, `contacts_written += 1`; commit.
  3. **soft-pause 403** (`HubSpotScopeError`): `last_writeback_status="error: missing_write_scope"`, `last_writeback_error` set, `is_active` **unchanged**; assert a subsequent inbound `sync_hubspot_org` still succeeds (read-sync untouched).
  4. **property 404** (`HubSpotNotFoundError` from GET/def or PATCH property) → `field_not_found` status, no `is_active` change.
  5. **contact 404** → skip that customer, status note `contact_not_found`, no global failure.
  6. **transient 429/5xx** (`HubSpotTransientError`) → `self.retry` invoked; `last_writeback_status="retrying"`.
  7. **missing encryption key** → `{"status":"error","reason":"missing_encryption_key"}`, no retry (mirror sync task).
- **Implementation:** load integration + enrichment + health via the worker mirror models; `_decrypt` token; `with HubSpotClient(token) as c:`; apply guards → PATCH → status writes. Reuse the `@shared_task(bind=True, max_retries=3, default_retry_delay=30)` decorator and the `HubSpotTransientError`→retry idiom verbatim.
- **Validation:** `cd services/worker-service && pytest tests/test_hubspot_writeback_task.py -v`; full suite green.
- **Commit:** `feat(crm-writeback): push_health_to_hubspot task (idempotent, soft-pause on failure)`

- **Files:** modify `services/backend-api/src/services/health_score_service.py`; test `services/backend-api/tests/test_health_writeback_enqueue.py`.
- **RED → GREEN:**

## 3. File-by-file build order

1. `worker-service/src/tasks/hubspot_writeback.py` — the task. Depends: write-client, mirror columns.
2. `worker-service/src/celery_app.py` — register in `include`.
3. `backend-api/src/services/health_score_service.py` — enqueue hook. Depends: celery_client.

5. Tests alongside each.

## 4. Testing strategy

- Worker: unit tests with mocked `HubSpotClient` + in-memory/SQLite session seeding the mirror models.
- Backend: mock `get_celery_app().send_task`; assert call/no-call and exception-swallowing.
- Regression: run the CRM characterization + health tests to prove read-side/score stability (G2).

## 5. Environment & secrets

- `LLM_ENCRYPTION_KEY` (already required by sync tasks), `REDIS_URL`/`CELERY_BROKER_DB` (already set). Redis + Celery worker must be running for manual verification.

## 6. Edge cases & error handling

- `update_customer_health` runs in both backend (in-request, CRM disconnect path) and worker (analysis/sync). The hook must be safe in both — hence lazy, ImportError-tolerant celery import + try/except.
- Overlapping pushes from rapid changes: last-writer-wins + idempotent skip; acceptable, unordered.
- Deleted/merged HubSpot contact → per-customer contact-404 skip; does not pause the org.
- Never flip `is_active` from the writeback path (that is the read-sync's liveness flag).

## 7. Agent execution notes

- **Do Phase 1 (worker task) and Phase 2 (backend hook) in that order** — the hook references the task name string; keep the fully-qualified task name identical in both places (`src.tasks.hubspot_writeback.push_health_to_hubspot`).
- **Checkpoint:** after Phase 2, run the characterization suite explicitly and report byte-identical result before proceeding (this is the G2 guarantee).
